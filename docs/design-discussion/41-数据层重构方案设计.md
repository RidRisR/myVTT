# 41 — 数据层重构方案设计

> 基于 [40-数据层架构现状分析与挑战](./40-数据层架构现状分析与挑战.md) 的分析结论，本文档定义具体的重构方案。

---

## 一、设计目标

1. **Gallery 与 Scene 解耦** — 素材库独立于游戏场景，不再"上传图片 = 创建 Scene"
2. **战斗预设系统** — 一个场景可关联多个战斗预设（Encounter），每个预设包含完整的战术桌面快照
3. **数据分层** — Yjs 回归实时协同本职，静态素材元数据迁移到独立存储
4. **按房间物理隔离** — 每个房间拥有独立的数据目录，自包含、可备份、可迁移
5. **Scene 结构精简** — 从 22 个打散字段精简为职责清晰的分层结构
6. **延迟 GC** — Entity 垃圾回收从实时触发改为房间空闲时执行，消除并发风险

---

## 二、架构总览

### 2.1 存储拓扑

```
data/
├── rooms.json                         ← 房间列表（全局）
├── rooms/
│   ├── {roomId}/                      ← 一个房间 = 一个自包含目录
│   │   ├── db/
│   │   │   ├── yjs/                   ← LeveldbPersistence（Y.Doc 持久化）
│   │   │   └── assets/               ← classic-level（素材元数据 KV）
│   │   └── uploads/                   ← 素材文件（图片/视频/音频）
│   ├── {roomId2}/
│   │   ├── db/
│   │   │   ├── yjs/
│   │   │   └── assets/
│   │   └── uploads/
```

**每个房间两个 LevelDB 实例**：

- `db/yjs/` — `y-leveldb` 的 `LeveldbPersistence` 实例，专门用于 Y.Doc 持久化（`getYDoc`、`storeUpdate`）
- `db/assets/` — `classic-level` 实例，用于素材元数据的通用 KV 读写（`get`、`put`、`del`、`iterator`）

> **为什么分开？** `y-leveldb` 的 `LeveldbPersistence` 只暴露 Y.Doc 专用方法（`getYDoc`、`storeUpdate`、`clearDocument`），**不支持通用的 `get()`/`put()`/`del()` 操作**。素材元数据需要标准 KV 存储，必须使用独立的 `classic-level` 实例。

素材文件存放在房间自己的 `uploads/` 目录。

**房间操作**：

- 删除房间 = 关闭 LevelDB 连接 + `rm -rf data/rooms/{roomId}/`
- 备份房间 = `tar czf {roomId}.tar.gz data/rooms/{roomId}/`
- 迁移房间 = 拷贝目录到另一台服务器

### 2.2 数据归属

| 数据                                                 | 存储位置                 | 原因                           |
| ---------------------------------------------------- | ------------------------ | ------------------------------ |
| 素材文件（图片/视频）                                | 房间 `uploads/` 目录     | 二进制文件                     |
| 素材元数据（名称/标签/尺寸/Blueprint 配置/讲义内容） | 房间 LevelDB `asset:*`   | 需要按标签过滤、独立于游戏状态 |
| Scene 定义（名称/氛围/预设）                         | Y.Doc `scenes` Y.Map     | 游戏文档，需要多端同步         |
| Entity（角色/NPC 数据）                              | Y.Doc `entities` Y.Map   | 需要 field-level CRDT          |
| 战斗运行时（token 位置/先攻/迷雾）                   | Y.Doc `combat` Y.Map     | 需要实时多人同步               |
| 聊天消息                                             | Y.Doc `chat_log` Y.Array | 实时对话，需要保序             |
| Seat/身份                                            | Y.Doc `seats` Y.Map      | 关联 Awareness                 |
| Showcase                                             | Y.Doc `showcase` Y.Map   | 运行时展示，未来形态待定       |

## 三、素材管理体系

### 3.1 素材元数据结构

所有素材（图片、Blueprint、讲义）统一存储在 LevelDB 中，key 格式为 `asset:{assetId}`，value 为 JSON 字符串。

**统一结构**：

```typescript
interface AssetMeta {
  id: string
  url: string // "/uploads/{filename}" 相对路径
  name: string // 显示名称
  type: 'image' | 'blueprint' | 'handout'
  tags: string[] // 自由标签，如 ["地图", "酒馆"]
  width?: number // 图片/视频宽度
  height?: number // 图片/视频高度
  createdAt: number // 创建时间戳

  // 类型特有数据（按 type 可选存在）
  blueprint?: {
    // type='blueprint' 时
    defaultSize: number
    defaultColor: string
    defaultRuleData?: unknown
  }
  handout?: {
    // type='handout' 时
    title: string
    description: string
  }
}
```

**共有字段**：id、url、name、type、tags、createdAt
**类型特有字段**：按 `type` 携带不同的嵌套数据，未来新增素材类型只需加新的可选字段。

### 3.2 素材 REST API

```
POST   /api/rooms/:roomId/assets          ← 上传素材（multipart + 元数据）
GET    /api/rooms/:roomId/assets          ← 列出素材（支持 ?tag=xxx 过滤）
GET    /api/rooms/:roomId/assets/:id      ← 获取单个素材元数据
PATCH  /api/rooms/:roomId/assets/:id      ← 更新元数据（改名、改标签等）
DELETE /api/rooms/:roomId/assets/:id      ← 删除素材（元数据 + 文件）

GET    /api/rooms/:roomId/uploads/:file   ← 静态文件服务（图片/视频）
```

### 3.3 素材与 Y.Doc 的关联方式

素材元数据在 LevelDB 中，游戏状态在 Y.Doc 中。两者通过 **URL 字符串**松耦合关联：

```
LevelDB                              Y.Doc
asset:a001                           scenes.scene_001.atmosphere:
  { url: "/uploads/abc.jpg",    ──→    { imageUrl: "/uploads/abc.jpg", ... }
    name: "酒馆外景",
    tags: ["地图","酒馆"] }
```

Y.Doc 不知道 LevelDB 的存在，它只存了一个 URL 字符串。删除素材不会自动清理 Y.Doc 中的引用——这是可接受的松耦合（类似网页引用被删的图片，显示破图标）。

### 3.4 Blueprint 从素材到 Entity 的流程

Blueprint 是模板（"类"），Entity 是实例（"对象"）：

```
1. GM 从素材库选择一个 Blueprint（type='blueprint'）
2. 读取 LevelDB 中的 asset 元数据
3. 创建 Entity 写入 Y.Doc：
   {
     name: "哥布林守卫 A",              ← 自动编号
     imageUrl: asset.url,               ← 拷贝 URL
     color: asset.blueprint.defaultColor,
     size: asset.blueprint.defaultSize,
     ruleData: asset.blueprint.defaultRuleData,  ← 深拷贝
     blueprintId: asset.id              ← 可选弱引用（用于批量更新）
   }
4. Entity 独立存活，修改 Entity 不影响 Blueprint
```

## 四、Y.Doc 结构重设计

### 4.1 改造前后对比

```
【改造前】9 个顶层 key，Scene 22 个字段
Y.Doc
├── room            { activeSceneId, pinnedShowcaseId }
├── scenes          每个 Scene 22 个打散字段 + tokens + entityIds
├── entities        field-level CRDT（不变）
├── blueprints      ← 迁移到 LevelDB
├── seats           身份（不变）
├── chat_log        聊天（不变）
├── showcase_items  ← 重命名为 showcase
├── handout_assets  ← 迁移到 LevelDB
├── team_metrics    ← 合并到 room

【改造后】7 个顶层 key，Scene 精简
Y.Doc
├── room            { activeSceneId, activeEncounterId, teamMetrics }
├── scenes          精简的场景定义 + encounters 预设
├── entities        field-level CRDT（不变）
├── combat          战斗运行时（新增顶层）
├── showcase        运行时展示（保留，未来自由演化）
├── seats           身份（不变）
├── chat_log        聊天（不变）
```

### 4.2 Scene 结构

```
scenes: Y.Map<Y.Map>
├── scene_001: Y.Map
│     name: "黑龙酒馆"                          ← 场景名称
│     sortOrder: 0                              ← 排序
│     atmosphere: {                             ← 叙事氛围（plain JSON）
│       imageUrl: "/uploads/tavern.jpg",
│       width: 1920,
│       height: 1080,
│       particlePreset: "embers",
│       ambientPreset: "tavern",
│       ambientAudioUrl: "/uploads/ambience.mp3",
│       ambientAudioVolume: 0.5
│     }
│     entityIds: Y.Map { "entity_001": true }   ← 场景中的角色引用
│     encounters: Y.Map {                       ← 战斗预设（见第五节）
│       "enc_001": { ... },
│       "enc_002": { ... }
│     }
```

**字段说明**：

| 字段         | 类型                 | 说明                                 |
| ------------ | -------------------- | ------------------------------------ |
| `name`       | string               | 场景名称                             |
| `sortOrder`  | number               | 场景列表排序                         |
| `atmosphere` | plain JSON           | 叙事层配置：背景图、尺寸、粒子、音频 |
| `entityIds`  | Y.Map<boolean>       | 场景中的角色引用（不变）             |
| `encounters` | Y.Map<EncounterData> | 战斗预设存储（新增）                 |

**从 Scene 移除的字段**：

| 移除的字段                                                                       | 去向                                        |
| -------------------------------------------------------------------------------- | ------------------------------------------- |
| `atmosphereImageUrl`                                                             | → `atmosphere.imageUrl`                     |
| `width`, `height`                                                                | → `atmosphere.width/height`                 |
| `particlePreset`                                                                 | → `atmosphere.particlePreset`               |
| `ambientPreset`, `ambientAudioUrl`, `ambientAudioVolume`                         | → `atmosphere.*`                            |
| `gridSize`, `gridSnap`, `gridVisible`, `gridColor`, `gridOffsetX`, `gridOffsetY` | → encounter 预设 / combat 运行时            |
| `tacticalMapImageUrl`, `battleMapUrl`                                            | → encounter 预设 / combat 运行时            |
| `combatActive`                                                                   | → `room.activeEncounterId != null` 隐含表达 |
| `initiativeOrder`, `initiativeIndex`                                             | → combat 运行时                             |
| `tokens`                                                                         | → combat 运行时                             |

### 4.3 Room 结构

```
room: Y.Map
  activeSceneId: string | null         ← 当前激活的场景
  activeEncounterId: string | null     ← 当前激活的战斗预设（null = 非战斗状态）
  teamMetrics: { ... }                 ← 团队追踪器（从独立顶层合并）
```

`combatActive` 布尔值不再需要——`activeEncounterId != null` 即表示战斗进行中。

### 4.4 Combat 结构

`combat` 是房间级别的**单例运行时容器**。整个房间同一时间只有一场激活的战斗。没有 ID——它的身份由 `room.activeEncounterId` 确定。

```
combat: Y.Map
  ── 从 encounter 预设展开 ──
  mapUrl: string                       ← 战术地图 URL
  mapWidth: number
  mapHeight: number
  grid: {                              ← Grid 配置（plain JSON）
    size: number,
    snap: boolean,
    visible: boolean,
    color: string,
    offsetX: number,
    offsetY: number
  }
  tokens: Y.Map {                      ← 运行时 token（支持多人实时拖动）
    "tok_001": { entityId, x, y, size, permissions, ... }
    "tok_002": { ... }
  }

  ── 战斗进程状态 ──
  initiativeOrder: string[]            ← 先攻顺序
  initiativeIndex: number              ← 当前回合

  ── 未来扩展（按需添加，不需要预定义）──
  fog: Y.Map { ... }                   ← 战争迷雾
  drawings: Y.Map { ... }              ← 手绘标注
  notes: Y.Map { ... }                 ← 地图便签
```

**生命周期**：

- **激活预设**：清空 combat Y.Map 所有 key → 从 encounter JSON 展开写入 → 设置 `room.activeEncounterId`
- **结束战斗**：（可选保存快照到 encounter）→ 清空 combat Y.Map → 设置 `room.activeEncounterId = null`
- **切换场景**：如果战斗进行中 → 先结束战斗 → 再切换场景

## 五、战斗预设系统（Encounter）

### 5.1 数据结构

Encounter 以 plain JSON 对象存储在 `scene.encounters` Y.Map 中：

```typescript
interface EncounterData {
  name: string // 预设名称
  mapUrl: string // 战术地图 URL
  mapWidth: number
  mapHeight: number
  grid: {
    size: number
    snap: boolean
    visible: boolean
    color: string
    offsetX: number
    offsetY: number
  }
  tokens: Record<
    string,
    {
      // Token 快照（自包含，不依赖外部引用）
      name: string
      imageUrl: string
      color: string
      size: number
      x: number
      y: number
      ruleData?: unknown // 数值快照
      blueprintId?: string // 可选弱引用
    }
  >
  // 未来扩展字段按需添加：fog, drawings, notes...
}
```

**关键设计决策**：Encounter 中的 token 是**自包含快照**，存储了完整的外观和数值数据，不依赖 Entity 或 Blueprint 是否仍然存在。`blueprintId` 仅作为可选的弱引用，用于"从模板同步更新"功能。

### 5.2 序列化与反序列化

**保存当前战场为预设**（combat → encounter）：

```typescript
function saveAsEncounter(sceneMap: Y.Map, combat: Y.Map, name: string) {
  const snapshot = combat.toJSON() // Y.Map.toJSON() 递归转为 plain object
  delete snapshot.initiativeOrder // 剥离运行时进程状态
  delete snapshot.initiativeIndex
  const encounters = sceneMap.get('encounters') as Y.Map
  encounters.set(generateId(), { name, ...snapshot })
}
```

**激活预设**（encounter → combat）：

```typescript
function activateEncounter(yDoc: Y.Doc, room: Y.Map, sceneMap: Y.Map, encounterId: string) {
  const encounter = (sceneMap.get('encounters') as Y.Map).get(encounterId)
  const combat = yDoc.getMap('combat')

  yDoc.transact(() => {
    // 清空所有战斗状态（不管之前有什么字段，一步清空）
    combat.forEach((_, key) => combat.delete(key))

    // 写入地图和 grid
    combat.set('mapUrl', encounter.mapUrl)
    combat.set('mapWidth', encounter.mapWidth)
    combat.set('mapHeight', encounter.mapHeight)
    combat.set('grid', encounter.grid)

    // 写入 token（创建 Y.Map 以支持实时多人操作）
    const tokens = new Y.Map()
    combat.set('tokens', tokens)
    for (const [id, t] of Object.entries(encounter.tokens)) {
      tokens.set(id, t)
    }

    // 初始化战斗进程
    combat.set('initiativeOrder', [])
    combat.set('initiativeIndex', 0)

    // 设置激活标记
    room.set('activeEncounterId', encounterId)
  })
}
```

**结束战斗**：

```typescript
function endCombat(yDoc: Y.Doc, room: Y.Map) {
  const combat = yDoc.getMap('combat')
  yDoc.transact(() => {
    combat.forEach((_, key) => combat.delete(key))
    room.set('activeEncounterId', null)
  })
}
```

### 5.3 Scene 与 Encounter 的关系

```
房间
├── 场景 A "黑龙酒馆"
│   ├── encounter "一楼大厅"    ← 各自独立的战斗预设
│   ├── encounter "二楼包间"
│   └── encounter "地下室"
├── 场景 B "迷雾森林"
│   └── encounter "伏击点"
│
└── combat（顶层单例）          ← 当前激活的战斗运行时
      来源：场景 A / encounter "一楼大厅"
```

一个房间同一时间只有一个激活的 encounter。`room.activeSceneId` + `room.activeEncounterId` 唯一确定当前战斗的来源。

## 六、服务端改造

### 6.1 按房间隔离的双 LevelDB

每个房间维护两个 LevelDB 实例：

```javascript
import { ClassicLevel } from 'classic-level' // 新增依赖

const roomYjsDbs = new Map() // roomId → LeveldbPersistence（Y.Doc 专用）
const roomAssetDbs = new Map() // roomId → ClassicLevel（素材元数据 KV）

function getRoomYjsDb(roomId) {
  if (!roomYjsDbs.has(roomId)) {
    const dbPath = path.join(DATA_DIR, 'rooms', roomId, 'db', 'yjs')
    fs.mkdirSync(dbPath, { recursive: true })
    roomYjsDbs.set(roomId, new LeveldbPersistence(dbPath))
  }
  return roomYjsDbs.get(roomId)
}

function getRoomAssetDb(roomId) {
  if (!roomAssetDbs.has(roomId)) {
    const dbPath = path.join(DATA_DIR, 'rooms', roomId, 'db', 'assets')
    fs.mkdirSync(dbPath, { recursive: true })
    roomAssetDbs.set(roomId, new ClassicLevel(dbPath, { valueEncoding: 'json' }))
  }
  return roomAssetDbs.get(roomId)
}

// y-websocket 持久化路由到房间级 LeveldbPersistence
setPersistence({
  bindState: async (docName, ydoc) => {
    const db = getRoomYjsDb(docName)
    const persistedYdoc = await db.getYDoc(docName)
    Y.applyUpdate(ydoc, Y.encodeStateAsUpdate(persistedYdoc))
    ydoc.on('update', (update) => db.storeUpdate(docName, update))
  },
  writeState: async () => {},
})
```

> **注意**：`classic-level` 需要作为新依赖安装（`npm install classic-level`）。它是 `level` 生态的标准 KV 存储实现，API 为标准的 `get(key)`、`put(key, value)`、`del(key)` 和 `iterator()`。

### 6.2 按房间隔离的文件上传

```javascript
// 上传 API 改为房间级
app.post('/api/rooms/:roomId/upload', upload.single('file'), (req, res) => {

  const uploadsDir = path.join(DATA_DIR, 'rooms', roomId, 'uploads')

  // multer 配置动态 destination...

// 静态文件服务改为房间级
app.use('/api/rooms/:roomId/uploads', (req, res) => {

```

### 6.3 素材元数据 CRUD

使用 `classic-level` 实例（`getRoomAssetDb`），支持标准 KV 操作：

```javascript
// 列出房间所有素材
app.get('/api/rooms/:roomId/assets', async (req, res) => {
  const db = getRoomAssetDb(req.params.roomId)
  const assets = []
  for await (const value of db.values()) {
    assets.push(value)
  }
  // 可选：按 tag 过滤
  const tag = req.query.tag
  res.json(tag ? assets.filter((a) => a.tags?.includes(tag)) : assets)
})

// 创建素材（配合文件上传）
app.post('/api/rooms/:roomId/assets', async (req, res) => {
  const db = getRoomAssetDb(req.params.roomId)
  const asset = { id: generateId(), ...req.body, createdAt: Date.now() }
  await db.put(asset.id, asset) // valueEncoding: 'json' 自动序列化
  res.status(201).json(asset)
})

// 更新素材元数据
app.patch('/api/rooms/:roomId/assets/:id', async (req, res) => {
  const db = getRoomAssetDb(req.params.roomId)
  const existing = await db.get(req.params.id)
  const updated = { ...existing, ...req.body }
  await db.put(req.params.id, updated)
  res.json(updated)
})

// 删除素材
app.delete('/api/rooms/:roomId/assets/:id', async (req, res) => {
  const db = getRoomAssetDb(req.params.roomId)
  const asset = await db.get(req.params.id)
  // 删除文件
  const filePath = path.join(
    DATA_DIR,
    'rooms',
    req.params.roomId,
    'uploads',
    path.basename(asset.url),
  )
  if (fs.existsSync(filePath)) fs.unlinkSync(filePath)
  // 删除元数据
  await db.del(req.params.id)
  res.json({ ok: true })
})
```

### 6.4 房间删除

```javascript
app.delete('/api/rooms/:id', async (req, res) => {
  const roomId = req.params.id
  // 关闭两个 LevelDB 连接
  const yjsDb = roomYjsDbs.get(roomId)
  if (yjsDb) {
    await yjsDb.destroy()
    roomYjsDbs.delete(roomId)
  }
  const assetDb = roomAssetDbs.get(roomId)
  if (assetDb) {
    await assetDb.close()
    roomAssetDbs.delete(roomId)
  }
  // 删除整个房间目录
  const roomDir = path.join(DATA_DIR, 'rooms', roomId)
  fs.rmSync(roomDir, { recursive: true, force: true })
  // 更新 rooms.json
  const rooms = readRooms().filter((r) => r.id !== roomId)
  writeRooms(rooms)
  res.json({ ok: true })
})
```

## 七、延迟 GC（Entity 垃圾回收）

### 7.1 现有问题

当前 `gcOrphanedEntities()` 在删除场景时立即执行，扫描所有场景的 entityIds。多客户端在线时存在竞态风险。

### 7.2 改造方案

将 GC 触发时机从"用户操作时"改为"房间空闲时"：

```
最后一个 WebSocket 连接断开
  → 延迟 5 秒（防止页面刷新误触发）
  → 如果仍然无连接 → 执行 GC
```

**GC 内容**：

1. **孤儿 Entity 清理**：扫描 `entities`，找出 `persistent=false` 且不在任何 `scene.entityIds` 中的 Entity → 删除
2. **断裂 Token 引用清理**：扫描所有 `scene.encounters` 中的 token，如果 `entityId` 指向不存在的 Entity → 清除 `entityId` 字段（token 退化为匿名）
3. **可选：孤儿文件清理**：扫描 `uploads/` 目录，找出不被任何素材元数据记录引用的文件 → 删除

**服务端实现**：

```javascript
// y-websocket 提供的连接追踪
wss.on('connection', (conn, req) => {
  const roomId = extractRoomId(req)
  // ... setupWSConnection

  conn.on('close', () => {
    // 检查该房间是否还有活跃连接
    setTimeout(() => {
      if (getRoomConnectionCount(roomId) === 0) {
        performGC(roomId)
      }
    }, 5000)
  })
})
```

**优势**：

- 零并发风险——没有人在线时操作 Y.Doc
- 删除操作变为纯 UI 行为（标记不可见），物理清理延后
- GC 逻辑集中在服务端一处，不分散在各个客户端操作中

## 九、设计决策记录

| 决策                  | 选项                                         | 选择              | 理由                                                                        |
| --------------------- | -------------------------------------------- | ----------------- | --------------------------------------------------------------------------- |
| 素材元数据存储        | SQLite / JSON 文件 / LevelDB                 | `classic-level`   | 同属 LevelDB 生态（一个新依赖），标准 KV API，按 key 独立读写，房间隔离自然 |
| 战斗预设存储          | SQLite / LevelDB / Y.Doc                     | Y.Doc             | 属于游戏文档，与 Scene 强关联，数据结构灵活（plain JSON）                   |
| 战斗运行时位置        | 嵌套在 Scene / 独立顶层                      | 独立顶层 `combat` | Scene = 定义，Combat = 运行时，职责分离；切换预设时整体清空不留残留         |
| 房间数据隔离          | 共享 LevelDB + key 前缀 / 每房间独立 LevelDB | 每房间独立        | 自包含、可备份/迁移、删除干净                                               |
| Entity GC 时机        | 实时（删除时触发） / 延迟（空闲时触发）      | 延迟              | 消除多客户端并发时的竞态风险                                                |
| 素材与 Y.Doc 关联方式 | ID 外键 / URL 字符串                         | URL 字符串        | 松耦合，两个存储互不依赖                                                    |
| Encounter token 存储  | 引用（entityId/blueprintId） / 自包含快照    | 自包含快照        | 无孤儿引用风险，Blueprint/Entity 删除不影响预设                             |
| Scene atmosphere 字段 | 打散的顶层字段 / 合并为 JSON                 | 合并为 JSON       | 减少字段数，语义聚合，GM 单人配置不需要 field-level CRDT                    |

### 已知局限

**Combat token 并发写入**：`combat.tokens` 中每个 token 的值是 plain object（非嵌套 Y.Map），同一个 token 的并发写入会 last-write-wins。但在实践中这不构成问题——token 只存储位置和视觉引用（`entityId`、`x`、`y`、`size`），所有数值数据（HP、属性等）存储在 `entities` Y.Map 中（有 field-level CRDT），两者写入不同的 Y.Map，互不干扰。唯一可能的并发场景是两个人同时拖动同一个 token，这在实际使用中几乎不会发生。
