# 安全与权限系统

## 背景

Yjs 的 CRDT 设计中所有客户端平等，任何客户端都可以绕过 UI 直接往 `chat_log` 写入任意数据。Yjs 协议层面没有"拒绝某条 update"的能力，因此预防不可能，只能检测 + 响应。

但 Yjs 提供了一个关键的底层机制：**Transaction Origin**。`Y.applyUpdate(doc, update, origin)` 的第三个参数会传递到 observer 的 `transaction.origin` 中，而 y-websocket 原生地将 WebSocket 连接对象（`conn`）作为 origin 传入。结合 WebSocket 鉴权，服务端可以为每条 Y.Doc 修改绑定不可伪造的真实身份。

## 核心机制：transaction.origin 信任链

```
WebSocket 鉴权（连接时验证身份）
  ↓ 挂载到连接对象
conn.userId = verifiedUserId
  ↓ y-websocket 原生传递（utils.cjs 第189行）
syncProtocol.readSyncMessage(decoder, encoder, doc, conn)
  ↓ y-protocols 传递（sync.js 第84行）
Y.applyUpdate(doc, update, conn)
  ↓ observer 回调
transaction.origin === conn  →  conn.userId = 真实身份
```

服务端自己写入时使用 `doc.transact(() => { ... }, 'server')`，origin 为 `'server'`。

这条信任链的关键：**客户端无法触碰 origin 的赋值过程**——它发生在服务端内存中，由 y-websocket 在 `applyUpdate` 时强制绑定。

## 总体原则

- **用户无感**：正常情况下，用户完全不知道防伪系统的存在。不显示"已验证"、不显示 ✓ 徽章、不显示任何验证状态。唯一例外：确认伪造时显示 ⚠ 警告
- **验证是基础设施，不是功能**：后台静默进行，不影响任何交互流程
- **兼容旧消息**：无防伪字段的历史消息正常展示，不验证不标记
- **不限流**：所有操作结果对所有人可见，社交约束优先于技术限制
- **渲染不等验证**：消息到达后立即渲染，审计在服务端异步进行

## 统一审计模型

四个议题中的前三个共享同一个服务端 observer，通过 `transaction.origin` 区分来源：

```
chat_log observer(event, transaction):
  origin = transaction.origin
  isServer = (origin === null || origin === 'server')
  conn = isServer ? null : origin
  realUserId = conn?.userId

  对于每条新增消息：
    ├─ type: 'roll' && !isServer     → 🚨 骰子只有服务端能写，伪造！
    ├─ type: 'text' && senderId !== realUserId  → 🚨 身份冒充！
    ├─ type: 'retract' → 验证撤回权限（senderId 匹配或 GM）
    └─ type: 'system' && !isServer   → 🚨 系统消息只有服务端能写！

  对于删除事件：
    → 🚨 chat_log 是 append-only，任何删除 = 非法篡改
```

## 议题

### 一、骰子防伪（服务端权威 + transaction.origin 审计）

骰子由服务端生成，通过 transaction.origin 天然防伪——服务端写入的消息 origin 为 `'server'`，客户端伪造的骰子 origin 是 conn 对象，一个判断即可区分。

- 方案：服务端掷骰 + transaction.origin 区分来源 + 客户端去重（防重放）
- 详见：[51-骰子防伪方案设计.md](51-骰子防伪方案设计.md)

### 二、文本消息身份验证（transaction.origin 审计 + 身份系统）

文本消息由客户端直写 Y.Doc（保持即时性），服务端 observer 通过 transaction.origin 追踪真实写入者，与消息中声称的 senderId 比对，不匹配即为冒充。

身份验证的前提是 WebSocket 连接绑定了可信身份。通过服务器级账号系统（用户名 + 密码）+ JWT cookie + WebSocket 鉴权实现信任链闭合。

- 方案：服务器级账号 + JWT HttpOnly cookie + WebSocket 鉴权 + transaction.origin 审计
- 详见：[53-身份系统与WebSocket鉴权方案设计.md](53-身份系统与WebSocket鉴权方案设计.md)

### 三、聊天记录完整性（append-only + 服务端审计）

chat_log 定义为 append-only：所有合法操作（发消息、撤回）都是追加，不存在合法的删除。服务端 observe chat_log，审计每条消息并检测非法删除。

- 方案：append-only chat_log + 服务端删除检测 + 撤回即追加 + 草稿与记录分离（awareness vs chat_log）
- 详见：[52-聊天记录完整性方案设计.md](52-聊天记录完整性方案设计.md)

### 四、权限与数据隔离（备团区 + 加密字段 + 身份系统）

当前所有数据在单一 Y.Doc 中，玩家打开 F12 即可看到 GM 隐藏的 NPC 属性、怪物真实 HP 等机密数据。通过两层保护解决：备团区（服务端 JSON 存储，玩家完全不可达）物理隔离 GM 的预备资料；加密字段（AES-GCM）让 Y.Doc 中的临场秘密在 F12 下只显示密文。

权限检查复用身份系统的 JWT cookie + 服务端角色查询，不引入额外认证机制。

- 方案：备团区（服务端 JSON）+ AES-GCM 加密字段 + JWT 角色授权 + 单 Y.Doc 架构
- 详见：[54-权限与数据隔离方案设计.md](54-权限与数据隔离方案设计.md)
