# 40 — 数据层架构现状分析与挑战

> 本文档是数据层重构的**前置分析**，目的是在做底层架构决策之前，系统性地梳理当前技术现状、识别核心问题、评估约束条件。

---

## 一、当前架构总览

### 1.1 技术栈

```
┌─────────────────────────────────────────────┐
│                  客户端                       │
│  React 19 + TypeScript 5.9 + Vite 7         │
│  zustand (状态管理) + react-konva (战术地图)  │
│  Tailwind CSS v4 (样式)                      │
├─────────────────────────────────────────────┤
│              实时同步层                       │
│  Yjs v13 (CRDT) + y-websocket v2            │
│  WebSocket 双向同步                          │
├─────────────────────────────────────────────┤
│                 服务端                        │
│  Express 5 + Node.js                        │
│  y-leveldb (Y.Doc 持久化)                    │
│  multer (文件上传 → 磁盘)                    │
└─────────────────────────────────────────────┘
```

### 1.2 数据流架构

```
用户操作 → zustand action → 写入 Y.Map/Y.Array
    ↓
Y.Doc 通过 WebSocket 广播给所有客户端
    ↓
zustand observer 读取 Y.Map → set() 更新 React 状态
    ↓
组件通过 selector 订阅 → 渲染
```

**核心特征**：Yjs 既是实时同步引擎，也是唯一的数据持久化层（通过 y-leveldb）。

### 1.3 Y.Doc 完整结构

```
Y.Doc (房间级别，一个房间一个 Y.Doc)
├── room: Y.Map               ← 房间运行时状态 (activeSceneId, pinnedShowcaseId)
├── scenes: Y.Map<Y.Map>      ← 场景定义 + entityIds + tokens + grid 配置
├── entities: Y.Map<Y.Map>    ← 全局实体存储 (PC + NPC)，字段级 CRDT
├── blueprints: Y.Map         ← Token 模板 (plain objects)
├── seats: Y.Map              ← 玩家座位 (plain objects)
├── chat_log: Y.Array         ← 聊天消息
├── showcase_items: Y.Map     ← 展示物品
├── handout_assets: Y.Map     ← 讲义资源库
└── team_metrics: Y.Map       ← 团队追踪器
```

### 1.4 服务端 API

| 端点                            | 功能             | 备注                       |
| ------------------------------- | ---------------- | -------------------------- |
| `ws://{host}/{roomId}`          | y-websocket 连接 | 所有游戏数据的实时同步通道 |
| `POST /api/upload`              | 文件上传         | multer → 磁盘，返回 URL    |
| `DELETE /api/uploads/:filename` | 删除文件         | 直接删磁盘文件             |
| `GET /api/rooms`                | 房间列表         | rooms.json 文件            |
| `POST /api/rooms`               | 创建房间         | 写 rooms.json              |
| `DELETE /api/rooms/:id`         | 删除房间         | 清 LevelDB + rooms.json    |

**关键观察**：服务端几乎没有业务逻辑——它只是一个 Y.Doc 中继 + 文件存储。所有游戏逻辑都在客户端。

## 二、核心问题识别

### 问题 1：Gallery 与 Scene 的概念耦合

**现状**：Gallery Tab 里的每张图片实际上是一个完整的 `Scene` 对象。上传一张图片 = 创建一个 Scene（包含 grid 配置、entityIds、tokens Y.Map、音频设置等 20+ 个字段）。

**影响**：

- **语义混乱**：GM 上传一张"酒馆外景"图片，系统创建了一个"场景"，但 GM 可能只想把它当背景素材备用
- **数据冗余**：每张图片都携带一整套不需要的默认配置（gridSize: 50, gridSnap: true, ambientPreset: 'none'...）
- **功能受限**：同一张图片不能同时被用于不同场景的背景/战术地图，因为它绑定在一个特定 Scene 上
- **清理困难**：删除"Gallery 中的图片"= 删除一个 Scene，可能触发 entity GC

### 问题 2：缺乏独立的素材管理体系

**现状**：三类素材分散在不同的存储中，没有统一的管理模式。

| 素材类型    | 存储位置               | 元数据结构                 | 管理方式        |
| ----------- | ---------------------- | -------------------------- | --------------- |
| 地图/背景图 | `scenes` Y.Map         | Scene 接口 (20+ 字段)      | 作为 Scene 管理 |
| Token 模板  | `blueprints` Y.Map     | Blueprint 接口 (6 字段)    | 独立管理        |
| 讲义        | `handout_assets` Y.Map | HandoutAsset 接口 (5 字段) | 独立管理        |

**影响**：

- 无法对素材进行统一的标签/分类/搜索
- 不同类型素材的 CRUD 模式不一致（Scene 有 field-level CRDT，Blueprint 是 last-write-wins）
- 素材之间不能交叉引用（一张图片不能既是背景又是讲义展示）

### 问题 3：战斗配置与场景强绑定

**现状**：每个 Scene 只能关联一个战术地图（`tacticalMapImageUrl`），token 配置（`tokens` Y.Map）和 grid 配置直接嵌在 Scene 中。

**影响**：

- 一个叙事场景（如"黑龙酒馆"）无法预设多个战斗区域（一楼、二楼、地下室）
- 切换战术地图 = 覆盖 `tacticalMapImageUrl`，之前的 token 布局丢失
- GM 无法在 prep 阶段预先配置多个战斗场景

### 问题 4：所有数据都塞在 Y.Doc 中

**现状**：从素材元数据到运行时状态，所有数据都存储在同一个 Y.Doc 里，通过 y-leveldb 持久化。

**技术债务**：

| 问题               | 说明                                                                                 |
| ------------------ | ------------------------------------------------------------------------------------ |
| **Y.Doc 只增不减** | CRDT tombstone 机制意味着删除的数据仍占空间。随着素材增多，Y.Doc 体积持续膨胀        |
| **全量同步**       | 每个客户端连接时必须同步完整 Y.Doc，包括所有素材元数据、所有场景配置、所有历史 token |
| **无查询能力**     | 不能按标签过滤、分页、排序——只能全量读入内存后在客户端过滤                           |
| **后端不可替换**   | 数据结构与 Yjs API 深度耦合，换后端（如 PostgreSQL、Supabase）需要重写整个数据层     |
| **无备份/导出**    | LevelDB 二进制格式，无法直接导出/导入游戏数据                                        |
| **无访问控制**     | 所有连接的客户端都能读写所有数据（双文档架构尚未实现）                               |

**哪些数据真正需要 Yjs 实时同步**：

| 数据                          | 需要实时同步？ | 原因                                         |
| ----------------------------- | -------------- | -------------------------------------------- |
| Token 位置                    | ✅ 强需求      | 多人同时拖动 token，需要毫秒级同步           |
| 聊天消息                      | ✅ 强需求      | 实时对话                                     |
| 战斗状态 (先攻/回合)          | ✅ 强需求      | 所有人需要看到当前回合                       |
| Entity 数值 (HP 等)           | ✅ 中等需求    | GM 和玩家可能同时编辑，但频率不高            |
| 当前激活场景                  | ✅ 中等需求    | 场景切换时所有人同步，但不频繁               |
| 素材库 (图片/模板/讲义)       | ❌ 弱需求      | 通常只有 GM 在 prep 阶段管理，不存在并发编辑 |
| 场景静态配置 (名称/背景/音频) | ❌ 弱需求      | 只有 GM 配置，配置完成后长期不变             |
| 战斗预设 (地图+token 布局)    | ❌ 弱需求      | prep 阶段由 GM 单人配置                      |

### 问题 5：文件资源缺乏管理

**现状**：上传的文件存储在 `server/uploads/` 目录，只有一个扁平的 UUID 命名的文件列表。

**影响**：

- 没有引用计数——无法知道哪些文件还在被使用，哪些是孤立的
- 没有文件去重——同一张图片上传多次会产生多个副本
- 没有缩略图——Gallery 直接加载原图作为缩略图
- 删除素材时不会自动清理磁盘文件

| ---------------------- | ------------------------ | ----------------------------------------------- |

| **Entity 全局存储**    | `01-场景与NPC组织.md`    | Entity 存储模式已确立，不应改变                 |
| **权限模型**           | `04-权限模型.md`         | EntityPermissions 结构已确立                    |
| **Blueprint 渐进增强** | `02-素材库与模板系统.md` | "上传→放置→添加数据→保存为模板"的流程已设计     |

### 3.2 产品原则约束

来自 `00-产品愿景与设计理念.md`：

1. **轻备团优先** — 任何新的数据结构不能增加 GM 的准备负担
2. **氛围感至上** — 场景切换、战斗地图加载必须流畅，不能因数据层变更导致加载延迟
3. **直觉式交互** — 素材管理不需要 GM 理解底层数据模型

### 3.3 技术约束

| 约束               | 说明                                              |
| ------------------ | ------------------------------------------------- |
| **y-websocket v2** | 不能升级到 v3（服务端 API 不兼容）                |
| **Yjs v13**        | 不能升级到 v14（y-leveldb 不兼容）                |
| **单服务器部署**   | 当前没有独立数据库服务（只有 LevelDB + 文件系统） |
| **无身份认证**     | 当前没有用户系统，只有房间级的座位概念            |
| **渐进式迁移**     | 不能一次性推翻重来，需要兼容已有的 Y.Doc 数据     |

## 四、需要解决的设计议题

基于以上分析，后续设计讨论需要回答以下问题：

### 议题 A：素材管理体系

1. 是否引入独立于 Yjs 的素材存储层（REST API + 持久化）？
2. 素材的分类/标签方案：固定类别、自由标签、还是用途驱动的视图？
3. 不同类型素材（地图、token、讲义）的元数据结构如何统一又保持各自特性？
4. 素材与游戏运行时数据（Scene、Entity）的引用关系如何设计？

### 议题 B：场景与战斗配置解耦

1. 战斗预设（Encounter）的数据结构：地图 + grid + token 布局 + 先攻预设？
2. Scene 与 Encounter 的一对多关系如何建模？
3. Encounter 激活时如何与 Yjs 运行时状态（tokens Y.Map）同步？
4. 是否保留当前 Scene 内嵌 tokens 的模式用于"临时战斗"？

### 议题 C：数据层分层架构

1. 哪些数据应该从 Yjs 迁移到 REST API？
2. REST API 的持久化选择：SQLite？继续用 LevelDB？JSON 文件？
3. 客户端数据层如何同时消费 REST API 和 Yjs 两个数据源？
4. 如何与已规划的双文档架构兼容？
5. 数据迁移策略：如何处理已有 Y.Doc 中的存量数据？

### 议题 D：文件资源管理

1. 是否引入文件引用计数和孤立文件清理？
2. 是否实现缩略图生成？
3. 文件去重策略（content hash？）

## 五、现状评估总结

**当前架构的优势**：

- 极简部署（单进程、无外部依赖）
- 实时协同开箱即用（Yjs + WebSocket）
- 离线容错（Yjs 本地缓存 + 断线重连自动合并）

**当前架构的核心矛盾**：

- **Yjs 被用作通用数据库**，但它的设计目标是实时协同编辑引擎
- **所有数据共享一个同步通道**，导致静态素材数据和实时游戏状态耦合
- **缺乏查询和管理能力**，随着素材增长，性能和可用性都会下降

**重构的核心目标**：

> 将 Yjs 回归其本职——实时协同状态同步，将静态/低频数据（素材、配置、预设）迁移到更适合 CRUD 的持久化层，同时保持当前极简部署和实时体验的优势。
