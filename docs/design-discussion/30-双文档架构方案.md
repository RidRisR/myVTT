# 双文档架构方案：Public Doc + GM Secret Doc

> 本文档记录将单 Y.Doc 拆分为公共文档 + GM 机密文档的架构设计，包括数据分区、认证方案、跨文档操作权衡分析。

---

## 一、问题陈述

### 当前安全模型

myVTT 目前所有数据存放在**单一 Y.Doc** 中。y-websocket 将完整文档同步给所有连接的客户端。

- 权限检查（`canSee`、`canEdit`）全部在**前端 UI 层过滤**
- 服务器**零身份验证**——任何拿到房间 URL 的人拥有完整读写权限
- 玩家打开 F12 可以看到所有"隐藏" NPC 属性、陷阱坐标、怪物真实 HP

### 目标

拆分为两个 Y.Doc：

| 文档              | 连接者        | 内容                                            |
| ----------------- | ------------- | ----------------------------------------------- |
| **Public Doc**    | GM + 所有玩家 | 所有玩家可见的数据                              |
| **GM Secret Doc** | 仅 GM         | GM 的机密数据（隐藏 NPC、真实数值、私密笔记等） |

服务器在 WebSocket 层面**拒绝玩家连接 Secret Doc**，物理上阻断数据泄露。

## 二、公开文档的数据结构是否需要调整？

### 结论：结构格式不变，语义有细微变化

当前公开文档的 9 个顶层 key 全部保留，Y.Map 嵌套方式完全不变：

```
scenes        Y.Map<Y.Map>     场景配置 + entityIds + tokens
entities      Y.Map<Y.Map>     实体（嵌套 permissions/ruleData Y.Map）
blueprints    Y.Map            Token 模板
seats         Y.Map            座位/角色
room          Y.Map            房间状态（activeSceneId 等）
chat_log      Y.Array          聊天记录
showcase_items Y.Map           展示内容
handout_assets Y.Map           讲义资源
team_metrics  Y.Map            团队追踪器
```

**不需要新增字段、不需要修改嵌套结构、不需要改变类型签名。**

### 变化的是什么？

1. **实体的"存在范围"变了**：目前 `permissions.default === 'none'` 的隐藏实体也在同一个文档里（只是前端过滤不显示）。拆分后，完全隐藏的实体**物理上不存在于 public doc 中**——它们住在 secret doc 里。

2. **"模糊 HP"场景下 ruleData 的语义变了**：

   |                                            | 拆分前                                        | 拆分后                                                  |
   | ------------------------------------------ | --------------------------------------------- | ------------------------------------------------------- |
   | `entities["goblin"].ruleData.resources.hp` | `{ cur: 7, max: 20 }` ← 真实值，玩家 F12 可见 | `{ cur: 0, max: 0 }` ← 清零或省略                       |
   | `entities["goblin"].ruleData.statuses`     | 可能有也可能没有                              | `[{ label: "重伤" }]` ← 模糊描述                        |
   | 真实 HP 存储位置                           | 同一个文档                                    | secret doc 的 `entity_secrets["goblin"].hp_current = 7` |

   数据**格式**完全一样（仍然是 `{ cur, max }` 和 `[{ label }]`），但 **GM 需要额外维护 secret 覆盖层**。

3. **对现有代码的影响极小**：
   - `useEntities.ts` — 不需要修改，继续从 `world.entities` 读写
   - `useScenes.ts` — 不需要修改
   - `useSceneTokens.ts` — 不需要修改
   - `PortraitBar`、`CharacterDetailPanel` 等组件 — 不需要修改
   - **仅在 RoomSession 层增加一个合并层**，将 secret 数据注入给 GM

## 三、认证方案

### 房间级 Token（推荐）

创建房间时，服务器生成两个随机令牌：

```json
{
  "id": "abc12345",
  "name": "我的战役",
  "gmToken": "gm_abc12345_<32字节hex>",
  "playerToken": "pl_abc12345_<32字节hex>"
}
```

- 令牌存储在 `rooms.json`（与现有元数据并存）
- 通过 URL hash 分发：`#room=abc12345&token=gm_abc12345_...`
- 客户端存入 `sessionStorage` 用于重连
- WebSocket 连接时作为 query param 传递

### 授权矩阵

| Token 类型   | `:public` | `:secret` |
| ------------ | :-------: | :-------: |
| GM token     |    ✅     |    ✅     |
| Player token |    ✅     | ❌ (4403) |
| 无效/缺失    | ❌ (4401) | ❌ (4401) |

### 为什么不用更复杂的认证？

- **无需用户账号系统**：VTT 是临时会话，不需要注册/登录
- **无需 JWT/OAuth**：房间生命周期短，token 轮换即可应对泄露
- **最小改动原则**：仅在 `rooms.json` 增加两个字段

## 四、文档分区详细设计

### 4.1 GM Secret Doc 数据结构

| 顶层 key          | 类型           | 内容                                                        |
| ----------------- | -------------- | ----------------------------------------------------------- |
| `secret_entities` | `Y.Map<Y.Map>` | 完全隐藏的实体（格式同 public `entities`，嵌套 Y.Map 结构） |
| `entity_secrets`  | `Y.Map<Y.Map>` | 覆盖层：按 entityId 索引，存储 public 实体的秘密数据        |
| `gm_notes`        | `Y.Map`        | GM 自由笔记                                                 |
| `secret_chat_log` | `Y.Array`      | GM 私密聊天/暗骰                                            |

### 4.2 实体的三种存在状态

```
┌──────────────────────────────────────────────────────────┐
│ 状态 A：完全公开                                          │
│ 位置：public entities                                     │
│ 例子：玩家的 PC、已暴露的 NPC                               │
│ 玩家看到：完整数据                                         │
│ GM 看到：完整数据                                          │
├──────────────────────────────────────────────────────────┤
│ 状态 B：可见但有秘密                                       │
│ 位置：public entities + secret entity_secrets 覆盖层       │
│ 例子：战场上的怪物（玩家看到模糊 HP，GM 看到真实 HP）          │
│ 玩家看到：public 中的模糊数据                               │
│ GM 看到：合并后的真实数据                                    │
├──────────────────────────────────────────────────────────┤
│ 状态 C：完全隐藏                                          │
│ 位置：secret secret_entities                              │
│ 例子：未出场的 Boss、隐藏陷阱 NPC                           │
│ 玩家看到：不存在（数据从未发送到玩家浏览器）                    │
│ GM 看到：带有"隐藏"标记的实体                               │
└──────────────────────────────────────────────────────────┘
```

### 4.3 状态转换

```
完全隐藏 (C) ──── 现身操作 ────→ 完全公开 (A) 或 可见但有秘密 (B)
                                    │
完全公开 (A) ──── 隐藏操作 ────→ 完全隐藏 (C)
                                    │
可见但有秘密 (B) ── 隐藏操作 ──→ 完全隐藏 (C)
                                    │
完全公开 (A) ─ 添加覆盖层 ──→ 可见但有秘密 (B)
```

## 五、跨文档操作：客户端搬运 vs 服务器搬运

### 关键发现

y-websocket v2 导出了 `getYDoc()` 函数和 `docs` Map。服务器可以直接获取活跃的 Y.Doc 实例，对其调用 `transact()` 修改数据，**修改会自动通过 WebSocket 广播给所有连接的客户端**。

这意味着**服务器端搬运在技术上完全可行**。以下是两种方案的详细对比：

### 方案 A：GM 客户端搬运

```
GM 浏览器
  ├── 连接 public doc (WebsocketProvider)
  ├── 连接 secret doc (WebsocketProvider)
  │
  │ [现身操作]
  │ 1. 从 secret doc 读取实体
  │ 2. 写入 public doc
  │ 3. 删除 secret doc 中的实体
  │
  │ 所有写入通过 Yjs CRDT 自动同步
  └── 玩家即时看到变化
```

**优点：**

- **服务器保持 schema-agnostic**：服务器不需要理解实体的数据结构，只做 Yjs 更新转发
- **实现简单**：GM 前端已经连接两个文档，直接用现有的 `readYMapEntity` / `setYMapFields` 函数
- **零 API 端点**：不需要新的 REST API
- **离线友好**：如果后续支持离线模式，客户端搬运天然兼容

**缺点：**

- **非原子操作**：写入 public 和删除 secret 是两个独立的 Yjs 事务（跨文档不能在一个 `transact()` 中完成）。GM 断网可能导致实体在两个文档中短暂重复
- **业务逻辑在客户端**：所有跨文档规则（如"现身时自动设为可见权限"）在前端代码中
- **GM 客户端是单点**：如果 GM 刷新页面时正好在搬运中途，需要幂等恢复机制

### 方案 B：服务器端 API 搬运

```
GM 浏览器                          服务器
  │                                 │
  ├── POST /api/reveal              │
  │   { roomId, entityId }  ────→   │ 1. getYDoc(roomId:secret)
  │                                 │ 2. 读取实体数据
  │                                 │ 3. getYDoc(roomId:public)
  │                                 │ 4. 写入实体数据
  │                                 │ 5. 删除 secret 中的实体
  │                                 │
  │   ←──── { ok: true }            │ Yjs update 自动广播
  │                                 │
  └── 玩家即时看到变化（通过 WS）     └──
```

**优点：**

- **操作更可靠**：服务器在同一个进程中操作两个文档，不会因为网络中断而只完成一半
- **服务器可以做校验**：比如验证 GM 身份再执行操作（REST API 也带 token）
- **未来可扩展**：如果后续需要更复杂的服务器逻辑（如 HP 伤害计算、自动状态判定），API 模式天然支持
- **GM 客户端更简单**：只需调用一个 API，不需要管双文档操作细节

**缺点：**

- **服务器需要理解 schema**：服务器代码必须知道 Entity 的字段结构、如何创建嵌套 Y.Map、如何写入 permissions/ruleData 等。这把 `readYMapEntity` / `setYMapFields` / `writePermissions` 等函数的逻辑搬到了服务端
- **服务器代码是 CJS**：y-websocket utils 是 CJS，Entity 类型定义是 TypeScript。服务器端需要手写等价的序列化/反序列化逻辑（不能直接 import 前端的 TS 模块）
- **新增 API 端点**：每种跨文档操作（现身、隐藏、更新秘密覆盖层、伤害计算...）都需要一个 REST API
- **文档可能不在内存中**：如果 secret doc 没有 GM 连接（理论上不应该发生，但边界情况），`getYDoc` 需要从 LevelDB 加载，增加延迟

### 推荐：混合方案

| 操作类型                                        | 推荐方案   | 理由                                                                                   |
| ----------------------------------------------- | ---------- | -------------------------------------------------------------------------------------- |
| **现身/隐藏**（实体搬运）                       | 客户端搬运 | 操作简单（读→写→删），幂等性好实现，服务器不需要理解 Entity 结构                       |
| **模糊 HP 更新**（GM 改真实 HP → 更新模糊状态） | 客户端搬运 | 两步写入（secret + public）由同一个 UI 操作触发，天然绑定                              |
| **伤害计算**（如果未来玩家调用 API 造成伤害）   | 服务器 API | 需要服务器读取 secret HP 扣血 → 计算状态 → 写回两个文档，玩家客户端无法访问 secret doc |

**短期内（本次改造）**：纯客户端搬运。理由：

1. 目前所有跨文档操作都由 GM 发起，GM 连接两个文档
2. 避免在服务器中引入 schema 依赖
3. 实现量小（一个 `crossDocOps.ts` 文件）

**长期演进**：当需要"玩家发起的操作影响 secret 数据"时（如玩家对怪物造成伤害，需要扣 secret HP），才引入服务器 API。此时可以把伤害计算逻辑作为规则层插件部署在服务器端。

## 六、WebSocket 路由与服务器连接授权

### 房间名规范

```
{roomId}:public    — 公共文档（如 "abc12345:public"）
{roomId}:secret    — GM 机密文档（如 "abc12345:secret"）
```

### LevelDB 持久化

无需修改。`setPersistence.bindState` 接收完整 `docName`，两个文档名自动作为独立条目存储。

### 服务器连接授权逻辑

```javascript
wss.on('connection', (conn, req) => {
  const url = new URL(req.url, `http://${req.headers.host}`)
  const docPath = url.pathname.slice(1) // "abc12345:public"
  const token = url.searchParams.get('token')

  const [roomId, docType = 'public'] = docPath.split(':')

  // 验证房间
  const room = rooms.find((r) => r.id === roomId)
  if (!room) {
    conn.close(4404, 'Room not found')
    return
  }

  // 验证文档类型
  if (docType !== 'public' && docType !== 'secret') {
    conn.close(4400, 'Invalid doc type')
    return
  }

  // 验证 token
  const isGM = token === room.gmToken
  const isPlayer = token === room.playerToken
  if (!isGM && !isPlayer) {
    conn.close(4401, 'Unauthorized')
    return
  }

  // 授权：仅 GM 可连接 secret
  if (docType === 'secret' && !isGM) {
    conn.close(4403, 'Forbidden')
    return
  }

  setupWSConnection(conn, req, { docName: docPath })
})
```

### 删除房间时清理

```javascript
await ldb.clearDocument(roomId) // 旧数据
await ldb.clearDocument(`${roomId}:public`)
await ldb.clearDocument(`${roomId}:secret`)
```

## 七、客户端改动概览

### 7.1 新建 `useDualDocConnection.ts`

替代 `useYjsConnection.ts`。根据 token 前缀判断是否 GM：

- 所有人连接 `{roomId}:public`
- GM 额外连接 `{roomId}:secret`
- 等待所有文档同步完成后解除 loading

### 7.2 扩展 `useWorld.ts`

新增：

```typescript
interface SecretWorldMaps {
  secretEntities: Y.Map<Y.Map<unknown>> // 完全隐藏的实体
  entitySecrets: Y.Map<Y.Map<unknown>> // 覆盖层
  gmNotes: Y.Map<unknown>
  secretChatLog: Y.Array<ChatMessage>

```

### 7.3 新建合并层 `useGmMergedView.ts`

**核心设计**：组件树下方完全不感知双文档的存在。

```typescript
// GM 看到的统一实体列表
const allEntities = useGmMergedView(publicEntities, secretEntities, entitySecrets)

```

### 7.4 新建 `crossDocOps.ts`

`revealEntity()` 和 `hideEntity()` 函数，处理实体在两个文档间的搬运。

### 7.5 修改 `App.tsx`

- URL hash regex 更新：支持 `token` 参数
- 使用 `useDualDocConnection` 替代 `useYjsConnection`

### 7.6 修改 `AdminPanel.tsx`

创建房间后显示 GM 链接和玩家链接。

## 八、数据迁移

### 阶段 1：新旧兼容

- `rooms.json` 中已有房间自动补充 token
- 过渡期允许无 token 连接 `:public`（日志警告）
- `:secret` 始终要求 GM token

### 阶段 2：LevelDB 文档迁移

`bindState` 中增加一次性迁移：加载 `{roomId}:public` 时若不存在，从裸 `{roomId}` 复制数据。

### 阶段 3：强制认证

移除过渡兼容，所有连接必须携带有效 token。

## 九、风险与缓解

| 风险                     | 缓解                                            |
| ------------------------ | ----------------------------------------------- |
| 搬运中途断网导致实体重复 | 操作幂等 + 启动时检查清理                       |
| 模糊 HP 两步写入不同步   | 封装在单函数中；最坏情况=状态文本过时（不泄密） |
| GM 双 WebSocket 连接性能 | Secret doc 很小，开销可忽略                     |
| Token 泄露               | 可增加 token 轮换 API                           |
