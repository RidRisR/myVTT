# 服务端权威骰子系统

## 背景与动机

当前骰子完全在客户端执行：`ChatInput.handleRoll()` → `rollCompound()` → `yChat.push([message])`。由于 Yjs 的 CRDT 设计中所有客户端平等，任何客户端都可以绕过 UI 直接往 `chat_log` 写入伪造的骰子结果。Yjs 协议层面没有"拒绝某条 update"的能力，也没有消息级别的元信息（如写入者身份）可供审计。

对于 TRPG 中关乎角色生死的骰子判定，需要服务端权威的投骰机制来保证公正性。

## 架构概览

### 三条独立的线

```
线1 投骰（客户端 → 服务端 → Y.Doc）
    玩家输入 .r 1d20
    → 客户端解析 @key 引用（需要 token/seat 属性上下文）
    → POST /api/rooms/:roomId/roll（发送 formula + 身份信息）
    → 服务端生成随机结果 + HMAC 签名 + rollId 注册
    → 服务端写入 Y.Doc chat_log
    → Yjs 自动同步给所有客户端

线2 渲染（Y.Doc → UI）
    客户端 Yjs observer 收到新 roll 消息
    → 立即渲染骰子动画（不等验证）
    → 同时将该消息加入验证队列

线3 验证（客户端 → 服务端，批量异步）
    每 50-100ms flush 一次验证队列
    → POST /api/rooms/:roomId/verify-rolls（批量发送）
    → 服务端对每条消息：重算 HMAC 对比签名 + 检查 rollId 存在性
    → 返回结果 → 客户端异步更新 ✓ 或 ⚠ 标记
```

投骰请求和验证请求完全解耦，互不阻塞。消息该发就发，验证按固定间隔批量处理。

### 安全模型

```
服务端持有（永不泄露）          客户端持有
─────────────────────     ──────────────
SECRET_KEY（内存）           签名（可见但无法伪造）
rollRegistry（Set<string>）  消息内容
```

**每个客户端独立验证，只信任自己的验证结果。** 作弊者最多只能骗自己的屏幕，其他人都会看到未验证警告。

## 安全机制详解

### HMAC 签名（防伪造 + 防篡改）

服务端启动时生成 `SECRET_KEY = crypto.randomBytes(32)`，存内存，永不发送给客户端。

每次投骰，服务端计算：

```
signature = HMAC-SHA256(SECRET_KEY, rollId + resolvedExpression + JSON(termResults) + total + timestamp)
```

签名与所有内容字段绑定。客户端没有密钥，无法伪造签名；修改任何字段（rollId、骰值、总数等）都会导致签名失效。

### rollRegistry（防重放）

纯内存 `Set<string>`，存储所有由当前进程生成的 rollId。

**重放攻击场景：** 攻击者复制一条合法的高点数消息（含有效签名），再次 push 到 chat_log。防御：验证时检查 rollId 是否存在于 registry——同一 rollId 只对应一次投骰。

**为什么不需要持久化：** SECRET_KEY 和 rollRegistry 生命周期一致，都在服务端进程内存中。服务端重启后 SECRET_KEY 重新生成，旧签名在新密钥下验证不过，rollId 集合自然也不需要保留。

**内存增长：** 一个 rollId 约 100 字节。4 小时 session、每分钟 5 次投骰 = 1200 条 ≈ 120KB。完全可忽略。

### 攻击场景分析

| 攻击方式             | 过程                                  | 结果                                       |
| -------------------- | ------------------------------------- | ------------------------------------------ |
| **凭空伪造**         | push 带假 signature 的消息到 chat_log | 服务端重算 HMAC 不匹配 → ⚠                 |
| **篡改数字**         | 复制合法消息，改 total 从 5 到 20     | 签名与内容绑定，改数字签名失效 → ⚠         |
| **重放旧结果**       | 复制一条 total=20 的合法消息再次 push | rollId 已存在，或密钥已换 → ⚠              |
| **刷骰子取最优**     | 连续发多次 roll 请求                  | 不限流，所有结果对所有人可见，刷骰行为自证 |
| **跳过本地验证**     | 修改客户端代码不调验证接口            | 只影响自己屏幕，其他人独立验证不受影响     |
| **服务端重启后重放** | 拿旧消息的签名在新进程中验证          | SECRET_KEY 已换，HMAC 不匹配 → ⚠           |

## 设计决策记录

### 为什么不在客户端验证签名？

HMAC 是对称加密——验证需要同一个密钥。如果把密钥给客户端，任何人都能伪造签名。非对称签名（如 Ed25519）可以实现客户端自主验证，但对 TRPG 场景来说过度工程化。当前方案中客户端通过 HTTP 请求服务端验证，延迟可忽略。

### 为什么不限流？

刷骰子的结果全部写入 chat_log，对所有人可见。这本身就是社交约束——GM 和其他玩家看到某人连投 10 次骰子，自然会质疑。技术限流反而会影响手感（玩家快速连投时卡顿）。

### 为什么 serverVerified 字段不可信？

Yjs 中任何客户端都可以写入任意字段。攻击者可以 push 一条 `{ serverVerified: true, signature: "fake" }` 的消息。所以 `serverVerified` 只是一个提示标记，真正的信任来源是 HTTP 验证端点对 HMAC 签名的校验。

### 为什么用批量验证而不是逐条验证？

服务端自动投骰（如 NPC 批量行动）可能一次性产生几十条 roll 消息。逐条验证会产生大量 HTTP 请求。50-100ms 的 interval flush 将同一窗口内的消息合并为一次请求，减少网络开销，且延迟对用户不可感知（骰子动画本身 ~1.5 秒）。

### 文本消息为什么不走服务端？

文本消息没有"公正性"需求，伪造文本消息的动机和影响远低于伪造骰子。保持 Yjs 直接写入，避免不必要的复杂度。

## 不变的部分

- 文本消息（`type: 'text'`）仍走 Yjs 直接写入
- 骰子动画逻辑不变（DiceReel 组件）
- 旧消息（无 `signature` 字段）正常展示，不验证不标记
- `diceUtils.ts` 的类型导出保持不变
