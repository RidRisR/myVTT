### 待改进方向（从代码审计与 UX 审查中发现）

**基础可用性缺陷:**

1. 🔴 **破坏性操作无确认**: 删除场景/Token/角色直接执行，无 confirm 对话框
2. 🔴 **操作反馈缺失**: 添加 Token、保存修改、上传完成等操作静默完成，无 toast/视觉反馈
3. 🔴 **加载状态缺失**: 图片上传、WebSocket 连接中无进度指示，UI 可能看似卡死
4. 🔴 **断线无提示**: WebSocket 断开时无任何 UI 反馈
5. 🔴 **空状态无引导**: 新房间零内容时不知从何开始

**功能缺口:** 6. 🟡 **Initiative 追踪**: PortraitBar 中标注 "Coming soon" 7. 🟡 **测量工具**: 路线图 Milestone 4 核心，未开始 8. 🟡 **光标同步**: 路线图中提到，未开始

**体验优化:** 9. 🟢 **键盘快捷键**: 仅 Tab/Enter/Escape，无帮助面板 10. 🟢 **响应式**: 100% 桌面设计，小屏体验差 11. 🟢 **无障碍**: 几乎无 ARIA 标签 12. 🟢 **动画不统一**: fade vs slide 混用，无统一过渡规范

**代码审查新发现（本次修订新增）:** 13. 🔴 **语义化 HTML 缺失**: 大量 `<div onClick>` 反模式，按钮未用 `<button>`，列表未用 `<ul>/<li>`，表单输入无 `<label>` 14. 🔴 **焦点管理缺失**: 弹窗无焦点陷阱、关闭后焦点未归还、无 focus-visible 样式 15. 🔴 **Error Boundary 缺失**: 任一组件崩溃导致全页白屏 16. 🟡 **渲染性能问题**: 全量数组 O(n) 重建、无 React.memo、prop drilling 导致整棵树重渲染 17. 🟡 **内联样式根本限制**: 无法支持 `:hover`/`:focus-visible` 伪类、`prefers-reduced-motion` 媒体查询、`@keyframes` 动画18. 🟡 **SVG 图标无统一体系**: 30+ 组件各自内联 `<svg>`，无统一尺寸/样式19. 🟢 **URL 不反映状态**: 仅 `#room=xxx`，活跃场景/面板状态/战斗状态不在 URL 20. 🟢 **`color-scheme: dark` 缺失**: 原生滚动条/表单控件颜色不匹配深色主题

## 跨议题关切（Cross-Cutting Concerns）

以下问题影响多个议题，需要在讨论早期（议题 11）形成共识：

### 1. 内联样式的根本限制

**影响议题**: 11, 17, 18

内联样式 **无法实现**:

- `:hover`, `:focus-visible`, `:active` 伪类
- `@media (prefers-reduced-motion)` 查询
- `@media (max-width)` 响应式断点
- CSS 动画 `@keyframes`（当前用 JS 注入 style 标签绕过）
- `::-webkit-scrollbar` 滚动条定制

**决策点**: 是否保持 100% 内联，还是引入混合方案（全局 CSS 处理伪类/媒体查询 + 内联处理动态值）？这直接决定后续所有 UI 改动的可行性。

### 2. 语义化 HTML

**影响议题**: 12, 14, 16, 18

代码审查发现的反模式清单：

| 反模式                           | 正确做法                      | 影响组件（举例）                                |
| -------------------------------- | ----------------------------- | ----------------------------------------------- |
| `<div onClick>` 作为按钮         | `<button>`                    | HamburgerMenu, BottomDock tabs, GmToolbar       |
| `<div>` 包裹点击区域             | `<button>` with full hit area | PortraitBar portraits, ChatPanel speaker picker |
| 表单输入无 `<label>`             | `<label htmlFor>` 或包裹      | ChatInput, MyCharacterCard name edit            |
| Icon-only 按钮无 `aria-label`    | 添加 `aria-label`             | 所有 SVG icon 按钮                              |
| 列表无语义标签                   | `<ul>/<li>`                   | SceneLibrary scene list, Blueprint list         |
| `outline: 'none'` 无替代焦点样式 | `focus-visible` ring          | 全部 input 元素                                 |
| 弹窗无焦点陷阱                   | `focus-trap` 或 `inert`       | ShowcaseOverlay, SceneLibrary modal             |

这是 **Web Interface Guidelines 的硬性要求**，需要在 UI 重设计中系统性修复。

### 3. 焦点管理

**影响议题**: 12, 13, 14, 16

- 打开弹窗/面板时焦点未移入
- 关闭弹窗时焦点未返回触发元素
- Tab 键在弹窗内无焦点陷阱（focus trap）
- 无 skip-to-content 链接

### 4. 国际化准备

**影响议题**: 11, 12

- 当前所有 UI 文本硬编码在组件中
- 如果未来需要支持多语言，现在的架构需要 i18n 抽取
- 数字/日期格式未使用 `Intl` API

## 议题清单

### 议题 11: 样式基础设施与设计 Token ⭐ 扩展版

> 在开始任何 UI 改动之前：当前 100% 内联样式是否可持续？是否需要引入设计 token 或 CSS 方案？**本议题的决策直接影响后续所有议题的实现方式。**

**待讨论要点:**

- 当前内联样式的痛点：颜色魔数散落在 30+ 组件中，无法全局调整
- 是否引入设计 token（CSS 变量 / JS 常量对象）
- 是否引入 CSS 方案（CSS Modules / Tailwind / 继续内联）
- z-index 分层策略（当前 10000+ 会出现层叠上下文问题）
- 动画/过渡的统一规范（duration、easing、方向）

**审查补充要点:**

- 🆕 **`color-scheme: dark` 与 `<meta name="theme-color">`**: 未在 `<html>` 设置，导致原生滚动条/表单控件/移动端标题栏颜色不匹配
- 🆕 **内联样式 vs 伪类/媒体查询的根本矛盾**: 内联无法实现 `:focus-visible`（无障碍硬性要求）和 `prefers-reduced-motion`（WCAG 要求），是否引入全局 CSS 层处理
- 🆕 **`transition: all` 反模式**: 代码中多处使用 `transition: 'all 0.2s'`，应显式列出属性（Web Interface Guidelines 规定仅动画 transform/opacity）
- 🆕 **SVG 图标统一体系**: 当前 30+ 组件各自内联 `<svg>`，应建立图标组件库（统一 viewBox、size、stroke-width）
- 🆕 **设计 token 范围扩展**: 除颜色外，还应包含 spacing scale、border-radius scale、shadow scale
- 🆕 **WCAG AA 对比度基线**: 从议题 17 前移至此，作为基础设施的一部分

### 议题 12: 操作反馈与容错 ⭐ 扩展版

> 当前 UI 在操作反馈、错误处理、确认对话框方面几乎为零。这是基础可用性问题。

**待讨论要点:**

- 破坏性操作确认策略（删除场景/角色/Token 前的 confirm）
- 全局 toast 通知系统（成功/错误/警告，自动消失时间，堆叠）
- 加载状态方案（skeleton / spinner / progress bar，各场景适用哪种）
- WebSocket 连接状态指示（在线/重连中/离线）
- 空状态引导（新房间/空场景/无角色时的 CTA 提示）

**审查补充要点:**

- 🆕 **Undo/Redo 机制**: Web Guidelines 建议破坏性操作提供 undo 窗口而非仅 confirm。Yjs 天然支持 `UndoManager`，应考虑利用
- 🆕 **表单验证反馈**: 角色卡编辑、场景配置等输入无验证反馈（如名称为空、数值越界）
- 🆕 **乐观更新 (Optimistic UI)**: 当前所有操作等待 Yjs observer 回调才更新 UI（50-200ms 延迟），应讨论是否引入乐观更新模式
- 🆕 **React Error Boundary**: 无 Error Boundary，任一组件崩溃会白屏
- 🆕 **`aria-live` 区域**: Toast 通知必须使用 `aria-live="polite"` 才能被屏幕阅读器感知
- 🆕 **离线模式策略**: WebSocket 断线后用户能否继续操作？断线期间的编辑如何处理？
- 🆕 **键盘快捷键体系**: 从议题 18 移入，包含快捷键帮助面板（`?` 触发）

### 议题 13: GM 工作流与战斗管理

> GM 的完整操作路径：从准备阶段（上传地图、创建角色）到游戏运行阶段（战斗、Initiative、回合管理）。

**待讨论要点:**

- 场景准备流程：上传地图 → 配置网格 → 放置 Token → 开始战斗的完整路径优化
- NPC 快速创建：右键菜单 vs 底部工具栏 vs 从素材库拖放
- Initiative 系统：排序条 UI、当前回合指示、延迟/就绪、自动排序
- 回合管理：下一回合按钮、回合计数、条件自动到期
- 场景切换过渡（当前硬切换 vs 淡入淡出）
- GM 笔记/私有信息管理

**审查补充要点:**

- 🆕 **GM 视角 vs 玩家视角预览**: GM 需要能预览"玩家看到的样子"（哪些 token 可见、哪些信息隐藏）
- 🆕 **场景元数据管理**: 场景名称、描述、背景音乐链接等元数据编辑入口不明显
- 🆕 **战斗开始/结束的状态转换**: 从场景模式切换到战斗模式目前是硬切换，需要过渡效果
- 🆕 **NPC 模板快速编辑**: 从素材库放置 NPC 后，应能快速编辑其属性而非必须打开完整角色卡
- 🆕 **批量 NPC 生成**: 常见需求：一次放置 5 个同类型 NPC（如"5 个哥布林"）
- 🆕 **GM vs 玩家布局差异**: 从议题 18 移入，GM 工具栏/面板的独有布局

### 议题 19: 渲染性能与状态管理 🆕

> 代码审查发现严重的渲染性能问题，在 20+ token、10+ 场景下会明显卡顿。需要在功能扩展前建立性能基线。

**待讨论要点:**

- **全量数组重建**: 每次 Yjs observer 触发，`useEntities`/`useScenes`/`useSceneTokens` 都执行 O(n) 全量 rebuild，即使只有 1 个 entity 变化。是否引入 per-entity observer 或 diff 机制
- **React.memo 策略**: 所有组件无 memo 包装，任何 state 变化导致整棵树重渲染。是否对 PortraitBar item、MapToken 等列表项引入 `React.memo` + 细粒度 key
- **Prop Drilling vs Context**: App.tsx 将 entities/scenes/tokens 全量传给子组件，子组件内部再 filter。是否引入 React Context 层（Identity/Permission/Theme context）
- **虚拟化**: PortraitBar、素材库列表无 virtualization，50+ 实体时 DOM 节点激增。是否引入 `react-window` 或类似方案
- **`useMemo`/`useCallback` 审计**: 派生数据（sceneEntityIds、seatProperties 等）是否正确 memo 化
- **Yjs observer 节流**: 高频操作（Token 拖拽、属性连续修改）的 observer 是否需要 debounce

### 议题 14: 角色与 Token 交互

> 角色/Token 的创建、编辑、放置、选中流程是否直观？

**待讨论要点:**

- 实体创建入口的可发现性（当前分散在右键菜单、底部工具栏、角色卡）
- Token 与 Entity 的关联方式与可视化
- 角色卡的信息密度与可读性
- Token 选中状态的视觉反馈（选中环、属性浮层）
- Token 右键菜单（编辑属性、设置条件、删除）
- 批量操作（多选 Token、批量移动）
- Token 状态可视化（光环、条件图标、生命值浮层）

**审查补充要点:**

- 🆕 **拖放反馈**: 从素材库拖放 Token 到地图时无拖影预览（drag ghost）
- 🆕 **Token 尺寸可视化**: 大型生物（2×2、3×3）的 Token 尺寸在地图上如何表现
- 🆕 **Token 层叠**: 多个 Token 重叠时如何选择底层 Token（z-order 管理）
- 🆕 **角色卡快捷操作**: 资源条点击 +/− 时无快捷键支持（如 Shift+Click 加 5）
- 🆕 **实体搜索**: 当实体数量 > 20 时，PortraitBar 和素材库需要搜索/过滤能力

### 议题 20: 首次使用体验 (Onboarding) 🆕

### 议题 22: 视觉风格设计 🆕

> 当前的深色玻璃风视觉方案是早期随机选择的，并未经过系统性设计。在对项目所有 UI 元素有完整认知后（议题 13-15 讨论完毕），需要重新审视整体视觉方向。

**待讨论要点:**

- **整体视觉风格定位**: 当前深色玻璃风是否是最佳选择？是否考虑其他方向（如深色 + 暖色调、赛博朋克、奇幻手绘风、极简暗黑等）
- **配色方案重设计**: 基于议题 11 确定的语义化 Token 结构，重新选择具体色值（当前色值为占位）
- **字体选择**: 当前使用系统 sans-serif，是否引入更有个性的字体组合（标题字体 + 正文字体）
- **面板/卡片视觉语言**: 统一所有面板的圆角、边框、阴影、背景处理
- **图标风格统一**: Lucide 图标的 strokeWidth、大小档位在全局的一致应用
- **品牌感/产品辨识度**: 让 myVTT 在视觉上区别于 Roll20、Foundry、Owlbear Rodeo

**前置条件**: 议题 13-15 讨论完毕，对所有 UI 元素有完整认知后开始。

### 议题 21: 状态持久化与 URL 🆕

> Web Interface Guidelines 明确要求 URL 应反映应用状态，支持 deep linking 和刷新恢复。

**待讨论要点:**

- **URL 状态映射**: 活跃场景 ID、战斗模式开关、打开的面板是否编码到 URL hash/query
- **刷新恢复**: 页面刷新后面板状态全丢失（角色卡关闭、聊天折叠、底部工具栏关闭），哪些状态需要持久化
- **Deep linking**: 是否支持分享"这个场景"的链接（如 `#room=xxx&scene=yyy`）
- **Session 恢复策略**: sessionStorage vs localStorage vs URL 的分工
- **浏览器前进/后退**: 场景切换是否产生浏览器历史记录

### 议题 18: 响应式适配（缩小范围）
