# UI 系统设计探索

> 状态：探索阶段 — 待 POC 验证后方可迁入正式设计文档
>
> 背景：Workflow 基础设施（workflow/ 目录）完成后的下一步，解决插件如何安全地定制 VTT 界面的问题。

---

## 一、问题背景

当前 RulePlugin 的 UI 扩展点（`surfaces`）均为**加法型**（注入新面板、新标签页）或**全量替换型**（`EntityCard`、`teamPanel`）。插件无法对现有 UI 元素进行外科手术式修改，也无法创建完全自定义的界面布局。

FVTT 虽然提供了高度的 UI 定制自由，但代价是：

- 插件依赖内部 DOM 结构和 class name，宿主一改就崩
- monkey patching 导致多插件间严重冲突
- 无隔离机制，插件崩溃可能拖垮整个页面

本文探索一套新的 UI 系统，目标是在**安全性**与**表达自由度**之间找到合适的平衡。

---

## 二、核心设计原则

1. **插件提供组件，用户决定布局** — 插件只声明"我有什么"，不声明"我在哪里"
2. **插件间隔离优先** — 插件不能修改另一个插件的内部实现，崩溃不传播
3. **数据/逻辑/视图三层分离** — 共享数据层和逻辑层，视图层各自独立实现
4. **暴露足够的 SDK，杜绝黑客手段** — 插件作者能做的事都有正式 API
5. **自愿暴露扩展点** — 插件可选择开放内部 Slot，默认封闭

## 三、两个 UI 原语

### 3.1 Panel — 功能面板

有交互内容的 UI 组件。由 GM 在布局配置中决定位置，玩家消费该布局。

```ts
sdk.ui.registerComponent({
  id: 'character-card', // 全局唯一 ID，建议 "<plugin-id>.<name>"
  component: CharacterCard, // React 组件
  defaultSize: { width: 320, height: 480 }, // 单位：像素
  minSize: { width: 200, height: 300 }, // 可选
})
```

**组件 props 契约**：宿主注入，只通过 `IComponentSDK` 传递，不暴露内部 store：

```ts
interface ComponentProps {
  sdk: IComponentSDK
}

interface IComponentSDK {
  data: IDataSDK // 读取游戏数据
  workflow: IWorkflowRunner // 触发工作流（复用现有 WorkflowRunner）
  context: ComponentContext
}

interface ComponentContext {
  instanceProps: Record<string, unknown> // 布局配置中绑定的实例参数（如 entityId）
  role: 'GM' | 'Player'
  layoutMode: 'play' | 'edit'
}
```

**多实例支持**：同一组件可在布局中放置多次，每个实例在布局配置里携带独立的 `instanceProps`（例如两个角色卡分别绑定不同的 `entityId`）：

```ts
// 布局配置示例（由 GM 或预设定义）
{
  "daggerheart.character-card#1": {
    x: 14, y: 14, width: 200, height: 280,
    instanceProps: { entityId: "aria-id" }
  },
  "daggerheart.character-card#2": {
    x: 220, y: 14, width: 200, height: 280,
    instanceProps: { entityId: "kael-id" }
  }
}
```

**隔离机制**：每个 Panel 实例被 ErrorBoundary 包裹。插件崩溃显示占位符，其他面板不受影响。

### 3.2 Layer — 视觉层

全屏渲染的装饰性内容，直接融入世界，不参与布局系统。

Layer 始终是 **DOM/React 渲染层**（不是 Konva 层），背景透明，叠加在场景之上。

```ts
sdk.ui.registerLayer({
  id: 'rain-effect',
  zLayer: 'above-canvas', // 详见层级说明
  component: RainLayer,
  pointerEvents: false, // 默认 false；true 时需在组件内部精确控制命中区域
})
```

**层级顺序（从下到上）**：

```
below-canvas    场景背景替换、世界地图底图
─────────────── Konva Canvas（地图、Token、网格）
above-canvas    粒子效果、Token 光晕、场景边框、薄雾
─────────────── Panels（用户布局的功能面板）
above-ui        全屏宣告横幅、Boss 演出动画
```

**Layer 适用场景**：粒子效果、天气、Token 光晕、场景边框装饰、暗角、战斗宣告横幅、环境动画。

> **注**：Konva 画布内部的绘制（如自定义 Token 渲染、地图特效）不属于 Layer 系统，需通过独立的 Canvas SDK（待后续设计）暴露。

## 四、布局系统

### 4.1 概念模型

布局是一份描述"哪个 Panel 实例在什么位置"的配置文件，与组件注册分离：

```ts
interface LayoutConfig {
  // key 格式：<componentId>#<instance>
  [instanceKey: string]: {
    x: number
    y: number
    width: number
    height: number
    visible: boolean
    instanceProps?: Record<string, unknown> // 传入 sdk.context.instanceProps
  }
}
```

### 4.2 权限模型

- **GM** 定义布局 — 分别配置 GM 视图和玩家视图
- **玩家** 消费布局 — 使用 GM 预设的布局游戏，无法自行调整（当前阶段）
- **布局预设** — 插件可随附默认布局，作为起点

```ts
sdk.ui.registerLayoutPreset({
  name: 'Daggerheart 标准布局',
  layout: defaultLayout, // LayoutConfig
})
```

### 4.3 布局编辑器（待实现）

当前阶段布局通过代码/配置文件定义。未来计划提供可视化编辑器：

- GM 进入编辑模式，所有 Panel 显示虚线边框，可拖拽和缩放
- 底部托盘展示"已注册但未放置"的组件
- 支持网格吸附、对齐辅助线（实现细节待另立设计文档）
- 确认后锁定布局，进入游戏

**此功能刻意推迟**——契约（API）设计完成后即可编程实现布局，可视化编辑器是独立的 UX 工程问题。

## 五、SDK 结构

### 5.1 注册阶段 SDK（`IUIRegistrationSDK`）

插件 `onActivate` 时收到，用于注册 UI 资源。**与现有 `IPluginSDK`（workflow 注册 SDK）是不同的接口**；两者在 VTTPlugin 激活时分别通过不同参数提供，或合并为一个更大的激活 SDK（实现阶段决定）。

```ts
interface IUIRegistrationSDK {
  ui: {
    registerComponent(def: ComponentDef): void
    registerLayer(def: LayerDef): void
    registerLayoutPreset(preset: LayoutPreset): void
    slot(slotId: string, component: React.ComponentType<SlotProps>): void
  }
}
```

### 5.2 组件运行时 SDK（`IComponentSDK`）

Panel 组件渲染时通过 props 收到，用于读取数据和触发逻辑：

```ts
interface IComponentSDK {
  data: IDataSDK // 读取游戏状态
  workflow: IWorkflowRunner // 复用 src/workflow/pluginSDK.ts 中的 WorkflowRunner
  context: ComponentContext
}

interface IDataSDK {
  entity(id: string): Entity | undefined
  entities(): Entity[]
  room(): Room
  scene(): Scene
  subscribe<T>(selector: (state: WorldState) => T, callback: (val: T) => void): () => void
}
```

`IWorkflowRunner` 直接复用现有 `src/workflow/pluginSDK.ts` 中的实现，无需新建。

### 5.3 三层分离的价值

数据层和逻辑层是**共享的稳定契约**，视图层可以并行实现：

```ts
// 官方实现
function OfficialCard({ sdk }: ComponentProps) {
  const entityId = sdk.context.instanceProps.entityId as string
  const entity = sdk.data.entity(entityId)
  return <OfficialUI entity={entity} onRoll={() => sdk.workflow.run(Roll, {})} />
}

// 社区替代实现 — 不替换，直接新建，基于同一 SDK
function CompactCard({ sdk }: ComponentProps) {
  const entityId = sdk.context.instanceProps.entityId as string
  const entity = sdk.data.entity(entityId)
  return <CompactUI entity={entity} onRoll={() => sdk.workflow.run(Roll, {})} />
}
// 用户在布局中选择放哪个，或两个都放
```

## 六、插件间协作：自愿 Slot

插件**默认封闭**，不允许外部修改内部实现。但插件可以主动暴露具名扩展点：

```tsx
// PortraitBar 插件内部，主动暴露 slot
import { Slot } from '@myvtt/sdk'

function PortraitBar({ sdk }: ComponentProps) {
  return (
    <div>
      <Slot name="portrait-bar:before" sdk={sdk} />
      <CharacterList sdk={sdk} />
      <Slot name="portrait-bar:after" sdk={sdk} />
    </div>
  )
}
```

```ts
// 另一个插件注册到该 slot（注册阶段）
sdk.ui.slot('portrait-bar:after', MyExtraButton)
```

- 多个插件注册同一 slot — 按注册顺序依次渲染
- 每个 slot 内的组件被独立 ErrorBoundary 包裹
- slot 的存在与否完全由宿主插件决定，外部无法强制注入

## 七、与现有系统的关系

### 7.1 与 RulePlugin 的共存

现有 `RulePlugin`（`src/rules/types.ts`）短期内保持不变。新 UI 系统以 `VTTPlugin` 的 UI 扩展形式存在，两套系统并行：

- RulePlugin 的 `characterUI.EntityCard` 等继续工作
- 新系统提供更细粒度的 `registerComponent` / `registerLayer`
- 长期目标：RulePlugin 的 UI 相关部分迁移到新系统

**冲突优先级**：若新系统注册的组件与 RulePlugin 的 `hideElements` 声明冲突（例如 VTTPlugin 注册了 `core-ui.portrait-bar`，而 RulePlugin 声明 `hideElements: ['portrait-bar']`），**新系统优先** — `hideElements` 仅作用于旧的宿主硬编码组件，不影响通过 `registerComponent` 注册的组件。

### 7.2 现有 UI 的迁移路径

现有宿主 UI 组件（PortraitBar、ChatPanel 等）可以逐步迁移为"官方插件"：

```ts
// plugins/core-ui/index.ts
export const coreUIPlugin: VTTPlugin = {
  id: 'core-ui',
  onActivate(sdk) {
    sdk.ui.registerComponent({
      id: 'core-ui.portrait-bar',
      component: PortraitBar,
      defaultSize: { width: 800, height: 56 },
    })
    sdk.ui.registerComponent({
      id: 'core-ui.chat-panel',
      component: ChatPanel,
      defaultSize: { width: 320, height: 480 },
    })
  },
}
```

**迁移难度低**：组件本身不需要改动，只是注册方式变化。

## 八、不支持的能力（刻意边界）

| 不支持的能力                      | 原因                                               |
| --------------------------------- | -------------------------------------------------- |
| 插件强制替换另一插件的 Panel 实现 | 三层分离已解决真实需求；替换会引入冲突解决的复杂度 |
| 直接访问 zustand store            | 必须通过 `sdk.data.*`，防止插件依赖内部状态结构    |
| 直接操作 DOM                      | 无法与 React 渲染协作，且破坏隔离                  |
| 创建新的 zLayer 级别              | zLayer 枚举由宿主固定，防止层级混乱                |
| Layer 直接绘制到 Konva 画布       | Konva 集成由独立 Canvas SDK 负责（待后续设计）     |
| 玩家自行调整布局                  | 当前阶段 GM 统一控制，未来可作为选项开放           |

## 九、开放问题（待 POC 后解答）

1. **`sdk.data` 的订阅粒度** — 如何让 Panel 组件高效响应数据变化，避免不必要重渲染？
2. **布局的持久化粒度** — 按房间存储？按场景？按用户角色？
3. **布局坐标系** — 像素坐标在不同屏幕尺寸上如何自适应？是否需要百分比支持？
4. **Panel 的显示条件** — 是否支持 `visibleWhen` 条件（如"仅在战术模式显示"）？
5. **`IUIRegistrationSDK` 与 `IPluginSDK` 的合并方式** — 激活时是否合并为单一 SDK 对象？
