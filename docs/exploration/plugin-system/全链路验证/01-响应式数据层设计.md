# 阶段 1：响应式数据层设计

> **POC 目标**：验证 `IDataSDK` 接口设计是否合理，以及数据变化时 UI 组件能否自动响应。
> **后端改造**（SQLite / REST / Socket.io）不在本阶段范围内，等接口验证完成后再统一规划。

---

## 一、已确认的最终方向

### 1.1 ECS 风格的组件化存储

当前 `Entity.ruleData: unknown` 是一个大 blob，多插件写入时存在竞争条件：

```
Plugin A 读: ruleData = { hp: 20 }
Plugin B 读: ruleData = { hp: 20 }
Plugin A 写: { hp: 18 }          ← 扣血成功
Plugin B 写: { spells: [...] }   ← 覆盖了 A 的写入，hp 回到 20
```

**最终方向**：将 `ruleData` 拆分为按插件命名空间隔离的 `components` map。每个组件对应存储层的独立行，天然避免并发覆盖。

```ts
// Entity 类型（最终）
interface Entity {
  id: string
  name: string
  imageUrl: string
  color: string
  // ...其他核心字段不变

  // ruleData: unknown  ← 废弃
  components: Record<string, unknown>
  // {
  //   'daggerheart-core:health':     { hp: 20, maxHp: 30, armor: 2 },
  //   'daggerheart-core:spellcasting': { slots: [3, 2], knownSpells: [] },
  //   'my-plugin:custom':            { ... },
  // }
}
```

**后端存储（最终，暂不实现）**：

```sql
CREATE TABLE entity_components (
  entity_id    TEXT NOT NULL,
  component_key TEXT NOT NULL,   -- 'pluginId:name'
  data         JSON NOT NULL DEFAULT '{}',
  updated_at   INTEGER,
  PRIMARY KEY (entity_id, component_key),
  FOREIGN KEY (entity_id) REFERENCES entities(id) ON DELETE CASCADE
);
```

每次写入是 `UPSERT` 单行，不同插件的写入操作物理隔离，无竞争。

**兼容 shim（后端迁移时）**：GET 请求返回时，服务端将 `entity_components` 组装为 `components` map，旧代码读 `entity.ruleData` 可通过 shim 转发，上层无感知迁移。

> **结论**：ECS 风格是本系统更合适的模型，但后端迁移（schema 变更、REST 路由、Socket.io 事件）推迟到全链路验证通过后统一规划。

### 1.2 组件注册机制

插件在激活时声明它为 Entity 提供的组件类型：

```ts
// 插件激活时
sdk.entity.registerComponent('daggerheart-core:health', {
  schema: { hp: 'number', maxHp: 'number', armor: 'number' },
  default: { hp: 10, maxHp: 10, armor: 0 },
})
```

注册信息的作用：

- 提供 `default` 值（新实体缺少该组件时的初始状态）
- 未来可用于 schema 校验和 IDE 类型提示
- 宿主可以知道"哪些实体有哪些组件"，支持结构化查询

### 2.2 WorkflowContext 写入扩展

```ts
interface WorkflowContext {
  // 已有（保留，用于修改核心字段）
  updateEntity(entityId: string, patch: Partial<Entity>): void

  // 新增：修改组件数据（函数式更新器，避免读-改-写竞争）

  // 已有（最终将改名，与 globals 对齐）
  patchGlobal(key: string, patch: { current?: number; [k: string]: unknown }): void
```

### 2.3 响应式机制

`IDataSDK` 的实现由宿主通过 zustand store 注入。组件调用 `sdk.data.entity(id)` 或 `sdk.data.component(id, key)` 时，实际上在读取 zustand store 的数据；store 变化时，React 订阅机制触发组件重新渲染。

这一细节对插件开发者完全透明——插件代码不直接接触 zustand，只通过 `sdk.data` 访问。

```
zustand store (mock)
      ↓  store 变化
IDataSDK 实现层（订阅 store）
      ↓  re-render
Plugin Component
```

## 三、POC 验收标准

1. **响应式**：调用 `updateComponent(entityId, 'core:health', () => ({ hp: 15 }))`，读取该组件的面板立即显示新值，无需手动刷新。

2. **多面板同步**：两个面板绑定同一个 entityId，其中一个触发 `updateComponent`，两个面板同时更新。

3. **Globals**：调用 `patchGlobal('Fear', { current: 3 })`，显示该 global 的组件立即反映变化。

4. **查询**：`sdk.data.query({ has: ['core:health'] })` 返回拥有该组件的实体子集。（`where` 条件 POC 暂不实现，标记为 future）

## 四、已确认方向但暂不实现的内容

### 4.1 Item 数据类型

**结论**：Item（物品/法术/能力）应作为独立数据类型，与 Entity 并列。

**原因**：

- Item 需要独立于角色存在（世界级模板、掉落物）
- Item 可在角色之间转移——若 Item 仅是 `entity.components['inventory']` 里的数组元素，转移操作需要读-删-写两个 entity，是并发覆盖的高风险场景
- Foundry VTT 等主流系统均将 Item 作为一等数据类型

**最终接口方向**（暂不实现）：

```ts
// IDataSDK 未来补充
items(entityId: string): Item[]           // 某角色持有的物品
item(itemId: string): Item | undefined    // 按 id 读取（世界级物品）

// WorkflowContext 未来补充
transferItem(itemId: string, fromEntityId: string, toEntityId: string): void
createItem(template: Partial<Item>): Item
deleteItem(itemId: string): void
```

**存储方向**（暂不实现）：Item 与 Entity 类似，也采用 ECS 组件模式存储，并有独立的 `items` 表和 `item_components` 表。

### 4.2 后端改造

| 内容                                       | 原因                                       |
| ------------------------------------------ | ------------------------------------------ |
| SQLite `entity_components` 表              | 后端改造，与接口验证无关                   |

> 以下内容来自全链路验证二审讨论，对本阶段设计的修订和补充。

### 5.1 数据读取 API 按语义分为两层

§2.1 的 `IDataSDK` 将"订阅变化"和"一次性读取"混在同一接口中，但两者语义完全不同：

- **订阅（hook）**：在组件 render 函数体中调用，数据变化时自动触发 re-render
- **一次性读取（命令式）**：在事件回调、`canDrop`、workflow 步骤中调用，只读取当前快照

同一个函数不能同时是 hook 和非 hook（React 规则：hooks 只能在组件函数顶层调用）。利用 React `use` 前缀约定，将两种语义显式分开：

```ts
// 一次性读取接口（命令式，任何地方可用）
interface IDataReader {
  entity(id: string): Entity | undefined
  component<T>(entityId: string, key: string): T | undefined
  global(key: string): Global | undefined
  query(spec: { has?: string[] }): Entity[]
}

// 组件 SDK
interface IComponentSDK {
  // 订阅版（hooks，只能在 render 顶层调用）——数据变化自动触发 re-render
  useEntity(id: string): Entity | undefined
  useComponent<T>(entityId: string, key: string): T | undefined
  useGlobal(key: string): Global | undefined

  // 一次性读取版（事件回调、canDrop 等非 render 上下文中使用）——只读快照，不订阅
  data: IDataReader

  workflow: IWorkflowRunner
  context: ComponentContext
  interaction?: IInteractionSDK
}

// Workflow context — 只需一次性读取
interface WorkflowContext<TState> {
  state: TState
  read: IDataReader // 与 sdk.data 共享同一接口（语义一致，都是一次性读取）
}
```

**组件使用示例**：

```tsx
function EntityCard({ sdk }: ComponentProps) {
  // render 顶层：hook 版，订阅变化
  const health = sdk.useComponent<Health>(entityId, 'core:health')

  const dropZoneProps = sdk.interaction.dnd.makeDropZone({
    canDrop: (payload) => {
      // 事件回调：命令式版，不触发 hook 规则
      const tags = sdk.data.component<Tags>(entityId, 'core:status-tags')
      return !tags?.tags.includes(payload.data.tagId)
    },
    onDrop: (payload) => { ... },
  })

  return <div {...dropZoneProps}>HP: {health?.hp}</div>
}
```

§2.3 中"复用 IDataSDK 同一接口"的表述修正为：`ctx.read` 和 `sdk.data` 共享 `IDataReader` 接口（合理共享，因为语义一致——都是一次性读取）。

### 5.2 Entity 类型改造提醒

现有 `Entity` 类型（`shared/entityTypes`）仍是 `ruleData: unknown` 模型。POC 需同时改造为 `components: Record<string, unknown>`，工作量可能比文档暗示的更大，需要在 POC 排期中考虑。
