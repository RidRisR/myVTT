## 关键决策摘要

1. **极小核心** — 只有场景渲染 + 战术画布是核心，其余全部插件注册
2. **方案选择** — 统一注册 + 布局引擎（方案 A），基于现有 ui-system 扩展
3. **隔离方案** — `@scope` + `contain: layout paint` + SDK 边界（排除 Web Components 和 iframe）
4. **V1 注册原语** — Panel、Layer、碎片型贡献（Slot），Canvas 级效果推后续
5. **日志渲染** — 广播 + 隐式监听，通过 Dispatcher 订阅，不走 Slot
6. **编辑/运行双模式** — GM 控制布局，布局按模式（叙事/战术）绑定
7. **增量迁移** — 新旧共存，逐组件迁移，任何阶段可停

## 排除的方案

- **Web Components / Shadow DOM** — React 事件/Context/Tailwind 兼容性代价过大
- **iframe 隔离** — 不必要，当前为第一方 + 未来可信社区插件
- **事件总线驱动布局（方案 B）** — 布局需要集中编辑，隔离壳需要统一注入
- **V1 的 sdk.events** — YAGNI，非日志事件无明确使用场景

## 业界对标

- WoW Addon 系统：事件广播 + Frame 独立渲染，最接近的灵感来源
- Foundry VTT：反面教材，`return false` 阻断是最大痛点
- Grafana：验证了 data/events 分离的正确性
- CloudEvents：`type` + `source` 协议标准可借鉴
- `@scope`：2026-01 达到 Baseline Newly Available（Firefox 146 支持）
