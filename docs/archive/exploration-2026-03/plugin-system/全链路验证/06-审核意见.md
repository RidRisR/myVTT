## 一、架构设计层面的潜在问题

### 1.1 Workflow 挂起期间的并发安全 ✅ 已解决

**涉及**：阶段 2、5

阶段 5 引入 `ctx.requestInput()` 后 workflow 会长时间挂起，存在四类并发/生命周期问题。经讨论均已找到解决方案：

#### 问题 A：两个 workflow 并发写同一 entity

**结论**：单客户端无问题（JS 单线程，同步块内不会交错）。`ctx.read` 始终返回最新 store 值，只要 read 和 write 在同一个同步块内即可。

#### 问题 B：read-modify-write 竞态（数组操作）

**结论**：废弃 `patchComponent`，统一使用 `updateComponent` 原子更新：

```ts
// updateComponent：read 和 write 在 zustand set() 回调内原子完成
ctx.updateComponent(entityId, 'core:status-tags', (current) => ({
  tags: [...(current?.tags ?? []), input.tagId],
}))
```

不再提供 `patchComponent`。`updateComponent` 是严格超集，简单赋值写 `() => ({ hp: 15 })` 即可，统一心智模型。

#### 问题 C：用户取消操作（Escape）

**结论**：`ISessionSDK` 新增 `cancelInput()` 方法：

```ts
interface ISessionSDK {
  resolveInput(value: unknown): void
  cancelInput(): void // ← 新增
}
```

`cancelInput()` 调用时 reject 存好的 Promise → `WorkflowRunner` 捕获，识别为 cancellation → workflow 状态变为 `'cancelled'`，不视为错误。Workflow 作者不需要写 try/catch，runner 统一处理。

#### 问题 D：组件 unmount 导致 Promise 永远不 resolve

**结论**：`pendingInteraction` 在 session store 里，不受组件生命周期影响。负责 resolve 的组件（地图层）在 unmount 时通过 `useEffect` cleanup 调用 `cancelInput()`：

```tsx
useEffect(() => {
  if (!pending) return
  return () => sdk.session.cancelInput()
}, [pending])
```

标准 React cleanup 模式，不需要 SDK 层特殊机制。

#### 多客户端并发（生产阶段）

**结论**：Entity 级乐观锁 + owner 天然隔离。

- 大多数 entity 有 owner 属性，天然隔离不同客户端的写入
- 极少数并发场景：客户端发起写入时广播锁（不等回复），其他客户端 UI 置灰
- 服务端串行处理同一 entity 的 REST 写入，先到先赢，后到返回冲突
- 锁的作用是 UI 层提前告知，减少冲突概率，不是互斥保证（服务端已保证）

### 1.3 `ctx.data` 命名歧义与 input/state 分离 ✅ 已解决

**涉及**：阶段 2

**原始问题**：文档提议将 `ctx.data` 分离为 `ctx.input`（只读）+ `ctx.data`（步骤间状态），且 `ctx.data` 与 `ctx.read` 命名混淆。

**讨论结论**：

1. **不做强制分离**。插件步骤需要修改原始意图的场景是合理的（如反射护盾重定向目标），强制冻结 input 反而限制了插件能力。

2. **`ctx.data` 改名为 `ctx.state`**（对标 Koa `ctx.state`），与 `ctx.read`（store 数据）明确区分。保持现有行为——调用方传入的参数和步骤间中间数据共享同一个可读写对象。

3. **新增可选的 `ctx.const`**（`Readonly`，`Object.freeze`）。workflow 定义者可以选择把某些参数放进 `const`，任何步骤都无法修改。大部分 workflow 不需要用，只有确实需要保护原始数据的场景才传。

**最终 `WorkflowContext` 接口**：

```ts
interface WorkflowContext<TState, TConst = undefined> {
  state: TState                     // 可读写，步骤间传值（原 ctx.data）
  const?: Readonly<TConst>          // 可选，冻结，不可修改的原始参数
  read: { ... }                     // 只读，查 store
  updateComponent(...): void        // 写 store（原子更新）
  // ...其他方法
}
```

## 二、验证覆盖度不足

### 2.1 缺少错误路径验证 ✅ 已分类处理

五个阶段全部只验证 happy path。经讨论，各失败场景分类如下：

| 场景                                  | 处理方式                   | 原因                                                                                 |
| ------------------------------------- | -------------------------- | ------------------------------------------------------------------------------------ |
| Workflow 步骤异常 → store 半写入      | 已知行为，不进 POC         | store 写入无事务语义是设计选择；VTT 场景下 GM 可手动修正，与 FVTT 自动化插件行为一致 |
| `updateComponent` 写不存在的 entityId | 单测覆盖，不进 POC         | 属于 API 边界校验，用自动化测试保障即可                                              |
| `canDrop` 返回 false 后视觉恢复       | 已有测试覆盖               | DnD 模块的现有测试已覆盖此场景                                                       |
| **EventBus handler 抛异常**           | **进 POC 阶段 4 验收标准** | 需验证一个插件的 handler 崩溃不影响其他 subscriber                                   |
| `requestInput` 取消 / 组件 unmount    | §1.1 已解决                | `cancelInput()` + `useEffect` cleanup                                                |

---

### 2.2 缺少跨插件交互验证（最重要的缺失）→ 新增贯穿场景

全部五个阶段都是单插件场景。如果 POC 不验证跨插件协作，验证的只是「一个应用的模块化」，不是「插件系统」。

**解决方案**：用一个**贯穿所有阶段的完整场景**替代分散的单插件验证。

#### 场景：伤害计算 + 状态效果（两个插件）

**插件定义**：

- `core`：定义 `dealDamage` workflow，注册 `core:health` component
- `status-fx`：拦截伤害计算，注册 `status-fx:resistances` component，播放受击动画

**完整链路**：

```
阶段 5: 点击地图上的 goblin
  → session.selection 更新 → 详情面板显示 goblin 信息

阶段 3: 从法术列表拖"火焰箭"到 goblin 面板
  → onDrop 触发

阶段 2: workflow dealDamage 开始执行
  ↓
  status-fx 拦截步骤（addStep before core）：
    读 status-fx:resistances（跨插件 component 读取）
    发现火焰抗性 → 修改 ctx.state.damage（跨插件意图修改）
  ↓
  core 默认步骤：
    ctx.read 读 core:health
    ctx.updateComponent 写入最终 hp（扣除减免后的伤害）

阶段 1: store 更新 → 所有绑定该 entity 的面板自动 re-render

阶段 4: workflow 同时 emit damageDealtEvent
  → status-fx 的 handler 播放受击动画
  → core 的 handler 显示伤害数字 toast
```

**覆盖的验证点**：

| 验证点                                            | 覆盖阶段 |
| ------------------------------------------------- | -------- |
| 跨插件 component 读取（status-fx 读 core:health） | 1        |
| `addStep` 拦截另一个插件的 workflow               | 2        |
| 拦截步骤修改 `ctx.state`（伤害减免）              | 2        |
| DnD → workflow 触发                               | 3        |
| 两个插件各自订阅同一 event                        | 4        |
| selection 驱动详情面板                            | 5        |
| 双面板同步（两个面板绑定同一 entity）             | 1, 2     |

**降级验证**：

- 禁用 `status-fx` → `dealDamage` workflow 正常执行，无抗性计算，无受击动画
- `status-fx` 的 EventBus handler 抛异常 → core 的 toast handler 仍正常执行（§2.1 阶段 4 验收项）

### 2.3 阶段 5 动态绑定 `$session.selection[0]` 缺少实现定义 ✅ 已解决

**原始问题**：文档使用 `$session.selection[0]` 字符串语法，但未定义解析方式、边界情况、扩展范围。

**结论**：POC 中不做字符串解析，改用 **props 工厂函数**：

```ts
// 静态
'entity-card#goblin': {
  instanceProps: { entityId: 'goblin-01' }
}

// 动态：函数接收 session 状态，返回 props
'entity-card#detail': {
  instanceProps: (session) => ({ entityId: session.selection[0] ?? null })
}
```

布局引擎内部判断 `instanceProps` 是对象还是函数，函数则调用后取返回值：

```ts
const props =
  typeof entry.instanceProps === 'function' ? entry.instanceProps(session) : entry.instanceProps
```

**优势**：类型安全、边界情况由开发者自行处理、不需要发明 DSL。将来如果需要给非开发者用的 GUI 编辑器，可以在 GUI 层生成这个函数，不影响底层 API。

## 三、POC → 生产的代表性风险

### 3.1 ~~单客户端 vs 多客户端~~ ✅ 已解决

**原始问题**：文档声称「生产阶段只需将写入回调从 `store.patch()` 改为 `fetch()`，组件代码和 workflow 代码零改动」，质疑此断言过于乐观。

**讨论结论**：

当前项目主流模式是 API-first + Socket 驱动更新（action 调 REST → 服务端广播 Socket 事件 → 所有客户端 store 更新）。分析 `updateComponent` 在此模式下的行为：

1. **`updateComponent` 内部无论同步写 store 还是 await API，对 workflow 步骤行为一致**——写完即可读到最新值。API-first 模式下 `await` 等服务端确认后再继续，不影响后续步骤的数据一致性。
2. **多客户端并发**已在 §1.1 解决（entity 级乐观锁 + 服务端顺序化）。
3. **`updateComponent` 签名不需要改**——当前 `void` 签名在生产中可改为 `Promise<void>`，workflow 步骤的 `run` 签名本就支持 `Promise<void> | void`。

**结论**：POC 中同步写 store 足以验证插件 API 设计。生产切换写入策略是基础设施层的实现细节，不影响 workflow/plugin 代码。

---

### 3.2 性能代表性不足 → 已确认，POC 中补充

POC 中只有 2–3 个 entity、2 个面板，不能代表：

- 50+ entity 时 `query({ has: [...] })` 的客户端过滤性能
- 10+ 面板同时订阅同一 entity 时的 re-render cascade
- 高频操作（拖拽 token 实时更新位置）的 workflow overhead

**行动项**：sandbox 中至少放 20 个 entity + 4–5 个面板，跑一个批量操作看 re-render 性能。

---

### 3.3 EventBus 模块单例的测试隔离问题 → 已确认，实现时处理

阶段 4 选择 `export const eventBus = new EventBus()` 模块单例。

- 单元测试间共享同一实例，listener 泄漏导致测试干扰
- SSR / 多 room 场景下语义不正确

**行动项**：保持 runtime 单例，但提供 `createEventBus()` 工厂用于测试。

### POC 成功能代表什么

- SDK 接口设计的开发者体验
- 单客户端内的响应式数据流通路
- Workflow 步骤编排和**跨插件**拦截机制（§2.2 贯穿场景）
- DnD → Workflow → Data → re-render 闭环
- EventBus 瞬时信号模型
- 插件降级（禁用 status-fx 后 core 独立运行）

### POC 成功不能代表什么

- ~~多客户端并发写入和同步~~ → 已有生产方案（§1.1、§3.1）
- ~~Workflow 挂起期间的并发安全~~ → 已解决（§1.1）
- 异步写入的错误处理和乐观更新（基础设施层问题，不影响插件 API）
- 大规模数据下的性能（§3.2 行动项部分缓解）

## 六、二审结论

> 二审在各阶段文档中分别追加了「二审补充」章节。以下为汇总。

### 接口设计修订

| 修订项                     | 涉及阶段 | 说明                                                                                                                                         |
| -------------------------- | -------- | -------------------------------------------------------------------------------------------------------------------------------------------- |
| 数据读取 API 按语义分层    | 1        | `sdk.useEntity()` / `sdk.useComponent()`（hook 订阅） + `sdk.data`（`IDataReader` 命令式）。`ctx.read` 和 `sdk.data` 共享 `IDataReader` 接口 |

### 文档修正

| 修正项                   | 涉及阶段 | 说明                                                     |
| ------------------------ | -------- | -------------------------------------------------------- |
| EventBus `emit` 异常隔离 | 4        | 代码示例加 try-catch                                     |

### 生产策略确认

| 项                         | 结论                                                                                                  |
| -------------------------- | ----------------------------------------------------------------------------------------------------- |
| `updateComponent` 生产模式 | 客户端执行 updater 得到结果值 → 发送到服务端 → 服务端广播 Socket.io。与现有 REST + Socket.io 架构一致 |
| 并发策略                   | last-write-wins + owner 隔离 + 软锁，VTT 场景足够                                                     |

### POC 中关注但非阻塞

| 项                       | 说明                                                                        |
| ------------------------ | --------------------------------------------------------------------------- |
| 跨插件 schema 安全       | `component<T>()` 泛型由调用方断言，无运行时保障。手动测试 schema 不匹配场景 |
| `registerComponent` 冲突 | 两个插件注册同一 key 的边界行为，至少验证冲突场景                           |
| 高频操作快速路径         | Token 拖拽位置更新走 workflow 的性能开销                                    |
| 性能代表性               | sandbox 中至少放 20 个 entity + 4-5 个面板                                  |
