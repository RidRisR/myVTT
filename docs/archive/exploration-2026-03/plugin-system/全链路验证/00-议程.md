## 阶段 1：响应式数据层

**核心问题**：`sdk.data` 目前背后是静态 mock，数据变化时组件不会重新渲染。整个系统的响应式基础尚未建立。

**要讨论的事项**：

- 用什么方式实现响应式（zustand store / React context / 其他）
- `IDataSDK` 的接口是否需要调整（目前是同步函数调用，是否足够）
- sandbox 里如何注入真实 store 而不影响测试

**验收标准**：直接修改 store 中的数据，对应的面板组件自动更新显示。

---

## 阶段 2：Workflow 真正写入数据

**核心问题**：目前 `WorkflowRunner` 执行 workflow 后没有真正写入任何数据。通路 2（UI → Workflow → Data → re-render）的完整闭环尚未跑通。

**要讨论的事项**：

- Workflow 如何拿到 store 的写入能力（注入 / 导入单例 / 其他）
- Workflow 执行结果与 store 更新之间的事务边界（失败时如何回滚或忽略）
- 现有 WorkflowContext 接口是否需要增加写入能力

**验收标准**：点击一个按钮 → workflow 执行 → Entity 字段变化 → 面板立即显示新值。

---

## 阶段 3：DnD → Workflow → 双面板联动

**核心问题**：DnD 的 `onDrop` 目前不触发任何真实的数据变更。需要验证"面板不直接通信、通过数据层间接联动"这一核心设计在 DnD 场景下是否成立。

**要讨论的事项**：

- 用什么业务场景验证（物品转移？施法？给 Entity 打标记？）
- 如何在 sandbox 里布置两个面板绑定同一个 Entity
- `onDrop` 回调的签名是否满足调用 workflow 的需要

**验收标准**：面板 A 拖拽落下 → 面板 B 的数据自动变化，无需任何直接通信。

---

## 阶段 4：Event Bus — Workflow 副作用到 UI

**核心问题**：Workflow 目前通过硬编码回调（`showToast` 等）处理 UI 副作用，插件无法扩展。Event Bus 方案尚未实现，也未在真实场景中验证其是否足够。

**要讨论的事项**：

- `IEventBusSDK` 的最小接口（参见 `ui/04-IEventBusSDK设计（概念草稿）.md`）
- 事件总线的实例化方式（模块单例 vs. 注入）
- Workflow 如何持有 emit 能力（`ctx.events` vs. 全局 import）
- 现有 `showToast` 等回调是立即废弃还是保留过渡期

**验收标准**：拖拽落下 → workflow 数据写入（阶段 3）+ 组件视觉反馈（动画/提示）同时发生，两条通路互不干扰。

---

## 阶段 5：Session State — 跨组件共享选中状态

**核心问题**：当前没有机制让多个组件共享"当前选中了哪个 Token"等 UI 交互状态。Event Bus 不适合（迟到者问题）。需要一个本地 UI 状态中心。

**要讨论的事项**：

- `ISessionSDK` 的接口设计（读/写分离？dispatch + selector 模式？）
- session store 与 data store 的边界（什么该在 session，什么该在 data）
- 是否支持动态上下文绑定（`instanceProps: { entityId: "$session.selection.primary" }`）
- 与布局引擎的集成时机

**验收标准**：点击面板 A 的某个元素 → 面板 B 自动切换显示对应内容，无任何直接通信。
