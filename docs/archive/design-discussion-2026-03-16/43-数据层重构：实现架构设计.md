## 一、存储拓扑

```
data/
├── global.db                          ← 全局数据库（用户账号、房间列表）
├── rooms/
│   ├── {roomId}/
│   │   ├── room.db                    ← 房间数据库（SQLite）
│   │   └── uploads/                   ← 素材文件（图片/视频/音频）
│   ├── {roomId2}/
│   │   ├── room.db
│   │   └── uploads/
```

**每个房间一个 SQLite 文件**（`room.db`），包含该房间的所有结构化数据。

**全局数据库**（`global.db`）存储跨房间的数据：用户账号、房间列表。

**房间操作**：

- 删除房间 = `rm -rf data/rooms/{roomId}/`
- 备份房间 = 拷贝 `{roomId}/` 目录
- 迁移房间 = 拷贝目录到另一台服务器 + 在目标服务器 `global.db` 中注册

### 2.3 设计说明

**单行表模式**：`room_state` 和 `combat_state` 始终只有一行（`id = 1`），用 `CHECK` 约束强制。读取是 `SELECT * FROM room_state WHERE id = 1`，更新是 `UPDATE ... WHERE id = 1`。这比用 KV 表更直观，也有 schema 约束。

**JSON 列的使用原则**：

| 用 JSON 列                                     | 用普通列                              |
| ---------------------------------------------- | ------------------------------------- |
| 自由扩展的数据（ruleData、atmosphere）         | 需要索引/查询的字段（name、scene_id） |
| 嵌套结构（grid config、permissions）           | 简单标量（color、size、sort_order）   |
| 不同记录结构不同（blueprint vs handout extra） | 所有记录结构一致的字段                |

**素材 `extra` 列的类型映射**：

| `type`      | `extra` JSON 内容                                 | 对应旧类型                                                           |
| ----------- | ------------------------------------------------- | -------------------------------------------------------------------- |
| `image`     | `{}`                                              | 无（纯图片素材）                                                     |
| `blueprint` | `{ defaultSize, defaultColor, defaultRuleData? }` | 旧 `Blueprint` 接口的 `defaultSize`/`defaultColor`/`defaultRuleData` |
| `handout`   | `{ title, description }`                          | 旧 `HandoutAsset` 接口的 `title`/`description`                       |

旧 `Blueprint.name` → `assets.name`，`Blueprint.imageUrl` → `assets.url`。Blueprint 和 Handout 都通过 `assets` API 管理（`GET /assets?type=blueprint`、`GET /assets?type=handout`），不需要独立端点。

**combat_state.tokens 为什么是 JSON 而不是独立表**：战斗 token 是临时数据（战斗结束即清空），生命周期与 combat_state 绑定。用独立表会增加 JOIN 复杂度和 GC 逻辑，JSON 列更简单（战斗结束时 `UPDATE combat_state SET tokens = '{}'`）。

**TokenData 类型定义**（`combat_state.tokens` 和 `encounters.tokens` JSON 中的值）：

```typescript
interface TokenData {
  id: string
  entityId?: string // 关联的 entity（可选，无则为匿名 token）
  x: number
  y: number
  size: number
  label?: string // 匿名 token 的标签
  imageUrl?: string // 匿名 token 的图片（有 entityId 时从 entity 读取）
  color?: string // 匿名 token 的颜色
  permissions: EntityPermissions
}
```

**ChatMessage 扩展类型**（新增 `retract` 和 `system` 类型）：

```typescript
interface ChatRetractMessage {
  type: 'retract'
  id: string
  senderId: string // 执行撤回的人
  targetId: string // 被撤回的消息 id
  timestamp: number
}

interface ChatSystemMessage {
  type: 'system'
  id: string
  subtype: string // 'tamper_alert' | 'combat_start' | 'combat_end' | ...
  content: string
  timestamp: number
}

type ChatMessage = ChatTextMessage | ChatRollMessage | ChatRetractMessage | ChatSystemMessage
```

**room_state.active_encounter_id 是战斗状态的唯一权威来源**。`combat_state` 表不存储 `encounter_id`——通过 `room_state.active_encounter_id` 即可确定当前战斗来自哪个预设。删除场景时，`active_encounter_id` 的 FK 约束 `ON DELETE SET NULL` 会自动清除激活标记；服务端在场景删除处理中应同时清空 `combat_state`。

### 3.2 Deep Merge PATCH 语义

`PATCH` 端点使用 deep merge 语义处理 JSON 字段：

```
现有数据：{ ruleData: { hp: { current: 20, max: 20 }, str: 14 } }
PATCH body：{ ruleData: { hp: { current: 15 } } }
结果：    { ruleData: { hp: { current: 15, max: 20 }, str: 14 } }
```

服务端实现：

```javascript
function deepMerge(target, patch) {
  for (const [key, value] of Object.entries(patch)) {
    if (
      value &&
      typeof value === 'object' &&
      !Array.isArray(value) &&
      target[key] &&
      typeof target[key] === 'object' &&
      !Array.isArray(target[key])
    ) {
      deepMerge(target[key], value)
    } else {
      target[key] = value
    }
  }
  return target
}
```

**数组和 null 值**不做深度合并——直接覆盖。注意这不是标准的 JSON Merge Patch (RFC 7396)——RFC 7396 用 `null` 表示删除 key，而我们的实现将 `null` 作为普通值写入。语义更接近 lodash `_.merge` 的递归合并。如需删除某个字段，客户端应显式传递不含该字段的完整父对象来覆盖。

### 3.3 角色过滤

所有返回数据的 GET 端点根据请求者角色过滤：

```javascript
// 中间件：从 JWT 获取角色
function withRole(req, res, next) {
  const { userId } = verifyJWT(req.cookies['myvtt-auth'])
  const seat = db.prepare('SELECT role FROM seats WHERE user_id = ?').get(userId)
  req.role = seat?.role || 'PL'
  next()
}

// 场景列表：玩家看不到 gm_only 场景
app.get('/api/rooms/:roomId/scenes', withRole, (req, res) => {
  const where = req.role === 'GM' ? '' : 'WHERE gm_only = 0'
  const scenes = db.prepare(`SELECT * FROM scenes ${where} ORDER BY sort_order`).all()
  res.json(scenes)
})

// 实体：玩家看不到 GM-only 字段
app.get('/api/rooms/:roomId/entities/:id', withRole, (req, res) => {
  const entity = db.prepare('SELECT * FROM entities WHERE id = ?').get(req.params.id)
  if (req.role !== 'GM') {
    // 剥离 GM-only 数据
    const ruleData = JSON.parse(entity.rule_data)
    delete ruleData.secretNotes
    delete ruleData.hiddenStats
    entity.rule_data = JSON.stringify(ruleData)
  }
  res.json(entity)
})
```

## 四、Socket.io 协议

### 4.1 连接与鉴权

```javascript
// 客户端
const socket = io('http://localhost:4444', {
  auth: { token: getCookie('myvtt-auth') },
  query: { roomId },
})

// 服务端
io.use((socket, next) => {
  const { token } = socket.handshake.auth
  const payload = verifyJWT(token)
  if (!payload) return next(new Error('Unauthorized'))

  const { roomId } = socket.handshake.query
  const seat = lookupSeat(roomId, payload.userId)

  socket.data = { userId: payload.userId, roomId, seatId: seat?.id, role: seat?.role }
  socket.join(roomId)
  next()
})
```

### 4.3 Awareness 等效实现

替代 Yjs Awareness 协议，用 Socket.io 事件实现相同功能：

```javascript
// 服务端
const clientStates = new Map() // socketId → { roomId, state }

io.on('connection', (socket) => {
  clientStates.set(socket.id, { roomId: socket.data.roomId, state: {} })

  socket.on('awareness:update', ({ field, state }) => {
    const clientState = clientStates.get(socket.id)
    clientState.state[field] = state
    // 广播给同房间其他客户端
    socket.to(socket.data.roomId).emit('awareness:update', {
      clientId: socket.id,
      seatId: socket.data.seatId,
      field,
      state,
    })
  })

  socket.on('disconnect', () => {
    socket.to(socket.data.roomId).emit('awareness:remove', {
      clientId: socket.id,
      seatId: socket.data.seatId,
    })
    clientStates.delete(socket.id)
  })
})
```

客户端 hook（替代 `useAwarenessResource`）：

```typescript
function useAwareness(socket, mySeatId, mySeatColor) {
  const [remoteStates, setRemoteStates] = useState(new Map())

  const broadcast = useCallback(
    (field, state) => {
      socket.emit('awareness:update', {
        field,
        state: { ...state, seatId: mySeatId, color: mySeatColor },
      })
    },
    [socket, mySeatId, mySeatColor],
  )

  const clear = useCallback(
    (field) => {
      socket.emit('awareness:update', { field, state: null })
    },
    [socket],
  )

  useEffect(() => {
    socket.on('awareness:update', ({ clientId, seatId, field, state }) => {
      setRemoteStates((prev) => {
        const next = new Map(prev)
        if (state === null) {
          const existing = next.get(clientId)
          if (existing) {
            delete existing[field]
            if (Object.keys(existing).length === 0) next.delete(clientId)
          }
        } else {
          next.set(clientId, { ...next.get(clientId), [field]: state })
        }
        return next
      })
    })
    socket.on('awareness:remove', ({ clientId }) => {
      setRemoteStates((prev) => {
        const next = new Map(prev)
        next.delete(clientId)
        return next
      })
    })
  }, [socket])

  return { broadcast, clear, remoteStates }
}
```

## 五、客户端架构改造

### 5.1 zustand store 变更

数据源从 Yjs observer 改为 REST API + Socket.io 事件：

```typescript
interface WorldState {
  // 数据（来自 REST API 初始加载 + Socket.io 增量更新）
  scenes: Scene[]
  entities: Entity[]
  assets: AssetMeta[]
  seats: Seat[]
  chatMessages: ChatMessage[]
  combatState: CombatState | null
  roomState: RoomState

  // Actions（发 REST API → 服务端处理 → WS 事件回来更新 store）
  // 场景
  createScene: (data: Partial<Scene>) => Promise<void>
  updateScene: (id: string, changes: Partial<Scene>) => Promise<void>
  deleteScene: (id: string) => Promise<void>
  // 实体
  createEntity: (data: Partial<Entity>) => Promise<void>
  updateEntity: (id: string, changes: Partial<Entity>) => Promise<void>
  deleteEntity: (id: string) => Promise<void>
  // ... 其他 CRUD actions

  // 初始化

```

### 5.2 初始化时序

```
App 启动
  → Socket.io 连接（携带 JWT cookie）
  → 并行请求：
      GET /rooms/:roomId/state
      GET /rooms/:roomId/scenes
      GET /rooms/:roomId/entities（当前场景）
      GET /rooms/:roomId/seats
      GET /rooms/:roomId/chat?limit=100
      GET /rooms/:roomId/combat
      GET /rooms/:roomId/team-trackers
      GET /rooms/:roomId/assets
  → 所有请求完成 → 写入 zustand store → 渲染
  → 注册 Socket.io 事件监听器（增量更新 store）
```

### 5.3 数据流示例：修改 Entity

```
用户在 CharacterCard 中修改 HP:
  → store.updateEntity('e-1', { ruleData: { hp: { current: 15 } } })
  → fetch('PATCH /api/rooms/:roomId/entities/e-1', { body: { rule_data: deepMerge patch } })
  → 服务端：验权 → SQLite deep merge 写入 → io.to(roomId).emit('entity:updated', { id, changes })
  → 本客户端：可选择乐观更新（先改 store，API 成功后确认；失败则回滚）
  → 其他客户端：收到 'entity:updated' → 更新 store → 组件重渲染
```

### 5.4 数据流示例：资源条拖拽

```
拖拽开始（pointerDown）:
  → setLocalDragValue(value)
  → socket.emit('awareness:update', { field: 'resourceDrag', state: { entityId, field, value, seatId, color } })

拖拽中（pointerMove, ~60fps）:
  → setLocalDragValue(newValue)
  → socket.emit('awareness:update', { field: 'resourceDrag', state: { entityId, field, value: newValue, seatId, color } })
  → 其他客户端：收到 awareness → 更新 remoteDragValue → ResourceBar 实时渲染

松手（pointerUp）:

  → socket.emit('awareness:update', { field: 'resourceDrag', state: null })
  → fetch('PATCH /api/rooms/:roomId/entities/:id', { ruleData: { resources: [...] } })
  → 服务端广播 entity:updated → 所有客户端更新最终值
```

与现有 `ResourceBar.tsx` 的接口完全兼容——只需改变回调函数的实现，组件本身不动。

### 5.5 Scene 字段迁移对组件的影响

当前 `Scene` 接口包含 22 个字段，其中 grid/combat 相关字段迁移到 `encounters` 和 `combat_state`：

```
旧 Scene.gridSize/gridSnap/gridVisible/...   → combatState.grid（从 store.combatState 读取）
旧 Scene.tacticalMapImageUrl/battleMapUrl     → combatState.mapUrl
旧 Scene.combatActive                         → store.roomState.activeEncounterId != null
旧 Scene.initiativeOrder/initiativeIndex      → combatState.initiativeOrder/initiativeIndex
旧 Scene.tokens                               → combatState.tokens
```

**组件读取规则**：之前从 `scene` 读 grid/token/initiative 的组件（`KonvaMap`、`KonvaTokenLayer`、`InitiativeTracker` 等），改为从 `store.combatState` 读取。Scene 只保留叙事层数据（name、sortOrder、atmosphere）。

### 5.6 数据流示例：Token 拖拽

```

  → 更新本地 React state（流畅渲染）
  → socket.emit('token:dragging', { tokenId, x, y })

松手（pointerUp）:

  → fetch('PATCH /api/rooms/:roomId/combat/tokens/:id', { x: snappedX, y: snappedY })
  → 服务端写入 SQLite → 广播 combat:token:updated → 所有客户端更新最终位置
```

### 6.2 SQLite 连接管理

```typescript
import Database from 'better-sqlite3'

const globalDb = new Database(path.join(DATA_DIR, 'global.db'))
const roomDbs = new Map<string, Database.Database>()

function getRoomDb(roomId: string): Database.Database {
  if (!roomDbs.has(roomId)) {
    const dbPath = path.join(DATA_DIR, 'rooms', roomId, 'room.db')
    fs.mkdirSync(path.dirname(dbPath), { recursive: true })
    const db = new Database(dbPath)
    db.pragma('journal_mode = WAL') // 读写并发
    db.pragma('foreign_keys = ON') // 外键约束
    initRoomSchema(db) // CREATE TABLE IF NOT EXISTS ...
    roomDbs.set(roomId, db)
  }
  return roomDbs.get(roomId)!
}
```

### 6.3 广播模式

REST API 处理完成后触发 Socket.io 广播：

```typescript
// routes/entities.ts
app.patch('/api/rooms/:roomId/entities/:id', withAuth, withRole, (req, res) => {
  const db = getRoomDb(req.params.roomId)
  const existing = db.prepare('SELECT * FROM entities WHERE id = ?').get(req.params.id)

  // deep merge JSON 字段
  const ruleData = deepMerge(JSON.parse(existing.rule_data), req.body.rule_data || {})
  const permissions = deepMerge(JSON.parse(existing.permissions), req.body.permissions || {})

  db.prepare(
    `UPDATE entities SET
    name = COALESCE(?, name),
    image_url = COALESCE(?, image_url),
    color = COALESCE(?, color),
    rule_data = ?,
    permissions = ?
    WHERE id = ?`,
  ).run(
    req.body.name,
    req.body.image_url,
    req.body.color,
    JSON.stringify(ruleData),
    JSON.stringify(permissions),
    req.params.id,
  )

  // 广播给同房间所有客户端（包括发起者——保持一致性）
  io.to(req.params.roomId).emit('entity:updated', {
    id: req.params.id,
    changes: req.body,
  })

  res.json({ ok: true })
})
```

## 七、迁移策略

### 7.1 迁移原则

- **渐进式迁移**：不一次性推翻，按模块逐步替换
- **兼容期**：迁移期间支持读取旧 Y.Doc 数据，写入新 SQLite
- **数据导出工具**：提供脚本从 Y.Doc 导出所有数据到 SQLite

### 7.2 迁移工具

提供 `scripts/migrate-ydoc-to-sqlite.ts` 脚本：

```
读取 y-leveldb 中的 Y.Doc
  → 遍历 scenes Y.Map → INSERT INTO scenes（atmosphere 字段聚合）
  → 遍历 entities Y.Map → INSERT INTO entities
  → 遍历 chat_log Y.Array → INSERT INTO chat_messages
  → 遍历 seats Y.Map → INSERT INTO seats
  → 遍历 blueprints Y.Map → INSERT INTO assets (type='blueprint', extra = { defaultSize, defaultColor, defaultRuleData })
  → 遍历 handout_assets Y.Map → INSERT INTO assets (type='handout', extra = { title, description })
  → 遍历 showcase_items Y.Map → INSERT INTO showcase_items
  → 遍历 team_metrics Y.Map → INSERT INTO team_trackers
  → 读取 room Y.Map → UPDATE room_state (active_scene_id, active_encounter_id)
  → 复制 uploads/ 文件
```

### 7.3 建议的迁移顺序

1. **基础设施**：安装依赖（socket.io、better-sqlite3），建立 SQLite schema，服务端文件结构
2. **素材管理**：最独立的模块，先迁移素材到 SQLite + REST API
3. **聊天系统**：从 Y.Array 迁移到 SQLite + REST + WS 广播
4. **实体系统**：从 Y.Map 迁移到 SQLite，改造 zustand store
5. **场景系统**：Scene 结构精简 + 迁移到 SQLite
6. **战斗系统**：combat 运行时 + token + encounter 预设
7. **座位/身份**：迁移到 SQLite + REST API
8. **清理**：移除 Yjs 相关代码和依赖

每一步完成后都是一个可运行的状态，不存在"半成品不能用"的阶段。
