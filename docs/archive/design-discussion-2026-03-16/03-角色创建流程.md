# 议题 3: 角色创建流程

> 讨论 PC 和 NPC 的创建入口、entities 的用户定位、linked/unlinked 创建模式。

---

## 已确认决策

### C1: PC 创建入口

Portrait Bar 上的 **+** 按钮 → 直接写入 entities。

- PC 天生跨场景，必须从 entities 创建
- 创建后自动加入所有已有场景的 entityIds
- 后续可迭代更丰富的创建流程（上传头像、选模板等）

### C2: NPC 创建 — 全部进 entities

GM 创建 NPC 时直接写入 entities + 加入当前场景 entityIds。（无 scene.entities）

- 从 blueprint 创建 → 新 entities Entity（unlinked，独立实例）
- 右键背景创建空 NPC → 新 entities Entity + 当前场景 entityIds
- 不需要 promote — 所有 Entity 天生在 entities 里

### C3: Linked vs Unlinked 创建模式

参考 FVTT 的 Actor + Token 模型：

- **Unlinked（默认）**：每次从 blueprint 创建 → 新 Entity → 加入当前场景
  - 5 个哥布林 = 5 个独立 Entity
- **Linked**：选择已有 Entity → 加入当前场景 entityIds（不创建新的）
  - 兽人酋长在 3 个场景 = 同一个 Entity，HP 全局同步

底层存储完全相同（都在 entities），区别只是创建时"新建"还是"复用"。

### C4: Roster 是基础设施，不是用户概念

GM 不需要知道"entities"这个词。Roster 解决的是**数据共享**问题：

- 所有 Entity 的唯一数据源
- PC 的"家"在 entities（不属于任何单一场景）
- GM 的操作语言是"从素材库放一个到场景"或"让已有角色也出现在这个场景"

### C5: 跨场景复用（Linked 放置）的 UI — 延后设计

GM 需要某种方式把已有角色引入当前场景（linked）。具体 UI 形态待定：

- 可能是素材库中的"已有实例"列表
- 可能是角色卡菜单中的"也出现在其他场景"
- 可能是独立的"角色管理"面板

延后到实际需求出现时再决定。当前默认行为全部是 unlinked（新建）。
