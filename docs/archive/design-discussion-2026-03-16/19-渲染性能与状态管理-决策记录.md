## 1. 状态管理 — zustand 作为 Yjs→React 桥接层

**问题**: 当前 22 处手写 observer，每次 Yjs 变化都执行 O(n) 全量重建数组，导致所有订阅组件重渲染。App.tsx 向子组件传递 50+ 个 props（prop drilling）。

**决策**: 引入 zustand 作为 Yjs 和 React 之间的桥接层。

**架构**:

```
Yjs (网络同步层)
  ↓ observer 写入
zustand store (状态管理层)
  ↓ selector 订阅
React 组件 (渲染层)
```

**具体做法**:

- Yjs observer 将数据写入 zustand store（替代原来的 useState + setState）
- 组件通过 zustand selector 订阅需要的数据切片
- selector 返回值没变 → 组件不重渲染

**解决的问题**:

- ✅ **全量重建问题**: Token #5 变化 → 只有订阅了 `state.tokens["5"]` 的组件重渲染
- ✅ **Prop Drilling**: 子组件直接从 store 读数据，不需要 App.tsx 中转 50+ props
- ✅ **Observer 模板**: 可提取通用 `useYjsMap()` Hook，22 处模板代码大幅精简

**zustand 选型理由**:

- GitHub 50k+ stars，React 生态最成熟的状态管理库之一
- 极轻量（~1KB），API 简洁
- 不绑定 Yjs — Yjs 负责网络同步，zustand 负责 React 渲染优化
- selector 机制天然提供细粒度渲染控制

**Yjs 层不变**: observer、nesting rules、transactions 全部保持现有控制力。zustand 只是"读取侧"的优化，不改变 Yjs 的写入逻辑。

## 2. 战术地图渲染 — react-konva

**问题**: 当前战术地图用 DOM 渲染（每个 Token 是 `<div>`），配合 react-zoom-pan-pinch 做缩放。这是快速验证时期的写法，Token 多时性能不理想。

**决策**: 引入 react-konva，战术地图整体迁移到 Canvas 渲染。

**react-konva 优势**:

- Canvas 批量绘制 Token、网格线、测量线，性能远优于 DOM
- Konva 内置 Stage 变换（缩放/平移），可替代 react-zoom-pan-pinch
- 提供事件系统（click、drag），API 和 DOM 事件类似
- 适合图形密集型场景（正是战术地图的特征）

**适用范围**: 仅战术地图。叙事模式 UI（聊天、角色卡、菜单等）继续用 DOM。

**迁移时机**: 在 UI 大重构中一起进行，战术地图组件整体重写。

## 3. 不引入 SyncedStore

**调研结论**: SyncedStore（`@syncedstore/react`）不适合本项目。

**排除理由**:

- **已停更**: 最后一次提交 2023 年 10 月，超过 2 年未更新
- **React 19 不兼容**: 底层依赖 `@reactivedata/react` 只声明 React 18 peer dep
- **永远 0.x**: 4 年从未到 1.0，API 未稳定
- **自动 Proxy 冲突**: 自动将所有嵌套对象包成 Y.Map，无法控制 CRDT 粒度。与我们项目显式管理的 nesting rules（哪些字段用嵌套 Y.Map、哪些用 last-write-wins）冲突
- **已知严重 bug**: Proxy 引用陷阱、深层嵌套栈溢出、数组操作过度重渲染

**其他替代方案同样不理想**:

- immer-yjs: 活跃但无 React 响应式绑定
- zustand-middleware-yjs: 锁定 zustand v3（当前是 v5）

## 4. 列表虚拟化 ⏸

**决策**: 暂不引入，按需再加。

**理由**:

- 实际使用中角色数量很少超过 30 个
- react-konva 接管战术地图后，Token 渲染已不是 DOM 问题
- 蓝图库列表可先用 CSS `content-visibility: auto` 做轻量优化
- 如果未来蓝图 > 100 个出现性能问题，再引入 `@tanstack/react-virtual` 等方案
