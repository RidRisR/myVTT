# 聊天记录完整性方案设计

## 问题

Yjs Y.Array 允许任何客户端删除或修改已有元素。攻击者可以删除对自己不利的骰子记录，或篡改他人的聊天消息。Yjs 协议层面无法拒绝特定 update（二进制 blob 不可分割，部分拒绝会导致 CRDT 状态分歧），因此**预防不可能，只能检测 + 响应**。

## 设计原则

- **Append-only**：chat_log 的所有合法操作都是追加，不存在合法的删除。任何删除 = 非法篡改
- **撤回是追加**：撤回一条消息 = 追加一条 `type: 'retract'` 消息，而不是从 chat_log 中删除原消息
- **草稿与记录分离**：打字预览通过 awareness protocol 广播（临时），只有点击发送后才进入 chat_log（持久）
- **服务端审计一切**：凡是别人能看到的消息，服务端一定已经记录（y-websocket 是中心枢纽，所有 update 经过服务端才到达其他客户端）

## 架构

### 两层分离

```
awareness（临时层）
  正在编辑的内容 → awareness.setLocalState({ typing: '...' })
  → 所有客户端实时看到
  → 可以随时修改/清空/删除
  → 服务端不记录

chat_log Y.Array（记录层，append-only）
  用户点击发送 → append 到 chat_log
  → 服务端 observer 记录到审计日志
  → 从此不可删除
```

### 消息类型

| 类型      | 写入方式         | 说明                                |
| --------- | ---------------- | ----------------------------------- |
| `text`    | 客户端直写 Y.Doc | 即时同步，服务端观察并记录          |
| `roll`    | 服务端写入 Y.Doc | 服务端权威，transaction.origin 审计 |
| `retract` | 客户端直写 Y.Doc | 撤回指令，引用 targetId             |
| `system`  | 服务端写入 Y.Doc | 系统消息（篡改警告等）              |

### 服务端审计流程

服务端通过 `transaction.origin` 追踪每条修改的真实来源（详见 [50-聊天防伪系统.md](50-聊天防伪系统.md)）。

```
服务端启动 → 为每个房间的 chat_log 注册 Y.Array observer

observer(event, transaction):
  origin = transaction.origin
  isServer = (origin === null || origin === 'server')
  realUserId = isServer ? null : origin.userId

  ├─ event.changes.added（新消息）
  │   → 身份审计：检查消息类型与 origin 的合法性
  │     - type:'roll' && !isServer → 伪造骰子 → append 篡改警告
  │     - type:'text' && senderId !== realUserId → 身份冒充 → append 篡改警告
  │     - type:'system' && !isServer → 伪造系统消息 → append 篡改警告
  │   → 记录到审计日志（LevelDB）
  │     key: {roomId}:{messageId}
  │     value: 消息完整内容 + 真实来源 + 状态 + 时间戳
  │
  └─ event.changes.deleted（删除事件）
      → 查找被删除的消息内容
      → 记录删除事件到审计日志（含 realUserId）
      → append 篡改警告到 chat_log：
        { type: 'system', subtype: 'tamper_alert',
          deletedMessageId, perpetratorId: realUserId, timestamp }
      → 所有客户端通过 Yjs 同步看到警告
```

## 撤回机制

### 设计

撤回不是删除，而是追加一条引用原消息的指令：

```
原消息：  { type: 'text', id: 'abc', senderId: 'alice', content: '你好' }
撤回指令：{ type: 'retract', id: 'ret-1', targetId: 'abc', senderId: 'alice', timestamp }
```

### 客户端渲染逻辑

1. 扫描 chat_log，建立 `retractedIds: Set<string>`（所有 retract 消息的 targetId）
2. 渲染消息时，如果 `retractedIds.has(message.id)`，显示为"XXX 撤回了一条消息"
3. 新的 retract 消息到达时，更新 Set，对应消息的 UI 实时变为撤回状态

### 撤回规则

- **文本消息**：发送者本人可以撤回，无时间限制
- **骰子消息**：不可撤回（骰子结果是既定事实）
- **撤回的撤回**：不支持（避免复杂性）
- **GM 权限**：GM 可以撤回任何人的消息（管理需要）

### 服务端验证（后置审计）

撤回消息由客户端直写 Y.Doc（保持即时性），服务端 observer 通过 `transaction.origin` 获取真实写入者身份，事后验证：

```
observer 收到 type: 'retract' 消息
  realUserId = transaction.origin.userId（不可伪造的真实身份）
  → 查找 targetId 对应的原消息
  → 验证 realUserId 是否为原消息发送者（或是否为 GM）
  → 验证 targetId 不是骰子消息
  → 不合法 → append 系统警告
  → 合法 → 记录到审计日志
```

注意：这里用 `transaction.origin.userId`（真实身份）而不是 `message.senderId`（声称身份）来做权限判断。即使攻击者在 retract 消息中伪造 senderId，服务端仍然知道实际操作者是谁。

## 攻击场景分析

| 攻击方式         | 过程                              | 结果                                                   |
| ---------------- | --------------------------------- | ------------------------------------------------------ |
| **直接删除消息** | 操作 Y.Doc 删除 chat_log 元素     | observer 检测到 deletion → append 篡改警告，所有人可见 |
| **伪造撤回**     | append retract 消息撤回别人的消息 | 服务端验证 senderId 不匹配 → append 警告               |
| **撤回骰子**     | append retract 消息引用骰子 id    | 服务端验证目标是骰子消息 → append 警告                 |
| **删除篡改警告** | 删除服务端 append 的 system 消息  | 再次触发 deletion 检测 → 服务端再次 append 警告        |

## 审计日志存储

使用独立的 LevelDB 实例（与 y-leveldb 的 Yjs 持久化分开）。

**Key 设计：** `{roomId}:{messageId}`

**Value 结构：**

```json
{
  "message": {
    /* 消息完整内容 */
  },
  "status": "active | retracted | tampered",
  "recordedAt": 1710000000000,
  "retractedAt": null,
  "tamperedAt": null
}
```

**为什么用独立 LevelDB？** y-leveldb 存储的是 Yjs 的二进制 update，不可读、不可查询。审计日志需要按 messageId 查询、按状态过滤，结构化存储更合适。

## 与骰子防伪的关系

骰子防伪（议题一）和聊天完整性（本方案）共享同一个 transaction.origin 审计机制：

| 保护层                 | 防御目标                           | 机制                             |
| ---------------------- | ---------------------------------- | -------------------------------- |
| transaction.origin     | 骰子伪造 / 身份冒充 / 伪造系统消息 | 服务端 observer 检查 origin 来源 |
| 客户端去重             | 骰子重放                           | 本地 Set 检测重复 rollId         |
| Append-only + 删除检测 | 消息删除                           | 服务端 observer + 篡改警告       |
| 撤回审计               | 伪造撤回                           | 服务端用 origin.userId 验证权限  |

## 不变的部分

- 文本消息仍由客户端直写 Y.Doc（即时同步不受影响）
- 骰子消息仍由服务端写入（权威性不变）
- 旧消息（无 id 字段的历史消息）不纳入审计，正常展示
- 打字预览通过 awareness protocol 实现，与 chat_log 完全分离
