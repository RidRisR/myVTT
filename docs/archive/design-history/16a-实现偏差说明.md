# 16 - 事件日志与骰子系统：实现偏差说明

> 对照设计文档 `16-事件日志与骰子系统架构.md`，记录实现中的偏差及原因。

## 已实现的设计要素

以下核心设计要素已完整实现，与文档一致：

- 统一 `game_log` 表（双 ID 设计：客户端 UUIDv7 `id` + server AUTOINCREMENT `seq`）
- Socket.io 传输层（`log:entry`、`log:roll-request`、`log:history`、`log:new`）
- Side-Effect Registry（`core:tracker-update`、`core:component-update`）
- 同事务快照写回（effect handler 执行后 UPDATE payload）
- Visibility 过滤（`{}`/`{ include }`/`{ exclude }`，逐 socket 广播）
- 执行者路由（server 强制覆盖 `executor` 为 `socket.data.seatId`）
- 级联保护（`triggerable` 标志 + `chainDepth` + `MAX_CHAIN_DEPTH = 10`）
- LogStreamDispatcher（watermark 跳过、triggerable 检查、chainDepth 检查、executor 匹配）
- TriggerRegistry（type + filter 匹配、串行执行）
- PluginSDK `registerTrigger()`
- WorkflowContext 重构（`emitEntry()`、`serverRoll()` → Socket.io）
- `updateComponent()` 和 `updateTeamTracker()` 通过 `emitEntry()` 发送日志条目
- 客户端状态同步（zustand `log:new` handler + snapshot sync for tracker/component）
- Bundle 加载（logEntries + logWatermark）
- Socket.io Connection State Recovery（2 分钟窗口）
- 去重：server 端 `INSERT OR IGNORE` + `info.changes > 0`；client 端 `logEntriesById` 检查
- 删除旧代码路径（`chat_messages`、`game_events` 表、REST chat/events 路由）

## 偏差列表

### 偏差 1：`core:tracker-update` 载荷格式扩展

**文档描述：**

```typescript
// payload: { trackerId: "t1", delta: -5 }
```

**实际实现：** effect handler 支持两种格式：

1. `{ trackerId, delta }` — 按 ID 查找 + 相对增量（文档格式）
2. `{ label, current }` — 按 label 查找 + 绝对值设置（`ctx.updateTeamTracker` 发送的格式）

**原因：** `ctx.updateTeamTracker(label, { current })` 是已有 API，调用方（workflow steps）只知道 tracker 的 label，不知道 ID。且该 API 已标记 `@deprecated`，未来 tracker 系统重新设计时会统一。当前阶段需要同时支持两种格式，保证现有调用链（如 Daggerheart 插件的 Hope/Fear tracker）能正确工作。

---

### 偏差 2：重复条目的 ack 格式

**文档描述：**

> ack 返回 `{ id, seq, status: 'duplicate' }`

**实际实现：** 重复条目 ack 返回完整的 `GameLogEntry`（通过 SELECT 查询已有行），不包含 `status: 'duplicate'` 字段。

**原因：** 简化客户端处理。客户端收到 ack 后无论是新条目还是重复条目，都得到一个完整的 `GameLogEntry` 对象，处理逻辑一致。`status: 'duplicate'` 字段增加了客户端需要处理的分支，但 TTRPG 场景下重复提交极其罕见，区分它没有实际收益。如果未来需要区分，可以加一个可选的 `duplicate?: boolean` 字段。

---

### 偏差 3：Opt-in 原子 Workflow 未实现

**文档描述：**

> 不含 `serverRoll()` 的纯计算 workflow 可以声明 `{ atomic: true }`，workflow 完成后一次性发送所有条目。

**实际实现：** 未实现。所有条目逐条即时发送。

**原因：** 文档自身将此列为 opt-in 优化，不是核心功能。当前 TTRPG 操作频率极低（每回合几条），逐条发送完全满足需求。原子 workflow 引入 per-workflow buffer 管理、abort 时 buffer 清理、serverRoll 运行时禁止等复杂度，收益极小。列入"延迟实现"列表。

---

### 偏差 4：Bundle 端点不做 Visibility 过滤

**文档描述：** 文档未明确说明 bundle 端点的 visibility 处理。

**实际实现：** Bundle REST 端点返回最近 200 条日志，不按 visibility 过滤。

**原因：** Bundle 是 REST 请求（`GET /api/rooms/:roomId/bundle`），没有 Socket.io 连接的 seat/role 上下文。要过滤需要通过 query param 或 cookie 传递 seat 信息，增加认证复杂度。在社会信任模型下，这是可接受的 v1 限制。已在代码中标注 `NOTE`。

---

### 偏差 5：`log:history` 可能返回少于请求数量的条目

**文档描述：** 文档未明确说明分页与 visibility 交互。

**实际实现：** 先查询 N 条，再按 visibility 过滤。如果被过滤掉的条目较多，返回数量会少于请求的 `limit`。

**原因：** 简单实现。正确的做法是循环 fetch 直到凑够目标数量或表耗尽，但引入循环查询复杂度。TTRPG 场景中大多数条目是公开的（`visibility: {}`），实际影响极小。已在代码中标注 `NOTE`。

---

### 偏差 6：`log:entry` 添加了输入验证

**文档描述：**

> 不做写入权限校验（社会信任模型）。

**实际实现：** 对 `submission.id` 和 `submission.type` 做非空字符串验证。

**原因：** 这不是权限校验，是数据完整性校验。`id` 和 `type` 是 `NOT NULL` 列，如果为空会导致 DB 错误或后续 `JSON.parse` 崩溃。这是防御性编程，不违背社会信任模型。

---

### 偏差 7：EventBus 迁移和渲染器驱动效果未实现

**文档描述：**

> 跨客户端效果改为由日志条目渲染器驱动，不再由 workflow step 触发。EventBus 废弃跨客户端效果，保留纯本地 UI 事件。

**实际实现：** EventBus 保持现有行为不变，未做渲染器驱动迁移。

**原因：** 设计文档明确将此列为架构方向而非本期任务。渲染器驱动效果属于 UI 层重构，需要先实现日志条目渲染器（`logEntryRenderers`），这是独立的 UI 任务。本期只实现基础设施层（传输、存储、触发），不涉及 UI 渲染。

---

### 偏差 8：日志条目的 `roll-result` payload 不包含 `total`

**文档描述：**

> payload: `{ dice, formula, resolvedFormula, rollType, actionName, rolls, total }`

**实际实现：** `log:roll-request` handler 的 `payload` 构建中不计算 `total`，只包含 `dice`、`rolls`、`formula`，以及可选的 `resolvedFormula`、`rollType`、`actionName`。

**原因：** 实际代码检查发现这是一个遗漏。`total` 的计算（`rolls.flat().reduce((a, b) => a + b, 0)`）是 trivial 的，但遗漏了。已列入修复项。

**修复：** 见下方修复提交。

---

### 偏差 9：`rowToEntry` 提取为共享模块

**文档描述：** 文档未提及 `rowToEntry` 的实现位置。

**实际实现：** 初始实现中 `logHandler.ts` 和 `bundle.ts` 各有一份 `rowToEntry`。代码审查后提取到 `server/logUtils.ts` 共享。

**原因：** DRY 原则。重复代码是维护风险。

---

### 偏差 10：Transport wiring 未完成（客户端发送链路）

**文档描述：** WorkflowContext 通过 `emitEntry()` 和 `serverRoll()` 发送日志条目到 server。

**实际实现：** `useWorkflowSDK.ts` 的 `buildDeps()` 中 `emitEntry` 是 no-op，`serverRoll` 直接 reject，`getActiveOrigin`/`getSeatId`/`getLogWatermark` 返回空默认值。

**原因：** 本期实现基础设施层（类型定义、Socket.io 传输协议、服务端处理逻辑、客户端状态接收），transport wiring（将 `buildDeps` 连接到实际 Socket.io 连接和 identity store）属于独立的集成层任务。基础设施层的所有组件（logHandler、effectRegistry、visibility、worldStore log:new handler）已通过独立测试验证。

**影响：** workflow 的 `ctx.emitEntry()` 在运行时静默丢弃，`ctx.serverRoll()` 抛异常。客户端→服务器日志发送链路不通。

---

### 偏差 11：LogStreamDispatcher / TriggerRegistry 未在运行时实例化

**文档描述：** LogStreamDispatcher 监听 `log:new` 广播，匹配触发器并调用 WorkflowEngine。

**实际实现：** `LogStreamDispatcher` 和 `TriggerRegistry` 类已实现并通过单元测试，但未在应用代码中实例化。`PluginSDK` 构造时未传入 `TriggerRegistry`，调用 `sdk.registerTrigger()` 会抛 `Error('TriggerRegistry not available')`。

**原因：** 声明式触发链依赖 transport wiring 完成后才能集成。当前无插件注册触发器，因此不会在运行时触发此错误。

**影响：** 声明式触发链（日志条目 → 匹配触发器 → 执行 workflow）在运行时不工作。

---

### 偏差 12：CRUD 操作不传播因果链（parentId / chainDepth）

**文档描述：**

> `parentId` 字段将日志条目链接为因果关系。`chainDepth` 表示因果链深度。

**实际实现：** `updateComponent()` 和 `updateTeamTracker()` 硬编码 `chainDepth: 0`，不设 `parentId`。这些方法产生的日志条目无法通过因果链追溯到触发它们的 workflow。

**原因：** 因果链传播需要 WorkflowContext 维护"当前链上下文"（从 `serverRoll` 返回的条目 id 和 chainDepth 继承），这属于 trigger chain 集成任务的一部分。当前 `updateComponent` 和 `updateTeamTracker` 都设 `triggerable: false`，不正确的 chainDepth 不会导致级联问题。

**影响：** CRUD 条目无法通过 `parentId` 归入因果链分组。UI 的"因果链分组"渲染（设计文档 §UI 渲染模型）将无法将 tracker-update/component-update 与触发它们的 roll-result 关联。需在 trigger chain 集成时修复。

---

## 延迟实现（明确不在本期范围）

以下功能在设计文档中提及，但明确标注为后续任务：

| 功能                                      | 文档章节                | 延迟原因                           |
| ----------------------------------------- | ----------------------- | ---------------------------------- |
| Opt-in 原子 Workflow (`{ atomic: true }`) | 日志提交                | 收益极小，复杂度高                 |
| UI 渲染器（`logEntryRenderers`）          | UI 渲染模型             | 独立 UI 任务                       |
| EventBus 迁移                             | EventBus 与跨客户端效果 | 依赖 UI 渲染器                     |
| 日志清理/归档                             | 开放问题                | 设计文档列为开放问题               |
| 类型注册表                                | 开放问题                | 设计文档列为开放问题               |
| 完整状态重建 fallback（`room:init`）      | 重连与状态恢复          | 已有 bundle 机制，CSR 覆盖短暂断线 |
| `core:retract` 撤回机制                   | 日志不可变性            | 无当前使用场景                     |
| 乐观并发检测（`baseSeq` 检查）            | 并发模型                | 文档明确说"当前不做任何检查"       |

## Assumptions

- 所有偏差均基于"本期只实现基础设施层"的前提，UI 渲染层是独立任务
- `updateTeamTracker` 的双格式支持是临时方案，tracker 系统重设计后会统一
- Bundle 端点的 visibility 问题在社会信任模型下可接受
- 历史分页的 visibility 过滤在 TTRPG 低频操作场景下影响可忽略

## Edge Cases

- 如果两个 tracker 同名（label 相同），label-based 格式只更新第一个匹配行
- Bundle 返回未过滤日志时，客户端 UI 层应自行过滤（目前无 UI 消费此数据）
- `log:entry` 的 id/type 验证拒绝空值，但不验证 UUIDv7 格式（允许任意非空字符串）
