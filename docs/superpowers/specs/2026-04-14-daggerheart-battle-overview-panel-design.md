# Daggerheart 右上战场总览面板设计

> 状态：讨论中，已完成第一轮需求收敛 | 2026-04-14

## 1. 文档目的

本文档整理当前分支围绕 Daggerheart 插件 UI 的已有进展，以及针对“右上常驻面板”目前已经达成的设计结论。

本文档的目标不是给出最终视觉定稿，也不是实现计划，而是把当前讨论中已经明确的信息沉淀下来，避免后续反复回忆上下文。

## 2. 分支背景

当前分支的主线目标，是将 Daggerheart 相关 UI 逐步迁移到插件化的 Region 系统中，并围绕统一掷骰流程、角色资源展示和玩家操作入口建立一套成体系的插件 UI。

从当前分支代码可确认：

- 已实现并注册的 Daggerheart persistent region 有 3 个：
  - `daggerheart-core:fear-panel`：左上 Fear 面板
  - `daggerheart-core:character-card`：左侧角色卡抽屉
  - `daggerheart-core:player-bottom-panel`：底部玩家操作面板
- 当前尚未实现右上 Daggerheart persistent region
- 因此，右上常驻面板是当前 Daggerheart UI 布局中的明确缺口，而不是一个尚未定义的想法

## 3. 现有设计资产

仓库中已经存在可供参考的历史设计和 mockup：

- `docs/design/24-Daggerheart插件UI布局设计.md`
- `nimbalyst-local/mockups/daggerheart-charlist-panel.mockup.html`
- `nimbalyst-local/mockups/daggerheart-vtt-layout.mockup.html`

这些资产说明：

- 右上区域在早期设计中被定位为“在场角色列表”
- 整体布局上，它与左侧角色卡、顶部 Fear、底部玩家面板共同组成 Daggerheart 的主要常驻 UI
- 历史 mockup 已经探索过“角色列表 + hover 详情卡”的方向

但需要注意：

- 历史 mockup 只能作为参考，不等于本轮讨论已经确认的最终 UI
- 本轮讨论目前优先确认的是功能定位、信息范围、交互边界与能力依赖，而不是视觉细节

## 4. 面板职责定位

本轮讨论已经确认，右上面板的定位不是“玩家队伍面板”，而是：

**战场总览面板**

它的职责是提供**在场单位总览**，帮助玩家快速扫视当前局势，而不是承担详细操作或完整角色卡阅读。

这意味着：

- 面板的关注对象不是只有玩家角色
- 玩家单位与敌方单位都应纳入总览
- “我方 / 敌方”是信息分组视角，不是两个完全独立的功能模块

## 5. 交互边界

本轮讨论已经确认，v1 先采用：

**只读面板**

即：

- v1 不承担复杂交互
- v1 不要求支持从该面板直接触发角色操作
- v1 的核心价值是信息总览，而不是快捷操作入口

未来可以再考虑：

- 点击聚焦单位
- 打开详情
- 与战术地图联动
- hover 详情卡

但这些都不属于当前已确认的 v1 范围。

## 6. Tab 结构决策

### 6.1 讨论过程

本轮讨论中曾考虑 3 种方向：

1. 固定语义 tab
2. 全自定义 tab
3. 混合式：固定 tab + 私有关注 tab

最终达成的结论是：

- 长期方向仍然偏向“混合式”
- 但 v1 不实现私有自定义 tab

### 6.2 v1 已确认方案

v1 采用：

- 固定 tab
- 单行混排
- 不保留私有 tab 占位

当前已确认的 v1 tab 为：

- `全部`
- `我方`
- `敌方`

即：

- tab 只有一行
- 不做第二层 tab
- 不保留 `关注` 占位 tab
- 不在 v1 暴露任何自定义分组入口

## 7. 玩家私有列表的决策与依赖

### 7.1 原始目标

讨论中原本希望支持“每个玩家自己定义关注角色”的能力，也就是：

- 每个玩家拥有自己的关注列表
- 这些列表彼此独立
- 理想情况下能跨刷新、跨设备保留

### 7.2 现状判断

经检查，当前仓库**缺少一套通用的玩家私有数据持久化能力**。

现有能力大致分为：

- `sessionStorage` 级别的本地恢复：目前主要用于 `mySeatId`
- 房间级共享持久化：例如 layout
- 客户端内存态：例如 `uiStore` / `sessionStore`

但目前没有一套现成的、明确支持“按玩家私有、服务端持久化”的偏好存储机制。

### 7.3 本轮决策

因此，本轮讨论已确认：

- v1 不实现玩家私有关注列表
- v1 不实现私有自定义 tab
- 后续若要支持该能力，需要先补齐服务端侧的玩家私有偏好持久化基础设施

### 7.4 追踪记录

已经记录 GitHub issue：

- `#204 支持玩家私有 UI 偏好持久化（为 Daggerheart 关注列表等功能做准备）`

该 issue 的作用是把“私有关注 tab”从当前 UI 讨论中抽离出来，明确为一个前置基础设施问题，而不是临时塞进当前功能。

## 8. v1 范围总结

截至目前，右上战场总览面板的 v1 范围已经收敛为：

- 一个 Daggerheart persistent region
- 放在屏幕右上
- 作为战场总览面板使用
- 面向所有在场单位，而不是仅玩家角色
- 只读
- 使用单行固定 tab
- v1 tab 仅有：`全部 / 我方 / 敌方`
- 不包含玩家私有列表
- 不包含自定义 tab
- 不包含私有 tab 占位入口

## 9. 当前明确延后的内容

以下内容已明确不进入当前 v1：

- 玩家私有关注列表
- 玩家自定义 tab
- 私有 tab 管理 UI
- 依赖服务端私有偏好存储的功能
- 复杂面板操作能力

以下内容虽然历史 mockup 中出现过，但本轮尚未确认是否进入 v1：

- hover 详情卡
- 点击联动地图或角色
- 右上面板中的快捷操作按钮

## 10. 当前尚未定稿的问题

本轮讨论还没有确定以下内容：

### 10.1 单位行的信息密度

目前还没有最终决定单位行应该采用：

- 高密度
- 中密度
- 极简

也还没有最终决定：

- 玩家单位行是否直接显示 Armor / Hope
- 敌方单位行是否只显示 HP / Stress
- 状态在行内是显示数量，还是显示 1 至 2 个短标签

### 10.2 视觉样式

当前刻意没有定稿以下视觉问题：

- 面板具体宽度与高度
- 行高与内边距
- 颜色与装饰风格
- 是否沿用历史 mockup 的头像弧形资源环样式
- tab 的具体外观

原因是当前讨论阶段优先确认的是功能边界与结构，而不是视觉精修。

### 10.3 hover 详情卡是否进入 v1

历史 mockup 中已经探索过 hover 详情卡，但当前尚未决定：

- v1 是否同步做 hover 详情卡
- 或者先只做列表总览，后续再补 hover 细节

## 11. 当前建议的讨论顺序

在本文件基础上，后续建议按以下顺序继续推进：

1. 先定“单位行的信息层级”
2. 再定“hover 详情卡是否进入 v1”
3. 最后再定“视觉样式与布局细节”

这样的顺序能避免在视觉细节上投入过早，而功能边界仍未收敛。

## 12. 当前结论摘要

如果只保留本轮最重要的结论，可以简化为：

- 右上缺的是 Daggerheart 的战场总览面板
- 该面板面向所有在场单位，不只是玩家角色
- v1 先做只读
- v1 用固定单行 tab：`全部 / 我方 / 敌方`
- 不做玩家私有列表
- 玩家私有列表要等服务端私有偏好持久化能力补齐后再做
- GitHub 已有对应基础设施 issue：`#204`
