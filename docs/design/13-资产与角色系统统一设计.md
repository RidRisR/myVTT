# 资产与角色系统统一设计

> **状态：设计讨论阶段**
>
> 本文档记录了对 Asset（资产）与 Entity（实体）系统的设计讨论结论。
> 已达成核心方向共识，待决设计点需在后续迭代中逐个确定。
> 对应 Issue：#110 Redesign Blueprint/Entity data model relationship

## 背景与动机

当前系统的 Blueprint（token 模板）和 Character（可复用角色）存在两个层面的问题：

### 1. Asset 系统的职责越界

`assets` 表的 `type` 字段混合了两种不同概念：

- **文件类型**（image/audio/video）—— 由文件本身属性决定
- **游戏语义**（blueprint/handout）—— 由使用场景决定

Blueprint 作为 Asset 类型存在，携带了游戏参数（`extra.blueprint.defaultSize/defaultColor/defaultRuleData`），但这些参数本质上属于 Entity 系统。Asset 系统应当只负责文件的存储和提供 URL，不应承载"实体工厂"的角色。

### 2. GM 视角的 UI 割裂

Blueprint 和 Character 从 GM 视角做的是同一件事——"一个可以放到场景里的角色"，但操作路径完全分离：

| 维度     | BlueprintDockTab                  | CharacterLibraryTab                                  |
| -------- | --------------------------------- | ---------------------------------------------------- |
| 数据源   | `assets.filter(type='blueprint')` | `entities.filter(lifecycle!='ephemeral' && 无owner)` |
| 布局     | 网格（72px 圆形 token）           | 列表（24px 头像 + 名称）                             |
| 筛选     | TagFilterBar（AND 逻辑）          | 文本搜索                                             |
| 点击行为 | spawn 新 ephemeral Entity         | addEntityToScene（复用已有）                         |
| 创建方式 | 上传图片 → 创建 Asset             | 点击 + → 创建空白 Entity                             |

GM 需要在两个不同的 tab、两套不同的数据系统中管理本质上相似的东西。

## 设计原则

- **Asset 管文件，Entity 管角色** —— 职责边界清晰，不混合
- **标签替代硬分类** —— 用 tags 多对多管理，不用 type 做枚举分类
- **就地使用，自动归档** —— 每个使用处都能上传或选择已有资产，上传后自动进入统一资产池
- **渐进式充实** —— 允许先创建空白角色，后续慢慢补充数据；不强制预配置

## 已达成共识

### 共识 1：统一资产池 + 就地使用

Asset 系统退化为纯文件管理，不携带游戏语义。

**改造前**（当前）：

```
assets 表
├── type: 'image' | 'blueprint' | 'handout'   ← 混合了文件类型和游戏语义
├── tags: string[]
└── extra: { blueprint: {defaultSize, defaultColor, defaultRuleData} }  ← 游戏参数
```

**改造后**（目标）：

```
assets 表
├── mediaType: 'image' | 'video' | 'audio' | 'pdf'   ← 由文件 MIME 类型决定
├── tags: string[]                                      ← 用户自由打标签
└── (无游戏参数)
```

核心变化：

- `type` 字段改为 `mediaType`，反映文件本身属性而非游戏用途
- 去掉 `extra.blueprint` 中的游戏参数（defaultSize/defaultColor/defaultRuleData）
- 同一张图片可以被多处引用（既当地图背景又当 token 图）
- 每个使用处（设置场景背景、创建角色、设置战术地图等）都提供"从资产池选择 / 就地上传"的统一入口

### 共识 2：标签替代硬分类

用 `tags: string[]`（JSON 数组）做多对多管理，Asset 和 Entity 都通过 tags 组织。

**性能评估**：per-room 数据量（几十到几百条）下，客户端 `useMemo` + `Array.filter` + `Array.every` 筛选完全没有性能问题。现有 `TagFilterBar` 组件可复用。

如果未来需要服务端筛选，SQLite 支持 `json_each()` 查询 JSON 数组，但目前不需要。

### 共识 3：不自动同步模板变更

从模板 spawn 出来的 Entity 与源模板之间不自动同步。spawn 时复制默认值，之后 Entity 完全独立。

理由：简单可预测，战斗中途模板变了导致场上 token 突变会让 GM 困惑。

### 共识 4：现有模型中合理的部分（不改动）

以下设计已验证合理，不在本次重构范围内：

- **Entity + lifecycle 三分法**（ephemeral/reusable/persistent）—— 已覆盖 Foundry VTT 的 Linked/Unlinked 语义
- **MapToken 与 Entity 分离** —— 位置（场景级）≠ 身份（全局级），`UNIQUE(scene_id, entity_id)` 约束合理
- **blueprint_id 回溯引用** —— Entity 表已有 `blueprint_id` 字段
- **scene_entities M2M 关联** —— ephemeral 1:1 + reusable/persistent M:M 的约束合理
- **场景间角色状态** —— 由 lifecycle 决定（ephemeral 场景级独立，reusable/persistent 跨场景共享）

### 共识 5：素材来源

MVP 阶段只支持 GM 自行上传。数据模型预留素材库扩展空间但不设计。

## 行业参考

### Foundry VTT — Actor / Prototype Token / Token 三层模型

- **Actor** = 游戏实体，包含完整角色数据
- **Prototype Token** = Actor 上的默认外观配置（图片、大小、视野等）
- **Token** = 场景中的实例
- **Linked Token**：与 Actor 同步（适合 PC、Boss），修改 Token HP = 修改 Actor HP
- **Unlinked Token**：以 Actor 为模板，spawn 后独立（适合小怪）
- 我们的 `lifecycle` 三分法已覆盖此语义

### Owlbear Rodeo — 简洁分类 Dock

- Dock 按用途分 tab：Scenes / Maps / Props / Characters / Notes
- 角色就是一张图 + 名字，拖入场景即用
- 极简设计，适合轻量场景，缺乏复杂数据管理能力

### Roll20 — 资产管理的反面教材

- Art Library 只支持三层文件夹嵌套，Tag 支持差
- Character 和 Token 通过 Journal 手动绑定，流程割裂
- 社区多年持续抱怨资产管理体验
- **教训**：标签 > 文件夹，操作便利性 > 分类精确性

## Assumptions

- 本文档中的"统一资产池"方案假设 per-room 数据量始终在几百条以内，客户端 JSON 数组标签筛选足够
- Asset 表的 `type` 字段可以在迁移中改名为 `mediaType`，不影响现有外部引用（没有外部系统依赖此字段）
- Blueprint 概念迁移到 Entity 后，现有 `blueprint_id` 字段语义可以从"指向 assets.id"调整为"指向 entities.id"，无第三方依赖
- Handout 的 `extra.handout` 数据结构可独立评估，不必与 Blueprint 同步处理
- 现有的 spawn 路由和 "Save as Blueprint" / "Save as Character" 功能可以在迁移中重写，不需要向后兼容旧 API

## Edge Cases

- 已有 `type='blueprint'` 的 Asset 在迁移后如何处理 —— 需要迁移脚本将其转为 template Entity + 纯 image Asset
- `blueprint_id` 指向已删除的 Asset 时 —— 当前代码已容忍此情况（spawn 名称退回默认值），迁移后同理
- 同一张图片被多处引用（地图 + token）时的删除行为 —— 需要引用计数或"删除时警告"机制
- 无图片的 Entity 在统一网格 tab 中如何展示 —— 需要 fallback 视觉（纯色圆 + 首字母缩写）
- Entity tags 与 Asset tags 的分离 —— 两者独立管理，Entity tags 用于角色图鉴筛选，Asset tags 用于文件库筛选
- Maps Tab 在 `type` 字段消失后的筛选方式 —— 改用 `mediaType='image'` 或保留独立的 tag 约定
