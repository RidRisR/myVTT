# 资产与角色系统重构设计

> **状态：设计已定，待决问题审查中**
>
> 本文档记录了 Asset（资产）、Blueprint（蓝图）、Entity（实体）三系统的重构设计。
> 对应 Issue：#110 Redesign Blueprint/Entity data model relationship

## 背景与动机

当前系统的 Blueprint（token 模板）和 Character（可复用角色）存在两个层面的问题：

### 1. Asset 系统的职责越界

`assets` 表的 `type` 字段混合了两种不同概念：

- **文件类型**（image/audio/video）—— 由文件本身属性决定
- **游戏语义**（blueprint/handout）—— 由使用场景决定

Blueprint 作为 Asset 类型存在，携带了游戏参数（`extra.blueprint.defaultSize/defaultColor/defaultRuleData`），但这些参数本质上属于独立的模板系统。Asset 系统应当只负责文件的存储和提供 URL，不应承载"实体工厂"的角色。

### 2. Blueprint 放错了位置

Blueprint 和 Entity 的字段看似相似（name, image, color, size, ruleData），但语义完全不同：

- **Blueprint** 存储的是"生成参数"（default_width, default_color）—— **类**
- **Entity** 存储的是"实例状态"（当前 HP, 当前位置, 权限）—— **实例**

将 Blueprint 放在 Asset 表中（通过 `type='blueprint'` + `extra.blueprint`），导致了 UI 上 BlueprintDockTab 和 CharacterLibraryTab 的操作路径割裂，让人误以为"合并这两个 tab"能解决问题。实际上根源在于 Blueprint 的归属错误。

## 设计原则

- **Asset 管文件，Blueprint 管模板，Entity 管实例** —— 三者各司其职，不混合
- **标签替代硬分类** —— 用 tags 多对多管理，不用 type 做枚举分类
- **就地使用，自动归档** —— 通过 AssetPicker 在使用处上传或选择图片，上传后自动进入统一资产池
- **写入时合并，读取时分离** —— 用户操作一步完成（如"上传图片创建蓝图"），后端拆写多张表

## 数据模型设计

### 三表分工

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   assets     │     │  blueprints  │     │   entities   │
│  (纯文件)     │     │  (模板工厂)   │     │  (游戏实例)   │
│              │     │              │     │              │
│  id          │◄────│  image_url ──│─┐   │  id          │
│  url         │     │  name        │ │   │  name        │
│  name        │     │  defaults {} │ │   │  image_url   │
│  media_type  │     │  tags []     │ │   │  color       │
│  tags []     │     │  created_at  │ │   │  width/height│
│  created_at  │     │              │ │   │  rule_data   │
│              │     │              │ │   │  lifecycle   │
│              │     │              │ │   │  blueprint_id│──► blueprints.id
│              │     │              │ │   │  tags []     │
└──────────────┘     └──────────────┘ │   └──────────────┘
                                      │
                           同一张图可被多处引用
```

### assets 表（纯文件管理）

```sql
CREATE TABLE IF NOT EXISTS assets (
  id TEXT PRIMARY KEY,
  url TEXT NOT NULL,
  name TEXT DEFAULT '',
  media_type TEXT NOT NULL DEFAULT 'image',  -- 由 MIME 类型决定: image/audio/video/pdf
  tags TEXT DEFAULT '[]',
  created_at INTEGER NOT NULL,
  extra TEXT DEFAULT '{}'  -- handout 专用 (title/description)
);
```

改造要点：

- `type` → `media_type`，反映文件本身属性而非游戏用途
- 移除 `extra.blueprint` 中的游戏参数
- 同一张图片可被多处引用（既当地图背景又当 token 图）
- Handout 的 `extra.handout` 暂时保留（handout 只是"展示内容"，没有"实例化"行为，留在 Asset 表合理）

### blueprints 表（新增，模板工厂）

```sql
CREATE TABLE IF NOT EXISTS blueprints (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL DEFAULT '',
  image_url TEXT DEFAULT '',
  tags TEXT DEFAULT '[]',
  defaults TEXT DEFAULT '{}',  -- JSON blob: { color, width, height, ruleData, ... }
  created_at INTEGER NOT NULL
);
```

设计决策与推导过程：

1. **为何独立建表，而非 `lifecycle='template'`？**
   Blueprint 不在场景中存活，`lifecycle`（ephemeral/reusable/persistent）对它毫无意义。将 template 塞进 lifecycle 会把"场景存活方式"和"是否为模板"两个正交维度压成一个字段。

2. **为何独立建表，而非 `is_template` 字段？**
   即使加了 `is_template` 标记，entities 表中的 `lifecycle`、`permissions`、`notes` 等字段对 template 行仍然不适用。字段结构 90% 重叠是因为 Blueprint 需要提供默认值、Entity 需要记录当前状态——目的不同，碰巧字段名一样。"结构相似 ≠ 语义相同"。

3. **为何用 JSON blob 存默认值，而非拆列？**
   Blueprint 是只读模板，使用模式是"整体读取 → 复制到新 Entity"。没有 `WHERE default_color = 'red'` 这种查询需求，不需要列级索引。JSON blob 更灵活：Entity 表加新字段时 Blueprint 表不用动，只需改 `defaults` 里的 JSON 结构。

### entities 表（调整）

```sql
CREATE TABLE IF NOT EXISTS entities (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL DEFAULT '',
  image_url TEXT DEFAULT '',
  color TEXT DEFAULT '#888888',
  width REAL NOT NULL DEFAULT 1,
  height REAL NOT NULL DEFAULT 1,
  notes TEXT DEFAULT '',
  rule_data TEXT DEFAULT '{}',
  permissions TEXT DEFAULT '{"default":"none","seats":{}}',
  lifecycle TEXT DEFAULT 'ephemeral' CHECK(lifecycle IN ('ephemeral','reusable','persistent')),
  blueprint_id TEXT REFERENCES blueprints(id) ON DELETE SET NULL,  -- 改为指向 blueprints 表
  tags TEXT DEFAULT '[]'  -- 新增，用于统一标签筛选
);
```

改造要点：

- `blueprint_id` 从指向 assets.id 改为指向 blueprints.id
- `ON DELETE SET NULL`：删除蓝图不影响已 spawn 的 entity
- 新增 `tags` 字段用于 Character 的标签筛选
- lifecycle 三分法不变（ephemeral/reusable/persistent）

## Spawn 流程

```ts
// POST /api/rooms/:roomId/scenes/:sceneId/spawn
// body: { blueprintId, tacticalOnly? }

const bp = db.prepare('SELECT * FROM blueprints WHERE id = ?').get(blueprintId)
const defaults = JSON.parse(bp.defaults) // { color, width, height, ruleData }

// 计算递增名称
const count = db
  .prepare('SELECT COUNT(*) as c FROM entities WHERE blueprint_id = ?')
  .get(blueprintId).c
const name = `${bp.name || 'NPC'} ${count + 1}`

// 从 blueprint 复制默认值到新 entity
db.prepare('INSERT INTO entities (...) VALUES (...)').run(
  newId,
  name,
  bp.image_url,
  defaults.color ?? '#888888',
  defaults.width ?? 1,
  defaults.height ?? 1,
  JSON.stringify(defaults.ruleData ?? {}),
  'ephemeral',
  blueprintId,
)
```

## UI 架构

### AssetPicker：统一图片选择组件

所有需要选择图片的地方都使用同一个 AssetPicker 组件：

```
┌─ 选择图片 ─────────────────────┐
│                                 │
│  ┌────┐ ┌────┐ ┌────┐ ┌────┐  │  ← 从资产池浏览已有图片
│  │ 🖼 │ │ 🖼 │ │ 🖼 │ │ 🖼 │  │
│  └────┘ └────┘ └────┘ └────┘  │
│                                 │
│           ┌──────────────┐     │
│           │  + 上传新图片  │     │  ← 就地上传，自动归档到资产池
│           └──────────────┘     │
└────────────────────────────────┘
```

使用场景：
| 调用方 | 选图后的动作 |
|--------|-------------|
| Blueprint "+" 按钮 | 创建 blueprint 记录（后端写 assets + blueprints 两张表，用户只做一步） |
| Entity 换头像 | 更新 entity.image_url |
| 场景背景设置 | 更新 atmosphere.imageUrl |
| 战术地图设置 | 更新 tactical_state.map_url |

**关键 UX 原则**：用户不需要先去"资产管理"上传图片再回来创建蓝图/角色。上传和创建在同一个流程内一步完成。资产池对用户来说是透明的——它只在"复用已有图片"时自然浮现。

### 资产管理入口

虽然日常操作通过 AssetPicker 完成，但仍需提供一个**次级入口**让 GM 能浏览和管理所有资产（批量清理、重命名、打标签、查看未引用文件、了解存储用量等）。

这个入口不是 Dock 里的一级 tab，而是作为次级功能存在（如设置菜单、AssetPicker 内的"管理全部"链接、或 Dock 底部的工具图标）。具体位置待 UI 实现时确定。

### Dock Tab 结构

```
┌─────────────────────────────────────┐
│  [Maps]  [Blueprints]  [Characters] │
└─────────────────────────────────────┘
```

| Tab        | 数据源                                   | 展示形式                       | 点击行为                  | 创建方式                          |
| ---------- | ---------------------------------------- | ------------------------------ | ------------------------- | --------------------------------- |
| Maps       | assets 表 (media_type='image')           | 缩略图网格                     | 设置场景背景/战术地图     | AssetPicker 上传                  |
| Blueprints | blueprints 表                            | 圆形 token 网格 + TagFilterBar | spawn 新 ephemeral entity | AssetPicker 选图 → 创建 blueprint |
| Characters | entities 表 (reusable/persistent, 非 PC) | 列表 + 文本搜索                | addEntityToScene 复用     | 创建空白 entity                   |

**去掉了 Gallery tab**：不再有独立的"资产浏览"界面。资产池只通过 AssetPicker 在各使用处按需暴露。

Blueprints 和 Characters 保持独立 tab，因为：

- 语义不同：模板（工厂）vs 实例（角色）
- 操作不同：spawn 创建新实例 vs 复用已有实例
- 之前想合并是因为 Blueprint 寄生在 Asset 表导致的"看起来割裂"。Blueprint 独立建表后，割裂的根源消失了，两者本来就是不同的东西，不需要合并。

## 已达成共识

### 共识 1：Asset / Blueprint / Entity 三表分离

- Asset 系统退化为纯文件管理（media_type 由 MIME 决定，不携带游戏语义）
- Blueprint 独立建表，JSON blob 存默认值
- Entity 保持不变，`blueprint_id` 改为指向 blueprints 表

### 共识 2：标签替代硬分类

用 `tags: string[]`（JSON 数组）做多对多管理，Asset、Blueprint、Entity 都通过 tags 组织。

**性能评估**：per-room 数据量（几十到几百条）下，客户端 `useMemo` + `Array.filter` + `Array.every` 筛选完全没有性能问题。现有 `TagFilterBar` 组件可复用。

### 共识 3：不自动同步模板变更

从 Blueprint spawn 出来的 Entity 与源 Blueprint 之间不自动同步。spawn 时复制默认值，之后 Entity 完全独立。

理由：简单可预测，战斗中途模板变了导致场上 token 突变会让 GM 困惑。

### 共识 4：AssetPicker 统一入口

所有需要选图的地方使用同一个 AssetPicker 组件，支持"从资产池选择 / 就地上传新图"。后端"写入时合并"（一个 API 调用同时写 assets + blueprints），用户感知不到多表操作。

### 共识 5：Blueprints 与 Characters 保持独立 Tab

不合并为"角色图鉴"。两者语义不同（模板 vs 实例），操作不同（spawn vs 复用），保持独立更清晰。Gallery tab 移除，资产池只通过 AssetPicker 暴露。

### 共识 6：现有模型中合理的部分（不改动）

以下设计已验证合理，不在本次重构范围内：

- **Entity + lifecycle 三分法**（ephemeral/reusable/persistent）—— 已覆盖 Foundry VTT 的 Linked/Unlinked 语义
- **MapToken 与 Entity 分离** —— 位置（场景级）≠ 身份（全局级），`UNIQUE(scene_id, entity_id)` 约束合理
- **scene_entities M2M 关联** —— ephemeral 1:1 + reusable/persistent M:M 的约束合理
- **场景间角色状态** —— 由 lifecycle 决定（ephemeral 场景级独立，reusable/persistent 跨场景共享）

### 共识 7：素材来源

MVP 阶段只支持 GM 自行上传。数据模型预留素材库扩展空间但不设计。

## 行业参考

### Foundry VTT — Actor / Prototype Token / Token 三层模型

- **Actor** = 游戏实体，包含完整角色数据
- **Prototype Token** = Actor 上的默认外观配置（图片、大小、视野等）
- **Token** = 场景中的实例
- **Linked Token**：与 Actor 同步（适合 PC、Boss），修改 Token HP = 修改 Actor HP
- **Unlinked Token**：以 Actor 为模板，spawn 后独立（适合小怪）
- 我们的 `lifecycle` 三分法已覆盖此语义
- **文件与游戏对象完全分离**：上传图片 → 进入文件管理器；创建 Actor → 从文件管理器选图。用户从不在上传时决定"这是什么"。我们的 AssetPicker 方案对标此设计。

### Owlbear Rodeo — 简洁分类 Dock

- Dock 按用途分 tab：Scenes / Maps / Props / Characters / Notes
- 角色就是一张图 + 名字，拖入场景即用
- 极简设计，适合轻量场景，缺乏复杂数据管理能力

### Roll20 — 资产管理的反面教材

- Art Library 只支持三层文件夹嵌套，Tag 支持差
- Character 和 Token 通过 Journal 手动绑定，流程割裂
- 社区多年持续抱怨资产管理体验
- **教训**：标签 > 文件夹，操作便利性 > 分类精确性

## Assumptions

- per-room 数据量始终在几百条以内，客户端 JSON 数组标签筛选足够
- 不需要考虑向后兼容，所有路由和功能可以直接重写（开发阶段，无生产环境）
- Handout 的 `extra.handout` 数据结构可独立评估，暂留在 Asset 表
- Blueprint 概念未来可扩展到非 Entity 类型（如场景模板、遭遇战模板），通过 `defaults` JSON 的不同结构支持，但当前只实现 Entity Blueprint

## Edge Cases

- 同一张图片被多处引用（地图 + token）时的删除行为 —— 需要引用计数或"删除时警告"机制
- 无图片的 Entity 在 Characters tab 中如何展示 —— fallback 视觉（纯色圆 + 首字母缩写）
- Entity tags 与 Blueprint tags 与 Asset tags 三者独立管理
- 删除 Blueprint 时已 spawn 的 Entity 不受影响（`ON DELETE SET NULL`）
- `blueprint_id` 指向已删除的 Blueprint 时 —— Entity 正常存在，仅名称递增功能失效

## 审查待决问题

以下问题在设计审查中提出，需要在实现前确定方案。

### 问题 1：image_url 裸 URL vs asset_id 外键

当前设计中 `blueprints.image_url` 和 `entities.image_url` 都是裸 URL 字符串，没有外键指向 `assets.id`。

**影响**：

- 删除 asset 时无法级联：GM 删了一张图片，blueprints/entities 中的 `image_url` 变成悬空引用，数据库层面无感知
- 引用计数难以实现：文档提到"删除时警告"，但没有外键关系，需要全表扫描所有 `image_url` 字段才能判断引用
- 同一张图片在不同 room database 中 URL 路径不同时，字符串匹配不可靠

**可选方案**：

- A）改用 `image_asset_id TEXT REFERENCES assets(id) ON DELETE SET NULL`，通过 JOIN 拿 URL
- B）保持裸 URL，接受其局限性，在文档中明确"引用计数通过 URL 字符串匹配实现"

### 问题 2：Maps Tab 筛选会混入 token 图片

重构后 Maps tab 使用 `media_type='image'` 筛选，但上传的 token 图片也是 image 类型——会同时出现在 Maps tab 中，导致 token 图淹没地图素材。

**可选方案**：

- A）上传时自动或手动打 tag（如 `map` / `token`），Maps tab 按 tag 筛选
- B）AssetPicker 在不同调用场景传入不同的 tag 预设
- C）Maps tab 只展示被当前 room 的场景实际引用过的图片（按使用筛选而非按类型筛选）

### 问题 3：Spawn 命名计数范围

设计中的 spawn 流程使用 `SELECT COUNT(*) FROM entities WHERE blueprint_id = ?` 做全局计数。

**问题**：如果 GM spawn 了 "Goblin 1-5" 后全部删除，再 spawn 会从 "Goblin 1" 重新开始（COUNT 为 0）。如果中间删了 3 号，下一个仍是 "Goblin 6"（COUNT 为 4 → +1=5...不对，COUNT 为 4 → "Goblin 5"，但 "Goblin 5" 已经存在）。

**建议**：改用 `MAX(序号)` 而非 `COUNT`，从已有同 blueprint 的 entity 名称中提取最大编号 +1。当前 `nextNpcName()` 的实现已经在做这件事，spawn 流程应复用而非重写。

### 问题 4：缺少迁移方案具体步骤

文档说"不需要向后兼容"，但现有数据库中已有 `type='blueprint'` 的 asset 和引用它们的 `entity.blueprint_id`。需要一个迁移脚本框架：

```
1. CREATE TABLE blueprints (...)
2. 对每个 type='blueprint' 的 asset：
   a. INSERT INTO blueprints (id, name, image_url, tags, defaults, created_at)
      VALUES (asset.id, asset.name, asset.url, asset.tags, asset.extra.blueprint, asset.created_at)
   b. UPDATE assets SET media_type='image', extra='{}' WHERE id = asset.id
      （或直接 DELETE，取决于该图片是否还需要保留在资产池中）
3. ALTER TABLE assets: RENAME COLUMN type → media_type
4. entities.blueprint_id 的值不需要改（如果 blueprint 复用了原 asset 的 id）
```
