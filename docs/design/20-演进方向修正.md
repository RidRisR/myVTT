# 20 — 演进方向修正

> **状态**：✅ 活跃参考 | 2026-04-08

## 一、背景

原三轨演进模型（A: 事件日志、B: UI 插件化、C: Workflow 化）的核心汇聚点——RulePlugin 退役——已在 commit aa5e9d5 中提前完成，绕过了原计划要求的 B6/B8 依赖。同时 EventBus 在 PR #179 中被完全移除而非迁移。这两项变化使原模型的前提假设失效。

## 二、当前架构现状

### 已达成

- **VTTPlugin 单一架构**：RulePlugin 接口完全删除，所有插件通过 VTTPlugin.onActivate(sdk) 接入
- **EventBus 移除**：跨客户端效果由 RendererRegistry + game_log 驱动
- **Entity Component System**：Phase 4 迁移完成，entity_components 表运行中
- **类型安全**：LogPayloadMap + ComponentTypeMap 编译时契约
- **RendererRegistry**：替代了 ExtensionRegistry 和 RulePlugin.rollCardRenderers
- **输入处理器系统**：registerInputHandler 支持 workflow 内交互式输入

### 未达成（原计划中）

- formulaTokens 废弃：仍在 baseWorkflows.ts 活跃使用，不可简单移除
- 部分 DH 操作的 Workflow 化：fear/hope 在 action-check resolve 中集成，非独立 workflow

## 三、战略方向调整

### 北极星

**让 Daggerheart 完全可玩。**

### 方法论

从原来的「三轨结构化演进」转为「实践驱动」：以 Daggerheart 插件的具体需求拉动基座 SDK 的完善，而非投机性的顶层架构设计。

### 最高优先级：UI 注册系统灵活性

当前 UI 系统的根本瓶颈：PanelRenderer 只有一种渲染模式——position: absolute 的裸 div。所有窗口行为（标题栏、关闭、拖拽、折叠、resize）都需要插件自行实现。

**具体缺失：**

| 能力                               | 现状                              | 影响                       |
| ---------------------------------- | --------------------------------- | -------------------------- |
| 窗口 chrome（标题栏、关闭/最小化） | 无                                | 插件各自实现，体验不统一   |
| Play 模式 resize                   | 仅 edit mode                      | 用户游戏中无法调整面板大小 |
| 窗口状态（最小化/折叠）            | LayoutEntry 仅存 x/y/width/height | 无法最小化窗口             |
| Modal/Dialog SDK 暴露              | Radix Dialog 已安装未接入 SDK     | 插件无法使用对话框         |
| 抽屉/侧边栏                        | 无                                | 无 docking 系统            |
| 可折叠区段                         | 无                                | 无 accordion 原语          |

**阻塞效果：** 无法展示完整角色卡（需要浮动窗口 + 标题栏 + resize + 关闭按钮）。

此项的具体设计方案待定，需在实施前单独出设计文档。

## 四、已确认决策

| 决策               | 说明                                                | 优先级                 |
| ------------------ | --------------------------------------------------- | ---------------------- |
| team_trackers 退役 | POC 权宜之计，统一到实体路径（entity + components） | 不阻塞，随时可做       |
| showcase 退役      | POC 权宜之计，叶子系统无依赖                        | 不阻塞，随时可做       |
| 插件私有持久化表   | 暂缓，先用 entity.components                        | 需求不明确时不提前设计 |
| formulaTokens      | 维持现状，仍在活跃使用                              | 不动                   |

## 五、待定事项

以下事项已识别但未展开设计，待实践中遇到时再具体规划：

- **UI 系统重构具体方案**：窗口 chrome 包装器 vs 组合式 SDK，需设计文档
- **地图交互工具**：registerTool SDK 扩展，支持插件注册地图工具
- **手牌/法术系统**：远期目标，依赖 UI 系统和实体模型的进一步完善
- **基座调用 Workflow 化**：将更多原生操作纳入 Workflow 系统
- **Fear 计数器双轨统一**：FearManager（实体路径）与 DHTeamPanel（team_trackers 路径）需统一，随 team_trackers 退役一并处理
