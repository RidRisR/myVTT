# 骰子防伪方案设计

## 问题

当前骰子完全在客户端执行：`ChatInput.handleRoll()` → `rollCompound()` → `yChat.push([message])`。任何客户端都可以绕过 UI 直接往 `chat_log` 写入伪造的骰子结果。对于 TRPG 中关乎角色生死的骰子判定，需要服务端权威的投骰机制来保证公正性。

## 设计原则

- **零误报**：宁可漏报也不误报。只有确定性信号才会触发警告
- **用户无感**：正常情况下用户完全不知道防伪系统的存在。不显示"已验证"、不显示 ✓ 徽章
- **渲染不等验证**：消息到达后立即渲染动画，审计在服务端静默进行
- **零签名**：不需要 HMAC 或非对称签名——transaction.origin 天然区分服务端写入与客户端写入

## 架构

### 两条独立的线

```
线1 投骰（客户端 → 服务端 → Y.Doc）
    玩家输入 .r 1d20
    → 客户端解析 @key 引用（需要 token/seat 属性上下文）
    → POST /api/rooms/:roomId/roll（发送 formula + 身份信息）
    → 服务端 rollCompound(resolvedExpression) 生成结果
    → 服务端 doc.transact(() => chatLog.push([message]), 'server')
    → Yjs 自动同步给所有客户端

线2 渲染（Y.Doc → UI）
    客户端 Yjs observer 收到新 roll 消息
    → 立即渲染骰子动画
    → 检查本地 rollId Set（防重放）
    → 重复 → 标记重放 | 正常 → 加入 Set
```

两条线完全解耦，互不阻塞。

### 安全模型

```
服务端                               客户端
─────────────────────────           ──────────────
transaction.origin 审计              已见消息 ID 集合（本地去重）
骰子只有 origin='server' 才合法      rollId 重复 = 重放
```

## 安全机制

### 1. transaction.origin 审计（防伪造）

服务端 observer 通过 `transaction.origin` 判断消息来源：

```
chat_log observer(event, transaction):
  origin = transaction.origin
  isServer = (origin === null || origin === 'server')

  对于每条新增消息：
    if message.type === 'roll' && !isServer:
      → 骰子消息只有服务端才能写入，这是伪造
      → append { type: 'system', subtype: 'tamper_alert' } 到 chat_log
```

**为什么不需要签名？** y-websocket 在 `applyUpdate` 时将 WebSocket 连接对象作为 origin 传入（已通过代码验证：y-websocket/bin/utils.cjs 第189行 → y-protocols/sync.js 第84行 → Y.applyUpdate 第三参数）。服务端自己写入时 origin 为 `'server'`，客户端写入时 origin 为 conn 对象。客户端无法伪造这个过程——它发生在服务端内存中。

**为什么比 HMAC/Ed25519 更好？**

| | HMAC | Ed25519 | transaction.origin |
|--|------|---------|-------------------|
| 密钥管理 | SECRET_KEY 持久化 | 私钥持久化 + 公钥分发 | 无 |
| 验证位置 | 服务端 HTTP 端点 | 客户端本地 | 服务端 observer |
| 额外网络请求 | 每批消息一次 | 无 | 无 |
| 消息格式改动 | +signature 字段 | +signature+keyId 字段 | 无 |
| 实现复杂度 | 签名函数+验证端点+客户端hook | 密钥生成+WebCrypto+分发 | 一个 if 判断 |

### 2. 客户端去重（防重放）

每个客户端维护一个 `Set<string>`（以 rollId 为 key）。observer 收到新 roll 消息时，检查是否已见过相同 rollId。重复则标记为重放。

**为什么是客户端而不是服务端？** 重放的消息是从合法消息复制的，服务端 observer 看到的 origin 仍然是客户端 conn，虽然能检测到"客户端写了骰子"（伪造），但无法区分"伪造全新骰子"和"复制已有骰子"。客户端通过"同一个 rollId 出现了两次"来确定性地判断重放——零误报。

**初始加载时的处理：** 历史消息的 rollId 在初始加载时全部加入 Set（只记录，不检查重复），之后 observer 的新消息才做去重检查。与 `initialLoadRef` 模式配合。

### 3. 只审计新消息

服务端 observer 只对实时到达的新消息执行审计。初始 Yjs 同步加载的历史消息不重新审计。

**为什么不审计历史？** 历史消息已经发生，结果已经被使用。且审计日志中已有完整记录。

## 表达式透明性

客户端发送 `formula`（用户输入的原始公式）和 `resolvedExpression`（@key 解析后的实际公式）给服务端。服务端用 `resolvedExpression` 掷骰，并将两者都写入消息。

UI 同时展示原始公式和实际计算的表达式，让玩家自行判断是否正确。服务端不校验两者的一致性——这是社交约束的范畴。UI 视觉层次区分：formula 主色大字，resolvedExpression 辅色小字。

## 攻击场景分析

| 攻击方式 | 过程 | 结果 |
| --- | --- | --- |
| **凭空伪造** | 客户端 push type:'roll' 到 chat_log | origin 是 conn 不是 'server' → 服务端 append 篡改警告 |
| **篡改数字** | 复制合法消息改 total | 客户端无法修改已同步的 Y.Array 元素内容（Yjs immutable content）；如果重新 push 则 origin 暴露 |
| **重放攻击** | 复制一条合法消息再 push | 服务端检测到客户端写骰子 → 篡改警告；客户端检测到重复 rollId → 标记重放 |
| **刷骰子取最优** | 连续发多次 roll 请求 | 不限流，所有结果对所有人可见，刷骰行为自证 |
| **表达式欺骗** | formula 写 1d20，实际发 1d20+100 | resolvedExpression 展示在 UI 中，其他玩家可见 |
| **删除消息** | 从 chat_log 中删除不利的骰子记录 | observer 检测到 deletion → append 篡改警告（详见 52-聊天记录完整性） |

## 补充约束

### chat_log 禁止绑定 Y.UndoManager

> **规则：`chat_log` Y.Array 永远不可绑定到任何 Y.UndoManager 实例。**

如果未来添加富文本编辑（如聊天输入框的编辑历史），其 UndoManager 必须 scope 到编辑器自身的 Y.Text，绝不能触及 chat_log——否则 Ctrl+Z 可能撤销服务端写入的骰子消息。

## 不变的部分

- 文本消息（`type: 'text'`）仍走 Yjs 直接写入
- 骰子动画逻辑不变（DiceReel 组件）
- 旧消息（无防伪字段的历史消息）正常展示，不验证不标记
- `diceUtils.ts` 的类型导出保持不变
