# 20 — UI 注册系统扩展方案

> **状态**：✅ 已实现（Input Handler 部分）| 2026-03-31 — PR #181 已合并
> 前置文档：[19-DaggerHeart 插件演进-头脑风暴](19-DaggerHeart插件演进-头脑风暴.md)

---

## 背景

在 DaggerHeart 插件演进的头脑风暴中，我们发现 UI 注册系统是唯一需要架构扩展的部分（实体系统和插件迁移是设计决策问题，不是基础设施缺口）。本文档记录能力评估结论和扩展方案。

---

## 一、现有能力评估

### 已具备的能力

| 能力               | 实现方式                                        | 备注                                                                               |
| ------------------ | ----------------------------------------------- | ---------------------------------------------------------------------------------- |
| 持久面板           | `registerComponent` + `PanelRenderer`           | 支持拖拽、缩放、布局持久化                                                         |
| 日志卡片渲染       | `RendererRegistry` 注册 `(surface, type)`       | 已有 typed token API                                                               |
| 面板内弹出选项     | Radix Portal（默认行为）                        | Radix Popover/Dropdown/Dialog 自动 Portal 到 body，**不受** `overflow:hidden` 裁剪 |
| 面板打开/关闭      | `sdk.ui.openPanel` / `closePanel`               | `IComponentSDK` 已提供                                                             |
| Workflow 暂停/恢复 | `requestInput` / `resolveInput` / `cancelInput` | `sessionStore.ts` 已实现                                                           |

### 关键发现：Radix Portal 绕过了 overflow:hidden

`PanelRenderer` 对每个面板设置了 `overflow: hidden` + `contain: layout paint`（`PanelRenderer.tsx:59-60`），这一度被认为是重大限制。但 Radix UI 组件默认通过 Portal 渲染到 `document.body`，弹出层不受面板容器裁剪。因此**面板内使用 Radix 弹出组件无需任何改造**。

### 不需要新基础设施的需求

| 需求                     | 方案                                                                                                    |
| ------------------------ | ------------------------------------------------------------------------------------------------------- |
| 上下文菜单（右键 token） | RendererRegistry 模式 — 在 token 处添加注册点 `registerRenderer('token-context-menu', type, component)` |
| 悬浮提示（hover token）  | 同上 — 注册点 `registerRenderer('token-tooltip', type, component)`                                      |
| EntityCard 迁移          | 肖像栏暴露注册点，插件通过统一 SDK + `registerRenderer('portrait-slot', type, component)` 注册          |
| 跨玩家交互               | Workflow 编排模式（见下文第三节），不需要新 UI 能力                                                     |
| 面板固定/取消固定        | Popover → `openPanel`（转为持久面板），反向同理。`openPanel` 需支持接收初始坐标                         |

## 二、需要新增的基础设施

### 唯一缺口：requestInput ↔ 瞬态 UI 桥接

当前 `requestInput` 返回 `Promise<unknown>`，可以暂停 workflow 等待用户输入，但**没有自动打开收集输入的 UI 的机制**。这是连接 workflow 和用户交互的最后一环。

### 设计方案

#### 2.1 注册输入处理器

插件注册能够处理特定输入类型的组件：

```typescript
// 插件 onActivate 中注册
sdk.ui.registerInputHandler('dh:dice-modifiers', {
  component: DiceModifierPanel,
  // 组件自己决定呈现方式（Dialog / Popover / inline）
})
```

- 输入类型名带命名空间，避免冲突（`dh:dice-modifiers`、`dnd:ability-modifiers`）
- 组件自行决定渲染形态（Radix Dialog 居中、Radix Popover 锚定、自定义位置等）
- 平台不约束定位方式，组件有完全的呈现自由

#### 2.2 requestInput 桥接流程

```
Workflow 调用 ctx.requestInput<T>(inputType, context?)
  → 系统查找 registerInputHandler(inputType) 的注册组件
  → 渲染该组件（通过 Portal 到 body 层，不进入 layout 系统）
  → 用户操作 → 组件调用 resolve(value) 或 cancel()
  → 组件卸载
  → Workflow 继续，拿到结果
```

#### 2.3 输入处理组件的 Props

```typescript
interface InputHandlerProps<TContext, TResult> {
  context: TContext // workflow 传来的上下文数据
  resolve: (value: TResult) => void // 提交结果
  cancel: () => void // 取消输入
}
```

- `context` 由 workflow 调用方自行编写，平台不做自动注入
- 组件通过 `resolve` / `cancel` 控制生命周期
- 不注入完整 `IComponentSDK` — 输入处理器是轻量的瞬态组件，不需要面板级能力

#### 2.4 requestInput API

```typescript
interface RequestInputOptions<TContext> {
  context?: TContext    // 传递给组件的上下文
  timeout?: number      // 可选超时（毫秒），默认无超时
}

interface InputResult<T> =
  | { ok: true; value: T }
  | { ok: false; reason: 'cancelled' | 'timeout' }

// workflow 侧调用
const result: InputResult<DiceModifiers> = await ctx.requestInput<DiceModifiers>(
  'dh:dice-modifiers',
  { context: { attribute: 'strength', entityId: 'hero-001' } }
)

if (result.ok) {
  // result.value: DiceModifiers
} else {
  // result.reason: 'cancelled' | 'timeout'
}
```

#### 2.5 设计决策

| 决策                           | 理由                                               |
| ------------------------------ | -------------------------------------------------- |
| 默认无超时                     | 桌游场景下永久等待是合理的（玩家可能暂时离开）     |
| 取消返回 Result 类型而非抛异常 | 迫使 workflow 作者显式处理取消情况，不会遗漏       |
| 组件控制渲染形态               | 平台无法预知所有定位需求，组件最了解自己的呈现场景 |
| Portal 渲染，不进入 layout     | 瞬态面板不应该被用户拖拽或持久化                   |
| context 由调用方编写           | 平台无需知道各输入类型的上下文结构                 |

## 三、跨玩家交互方案

### 问题

GM 触发 workflow 后，需要某个玩家做出选择（如：掷豁免骰）。是否需要跨客户端的 `requestInput`？

### 结论：不需要。用 workflow 编排解决。

**业界参考**：几乎没有 VTT 将"推送 UI 到玩家屏幕"作为核心能力。Foundry 的主流模式是聊天卡片 + 自愿响应。

**我们的方案** — 编排（choreography）而非指挥（orchestration）：

```
1. GM workflow → 发出日志 (type: 'dh:save-request', triggerable: true, target: playerX)
2. 日志通过 Socket.io 广播到所有客户端
3. 玩家 X 的客户端 trigger 监听到 'dh:save-request' → 自动启动本地 workflow
4. 本地 workflow 调用 requestInput → 弹出本地模态框 → 玩家操作
5. 本地 workflow 完成 → 发出结果日志 (type: 'dh:save-result')
6. GM 客户端 trigger 监听到 'dh:save-result' → 继续处理
```

**优势**：

- 所有通信走现有的日志 + Socket.io 广播，不需要新传输机制
- `requestInput` 保持纯本地，复杂度大幅降低
- 每个客户端只运行自己的 workflow，架构简洁
- 聊天中的日志卡片只做信息展示，不需要交互能力

## 四、注册点扩展规划

以下注册点使用现有 `registerRenderer(surface, type, component)` 模式，不需要新 API：

| 注册点         | Surface 名           | 宿主负责                           | 插件负责              |
| -------------- | -------------------- | ---------------------------------- | --------------------- |
| Token 右键菜单 | `token-context-menu` | 监听右键事件，收集注册项，渲染菜单 | 注册菜单项组件        |
| Token 悬浮提示 | `token-tooltip`      | 监听 hover 事件，在 token 旁渲染   | 注册 tooltip 内容组件 |
| 肖像栏角色卡   | `portrait-slot`      | 提供插槽位置，传递 entity 数据     | 注册 EntityCard 组件  |
| 聊天日志卡片   | `chat`（已有）       | LogEntryCard 查找渲染器            | 注册卡片渲染组件      |

宿主（canvas 层、肖像栏、聊天面板）负责定义注册点的接口契约（传什么 props），插件按契约注册组件。
