# 实体生命周期重构设计

## 背景与动机

当前实体系统存在以下体验问题：

1. **创建 NPC 流程繁琐** —— 需要从蓝图上传到蓝图区 → 从蓝图创建实体 → 将实体加入场景，至少 3 步操作
2. **一次性 NPC 污染实体库** —— 所有 NPC 都永久存在于全局 `entities` 表中，即使只在一个场景用过一次。类似 Foundry VTT 社区公认的 Actors 目录膨胀问题
3. **缺少"离场"概念** —— 实体要么在场景中（角色栏可见），要么不在。GM 无法预先安排 NPC 到场景中但暂不上场
4. **UI 布局不合理** —— 蓝图库（原 Token tab）命名混淆，缺少独立的角色库，PC 和 NPC 混在同一个面板中

## 设计原则

- **轻备团优先** —— 一次性 NPC 一键生成，用完即弃
- **直觉式交互** —— 底部 Dock = 素材来源（跨场景），侧面面板 = 当前场景管理
- **同步删除，无 GC** —— 不需要后台垃圾回收，删除在路由中同步完成，用 SQLite 事务保证原子性
- **延迟删除 + Toast 撤销** —— 删除操作先从 UI 隐藏，延迟 5 秒后真正执行，期间可通过 Toast 撤销

## 数据模型

### 两张核心表

```
entities 表（所有角色，全局唯一）
───────────────────────────────────────────────────
id | name | lifecycle | image_url | color | size | ...
───┼──────┼───────────┼───────────┼───────┼──────┼────
e1 | 战士  | persistent| ...       | ...   | 1    |
e2 | 商人  | reusable  | ...       | ...   | 1    |
e3 | 哥布林| ephemeral | ...       | ...   | 1    |

scene_entities 表（角色-场景关联 + 可见性）
─────────────────────────────────
scene_id | entity_id | visible
─────────┼───────────┼────────
tavern   | e1        | 1        ← 战士在酒馆，在场
tavern   | e2        | 1        ← 商人在酒馆，在场
tavern   | e3        | 1        ← 哥布林在酒馆，在场
tavern   | e6        | 0        ← 暗杀者在酒馆，离场
forest   | e1        | 1        ← 战士在森林
```

### schema 定义

**entities 表**：使用 `lifecycle` 三值枚举替代原来的 `persistent` 布尔值

```sql
CREATE TABLE IF NOT EXISTS entities (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL DEFAULT '',
  image_url TEXT DEFAULT '',
  color TEXT DEFAULT '#888888',
  size REAL DEFAULT 1,
  notes TEXT DEFAULT '',
  rule_data TEXT DEFAULT '{}',
  permissions TEXT DEFAULT '{"default":"none","seats":{}}',
  lifecycle TEXT DEFAULT 'ephemeral' CHECK(lifecycle IN ('ephemeral','reusable','persistent')),
  blueprint_id TEXT
);
```

**scene_entities 表**：包含 `visible` 字段

```sql
CREATE TABLE IF NOT EXISTS scene_entities (
  scene_id TEXT NOT NULL REFERENCES scenes(id) ON DELETE CASCADE,
  entity_id TEXT NOT NULL REFERENCES entities(id) ON DELETE CASCADE,
  visible INTEGER DEFAULT 1,  -- 1=在场, 0=离场
  PRIMARY KEY (scene_id, entity_id)
);
```

**索引**：

```sql
CREATE INDEX IF NOT EXISTS idx_scene_entities_scene ON scene_entities(scene_id);
CREATE INDEX IF NOT EXISTS idx_entities_lifecycle ON entities(lifecycle);
```

### 三种生命周期

|                         | ephemeral（一次性）  | reusable（重要 NPC）   | persistent（PC / 随队）               |
| ----------------------- | -------------------- | ---------------------- | ------------------------------------- |
| 典型角色                | 哥布林、路人甲       | 商人马库斯、反派 BOSS  | 战士凯恩（PC）、雇佣兵                |
| 创建来源                | 蓝图库一键生成       | 角色库手动创建         | 玩家认领座位 / GM 指定                |
| 角色库（底部 Dock）显示 | 否                   | 是                     | 是（玩家 Tab）                        |
| 创建新场景时            | 不自动加入           | 不自动加入             | **自动加入**（INSERT scene_entities） |
| 从场景移除时            | **删除 entity**      | 仅删 scene_entities 行 | 仅删 scene_entities 行                |
| 跨场景复用              | 不支持（仅限单场景） | 手动加入               | 自动                                  |

**约束：ephemeral entity 仅限单场景。** 将 ephemeral entity 加入第二个场景的 API 请求应返回 400 错误。如需跨场景使用，GM 应先"保存为角色"将其提升为 reusable。

### lifecycle 与 visible 的职责分离

这两个字段各管各的，没有重叠：

- **lifecycle**：决定实体的生命周期行为（创建新场景时是否自动加入、移除时是否删除、角色库是否显示）
- **visible**：决定实体在某个场景中是否对玩家可见（在场 vs 离场）

`lifecycle='persistent'` 的含义是"自动加入新场景"，不是"必定在所有场景" —— GM 仍可手动将 persistent entity 从特定场景移除。因此 scene_entities 始终是场景成员关系的唯一真相源。

### 同步删除（替代 GC）

不再需要后台 GC 系统。所有清理在路由中同步完成，用 SQLite 事务保证原子性：

**从场景移除 NPC：**

```
DELETE /api/rooms/:roomId/scenes/:sceneId/entities/:entityId

BEGIN TRANSACTION;
  1. 删除 scene_entities 行
  2. IF entity.lifecycle == 'ephemeral':
       DELETE entity（CASCADE 自动清理其他 scene_entities 行）
       广播 entity:deleted 事件
  3. 广播 scene:entity:unlinked 事件
COMMIT;
```

ephemeral entity 仅限单场景，移除即删除，不需要检查其他场景。

**删除场景：**

```
DELETE /api/rooms/:roomId/scenes/:sceneId

BEGIN TRANSACTION;
  1. 找出该场景中 lifecycle='ephemeral' 的实体 ID
  2. 删除这些 ephemeral entity
  3. 删除场景（CASCADE 自动清理 scene_entities + encounters）
  4. 广播 entity:deleted（每个被删的 ephemeral entity）
  5. 广播 scene:deleted 事件
COMMIT;
```

**直接删除实体：**

```
DELETE /api/rooms/:roomId/entities/:entityId

BEGIN TRANSACTION;
  1. 删除 entity（CASCADE 自动清理 scene_entities）
  2. 扫描 combat_state.tokens 和 encounters.tokens JSON，
     将 entityId 匹配的 token 的 entityId 置为 null（退化为匿名 token）
  3. 广播 entity:deleted 事件
COMMIT;
```

### "保存为蓝图"与"保存为角色"

这两个操作对应不同的概念，类似 Foundry VTT 的 unlinked vs linked：

**保存为蓝图**（unlinked）—— 创建模板，后续生成独立副本：

```
POST /api/rooms/:roomId/assets  (type='blueprint')
  body: {
    name: entity.name,
    url: entity.imageUrl,
    type: 'blueprint',
    extra: {
      blueprint: {
        defaultSize: entity.size,
        defaultColor: entity.color,
        defaultRuleData: entity.ruleData
      }
    }
  }
```

**保存为角色**（linked）—— 将一次性 NPC 提升为可复用角色：

```
PATCH /api/rooms/:roomId/entities/:entityId
  body: { lifecycle: 'reusable' }
```

实体仍在当前场景中，同时出现在底部角色库。后续加入其他场景时，是同一个实体（修改名字所有场景同步变化）。

## API 变更

### 新增路由

| 路由                                                          | 用途                                                         |
| ------------------------------------------------------------- | ------------------------------------------------------------ |
| `PATCH /api/rooms/:roomId/scenes/:sceneId/entities/:entityId` | 更新 scene_entities 行（切换 visible）                       |
| `POST /api/rooms/:roomId/scenes/:sceneId/spawn`               | 从蓝图一键生成：创建 ephemeral entity + 加入场景（原子操作） |

**PATCH scene entity 路由：**

```
PATCH /api/rooms/:roomId/scenes/:sceneId/entities/:entityId
  body: { visible: boolean }
  response: { sceneId, entityId, visible }
  广播: scene:entity:updated { sceneId, entityId, visible }
```

**Spawn 路由：**

```
POST /api/rooms/:roomId/scenes/:sceneId/spawn
  body: { blueprintId: string }
  response: { entity: Entity, sceneEntity: { sceneId, entityId, visible: true } }

BEGIN TRANSACTION;
  1. 从 blueprint asset 读取模板数据
  2. 创建 ephemeral entity（自动命名）
  3. INSERT scene_entities (visible=1)
  4. 广播 entity:created + scene:entity:linked
COMMIT;
```

将创建 entity 和加入场景合并为单个事务，避免两次 API 调用之间的部分失败。

### 变更的路由

| 路由                                         | 变更内容                                                     |
| -------------------------------------------- | ------------------------------------------------------------ |
| `DELETE /scenes/:sceneId/entities/:entityId` | 新增 lifecycle 感知：ephemeral 时同时删除 entity             |
| `POST /scenes/:sceneId/entities/:entityId`   | 新增约束：拒绝 ephemeral entity 加入第二个场景               |
| `GET /scenes/:sceneId/entities`              | 响应格式变更：返回 `{ entityId, visible }[]` 而非 `string[]` |
| `POST /scenes`                               | 创建场景时自动为 persistent entity 插入 scene_entities 行    |

### 新增 Socket.io 事件

| 事件                   | 载荷                             | 触发时机      |
| ---------------------- | -------------------------------- | ------------- |
| `scene:entity:updated` | `{ sceneId, entityId, visible }` | 切换在场/离场 |

### worldStore 变更

- `sceneEntityMap` 类型从 `Record<string, string[]>` 变更为 `Record<string, { entityId: string, visible: boolean }[]>`
- 新增 Socket listener：`scene:entity:updated`
- 新增 action：`toggleEntityVisibility(sceneId, entityId, visible)`
- 新增 action：`spawnFromBlueprint(sceneId, blueprintId)` —— 调用 spawn 路由
- 修改 action：`removeEntityFromScene` —— 返回时检查是否同时收到 `entity:deleted`

## 删除策略

统一原则：**延迟删除 + Toast 撤销。** 删除操作先从 UI 隐藏，延迟 5 秒后真正执行服务端删除，期间用户可通过 Toast 中的「撤销」按钮取消操作。

| 操作                                | 确认方式         | 理由                               |
| ----------------------------------- | ---------------- | ---------------------------------- |
| 移除一次性 NPC                      | Toast 撤销（5s） | 成本低，但提供后悔窗口             |
| 移除 reusable/persistent NPC 出场景 | 不确认           | 实体保留在角色库                   |
| 永久删除 reusable/persistent NPC    | Toast 撤销（5s） | 延迟删除保证可恢复                 |
| 删除场景                            | 确认对话框       | 级联删除 ephemeral NPC，影响范围大 |
| 删除蓝图                            | Toast 撤销（5s） | 延迟删除保证可恢复                 |

## 不在本次范围内

- **战术模式 token 数据模型** —— token 自带 label/imageUrl/color 与 entity 数据冗余的问题，留待战术模式设计解决。删除 entity 时，服务端会将 combat_state/encounters 中引用该 entity 的 token 的 entityId 置为 null（退化为匿名 token），但 token 数据模型本身的重构不在本次范围内
- **权限系统** —— 离场 NPC 对玩家不可见的权限控制，依赖未来的安全系统
- **角色导入/导出** —— 跨房间的角色迁移功能
