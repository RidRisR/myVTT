# myVTT 产品愿景与设计理念

> 本文档记录项目的核心设计哲学和产品定位，作为所有 UI/UX 决策的根基参考。

---

## 一、核心设计哲学

### 轻备团优先（Zero-Prep-First）

系统的最小启动成本 = **一张氛围图**。GM 上传一张背景图，分享链接，所有人就可以开始跑团。

- 网格、Token、战斗地图、角色数据——**全部可选**
- 所有功能按需叠加，不强制任何准备流程
- 同时支持重备团：愿意预先安排一切的 GM 也能高效操作

### 氛围感至上

进入房间的第一印象 = **全屏铺满的氛围图**。这是 myVTT 的"wow moment"。

- 氛围图永远在最底层，是贯穿始终的视觉基底
- 所有 UI 元素都是叠加在氛围图之上的浮层
- 任何功能的加入都不应破坏氛围感

### 直觉式交互（One-Click / Direct Manipulation）

所有高频操作追求**一键完成、所见即所得**：

- 血量条直接拖动修改
- 鼠标悬浮即可看到详细信息
- 右键空白处立即创建 Token（先创建，后完善细节）
- 一键投骰子
- 一键修改状态

### 骨架 + 规则层（Skeleton + Rules Plugin）

系统提供通用骨架（场景、实体、Token、权限、聊天），规则层作为独立模块插入：

- 骨架不绑定任何 TRPG 规则系统
- 不同规则系统（DnD 5e、PF2e、CoC 等）通过规则层适配
- 骨架负责：展示、交互、同步；规则层负责：数值计算、约束验证

## 二、两种模式：叙事 + 战术

### 叙事模式（默认）

**视觉**：全屏氛围图 + 浮动 UI 面板

**玩家可做的事**：

- 看氛围图，感受场景
- 聊天、角色扮演
- 投骰子
- 修改角色状态（血量、属性、条件等）
- 查看/回顾展示材料（Handout）

**GM 可做的事**：

- 切换场景（氛围图变化）
- 推送展示材料（Showcase）
- 管理角色/NPC
- 准备下一场战斗（隐性操作，玩家不可见）

### 战术模式（按需切入）

**视觉**：氛围图保持在背景 + 战术地图**浮现在氛围图之上**（不切屏！）

> ⚠️ 关键设计：当前实现是切屏（氛围图消失、战斗地图取代），**期望改为**战术地图以某种不违和的方式叠加在氛围图上，保持氛围感。

**切入方式**：GM 一键打开战术地图 → 地图浮现在屏幕上 → 大家开始操作

**切出方式**：GM 一键收起战术地图 → 回到叙事模式 → Token 暂时隐藏但数据保留

**Token 行为**：

- Token 留在战术地图上，收起时隐藏，再次打开时恢复
- PC Token 跟随整个流程始终存在
- NPC Token 生命周期跟随场景设计（见之前文档 01-场景与NPC组织.md）

**两种备团模式**：

- **重备团**：GM 提前在战术地图上安排好所有 Token、视野等，收起地图。战斗开始时一键放出
- **轻备团**：GM 不做任何预设。进入战斗时打开空白/透明/带网格的战术地图，临时右键创建 Token，直接开打

## 三、展示材料系统（Showcase / Handout）

### 核心交互

1. **强制推送**：GM 推送一张图片/文字 → 强制出现在所有玩家面前（强制关注）
2. **可暂时关闭**：玩家可以选择"暂时不看"，关闭推送
3. **历史可回溯**：之前展示过的材料以某种方式收纳，玩家可以随时重新打开查看

### 进阶设想（优先级待定）

- 背景音乐/音效
- 临时动效/画面弹出（用于氛围塑造）
- 材料批注/白板功能（存疑：可能影响氛围感）

## 四、竞品定位

### Owlbear Rodeo — 过于简单

- 优点：轻量、上手快
- 缺点：缺乏数据骨架（没有数值系统、没有骰子系统），本质是共享白板

### Foundry VTT — 过于沉重

- 缺点：
  - 操作不直观（配一张卡要很长时间）
  - 信息过载（大量文件直接暴露）
  - 界面不好看
  - 开发者自定义困难（接口过时、开发难度大）

### Alchemy RPG — 当前对标方向 ⭐

- 优点：
  - **视觉风格高级感**：一进入就感觉精致、沉浸
  - **叙事优先**：场景由氛围画面主导，不是网格地图
  - **战术地图叠加而非替换**：Tactical Mode 坐在 Story Mode 之上，隐藏地图时显示底层场景画面
  - **操作直观**：投骰一键完成、切场景快速、音频一键切换
  - **Zen Mode**：玩家可以让所有 UI 消失，只留氛围画面（纯沉浸）
  - **设计哲学**："Remove the speedbumps in the storytelling process"（消除叙事过程中的减速带）
- 缺点：完全不开放自定义能力，无法作为开发者自行扩展规则系统

### myVTT 定位

**Alchemy RPG 的打磨水准 + 开发者友好的自定义能力 + 规则无关的通用骨架**

```
Owlbear (太简单) ←——— myVTT (甜蜜点) ———→ Foundry (太复杂)
                         ↑
                    Alchemy RPG 的体验水准
                    + 开发者自定义能力
```

**从 Alchemy 学习的核心 UX 模式**：

1. 场景 = 氛围画面 + 粒子特效 + 音乐/音效（不是地图 + 网格）
2. 战术模式叠加在叙事模式上（不切屏），隐藏战术地图即回到叙事
3. 最高频操作一键完成（投骰、切场景、开/关战术地图）
4. UI 干净可导航，不信息过载

## 五、关键 UX 原则（从上述理念推导）

1. **默认最简**：任何新功能的初始状态应该是"关闭/隐藏"的，用户主动选择开启
2. **渐进披露**：信息按需展示——hover 看摘要，click 看详情，展开看完整
3. **不切屏**：模式切换（叙事↔战术）应该是叠加/浮现，不是页面替换
4. **氛围图永远可见**：它是视觉锚点，任何操作都不应完全遮盖它
5. **右键即创建**：空白处右键 = 就地创建新 Token/实体，零门槛
6. **一键操作**：投骰、改状态、切战斗——核心操作不超过一次点击
7. **GM 的准备工作对玩家不可见**：GM 在幕后准备的东西，只在 GM 选择的时刻才展示给玩家
