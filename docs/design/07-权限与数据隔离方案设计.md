# 权限与数据隔离方案设计

> **状态**：⚠️ 需重新设计（Yjs 时代方案）| 2026-03-26
>
> ⚠️ **注意**：此文档为 Yjs 时代数据隔离方案。当前架构已迁移至 per-room SQLite + Express 路由层权限校验。上线前需根据现有架构重新设计权限隔离方案，此文档仅供概念参考。
> **相关架构**：当前权限入口见 `server/middleware.ts`（withRole）和 `server/visibility.ts`（消息过滤）。

## 问题

当前所有数据在单一 Y.Doc 中，玩家打开 F12 即可看到 GM 隐藏的 NPC 属性、怪物真实 HP、未展示的场景等机密数据。需要一种机制保护 GM 的机密内容，同时不破坏 Yjs 实时协作的架构。

## 威胁分析

### 两类需要保护的数据

| 类型         | 例子                                             | 泄露影响                        | 保护需求                     |
| ------------ | ------------------------------------------------ | ------------------------------- | ---------------------------- |
| **备团资料** | 预设的战斗遭遇、NPC 模板、剧情笔记、未展示的地图 | **高**——剧透整个剧情走向        | 物理隔离，玩家完全不可见     |
| **临场秘密** | 怪物真实 HP、隐藏属性、GM 私密备注               | **低~中**——知道怪物 HP 影响有限 | 合理遮挡即可，不需要绝对安全 |

关键洞察：**大多数"秘密"其实是备团资料，不是临场数据**。一旦 GM 把怪物放上场，其存在已不是秘密——真正危险的是剧透 GM 还没展示的内容。

### 攻击者模型

目标防御的是"好奇的玩家按 F12"，不是专业攻击者。具体来说：

- **要防**：devtools 直接看到明文数据、Yjs 内存中的原始值
- **不防**：反编译加密逻辑提取密钥、中间人攻击（属于传输层问题，见 doc 53）

## 方案：备团区 + 加密字段

### 架构总览

```
┌─────────────────────────────────────────────┐
│                  服务端                       │
│                                              │
│  ┌──────────────┐    ┌───────────────────┐   │
│  │  备团区       │    │   Y.Doc (房间)     │   │
│  │  (JSON 文件)  │    │                   │   │
│  │              │    │  scenes            │   │
│  │  遭遇预设    │───→│  entities          │   │
│  │  NPC 模板    │部署│  chat_log          │   │
│  │  剧情笔记    │    │  ...               │   │
│  │              │    │                   │   │
│  └──────────────┘    └───────────────────┘   │
│   GM 专属，REST API      全员共享，Yjs 同步    │
│   玩家完全不可达          玩家可读可写           │
└─────────────────────────────────────────────┘
```

两层保护，各司其职：

1. **备团区（Preparation Area）**：服务端 JSON 存储，仅 GM 可通过 REST API 访问。玩家在任何层面都看不到这些数据——不在 Y.Doc 中，不经 WebSocket 传输。
2. **加密字段（Encrypted Fields）**：Y.Doc 中实体的 `secretData` 字段使用 AES-GCM 加密。F12 只能看到密文。加密密钥仅 GM 角色可通过 API 获取。

### 为什么不用双 Y.Doc？

最初考虑过将数据拆分为 Public Y.Doc + GM Secret Y.Doc，但存在实际问题：

| 问题           | 说明                                                                                                                     |
| -------------- | ------------------------------------------------------------------------------------------------------------------------ |
| **跨文档操作** | 玩家对怪物造伤需要修改 secret doc 中的 HP，但玩家连不上 secret doc。必须引入中转机制（客户端搬运或服务端 API），复杂度高 |
| **数据一致性** | 两个 Y.Doc 间没有事务保证，entity 的公开部分和秘密部分可能短暂不一致                                                     |
| **架构侵入性** | 所有读写 entity 的代码都需要适配双文档，改动面极大                                                                       |

混合方案（备团区 + 加密字段）避免了以上问题：备团资料根本不在 Y.Doc 中；临场秘密仍在同一个 Y.Doc 中，只是加密存储。

## 备团区设计

### 存储

遵循 [41-数据层重构方案设计](./41-数据层重构方案设计.md) 的按房间物理隔离原则，备团区位于房间目录内：

```
data/rooms/{roomId}/
  ├── db/
  │   ├── yjs/       ← Y.Doc 持久化（doc 41）
  │   └── assets/    ← 素材元数据 KV（doc 41）
  ├── uploads/        ← 素材文件（doc 41）
  └── prep/           ← 备团区（本文档）
      ├── encounters/     # 战斗遭遇预设
      │   ├── {id}.json
      │   └── ...
      ├── npcs/           # NPC 模板
      │   ├── {id}.json
      │   └── ...
      └── notes/          # 剧情笔记
          ├── {id}.json
          └── ...
```

每个条目是独立的 JSON 文件，结构简单，便于调试和备份。房间删除时 `rm -rf data/rooms/{roomId}/` 一并清理，与 doc 41 的房间操作模式一致。

### API

```
GET    /api/rooms/:roomId/prep/:category           列出该分类下所有条目（GM only）
POST   /api/rooms/:roomId/prep/:category           创建条目（GM only）
GET    /api/rooms/:roomId/prep/:category/:id        读取条目（GM only）
PUT    /api/rooms/:roomId/prep/:category/:id        更新条目（GM only）
DELETE /api/rooms/:roomId/prep/:category/:id        删除条目（GM only）
POST   /api/rooms/:roomId/prep/:category/:id/deploy 部署到 Y.Doc（GM only）
```

所有端点需要 JWT cookie + GM 角色验证。玩家调用返回 403。

### 部署流程

"部署"是指 GM 把备团区的内容投入使用：

```
GM 点击"部署遭遇"
  ↓
POST /api/rooms/:roomId/prep/encounters/:id/deploy
  ↓
服务端：
  1. 读取 JSON 预设
  2. 获取房间 Y.Doc: getYDoc(roomId)
  3. doc.transact(() => {
       // 将预设中的 entity 写入 entities map
       // 将 entityId 添加到当前场景的 entityIds
       // 如有 secretData，加密后写入
     }, 'server')
  ↓
Yjs 同步 → 所有客户端看到新加入的怪物/NPC
```

部署后，备团区的原始数据不受影响（可反复部署）。部署是单向的：备团区 → Y.Doc，不存在回写。

**与 doc 41 战斗预设系统的关系**：备团区的 `prep/encounters/` 存储 GM 尚未展示的遭遇预设（玩家完全不可见）。部署后写入 doc 41 定义的 `scene.encounters` Y.Map（全员可见），再由 GM 激活到 `combat` 运行时。两者是同一个遭遇在不同生命阶段的存储位置：

```
prep/encounters/{id}.json  ──部署──→  scene.encounters Y.Map  ──激活──→  combat Y.Map
   （GM 私有）                         （全员可见的预设）                  （实时战斗）
```

## 加密字段设计

### 场景

GM 想让怪物出现在地图上，但隐藏其真实 HP、隐藏备注等。这些数据已经在 Y.Doc 中（随 entity 一起同步），需要防止 F12 直接读取。

### 加密方案：AES-GCM

```
Entity in Y.Doc:
{
  id: "e-abc",
  name: "神秘骑士",        // 明文，所有人可见
  imageUrl: "/assets/...", // 明文
  color: "#ff0000",        // 明文
  ruleData: { ... },       // 明文（公开属性）
  secretData: "base64(iv + ciphertext + tag)"  // 密文，仅 GM 可解
}
```

- **算法**：AES-256-GCM（WebCrypto API，浏览器硬件加速）
- **密钥**：每个房间一个，服务端生成并持久化到 `{PERSISTENCE_DIR}/rooms/{roomId}/secret.key`
- **IV**：每次加密随机生成 12 字节，拼接在密文前
- **明文格式**：JSON 字符串，包含 GM 想隐藏的任意字段

### 密钥分发

```
GM 客户端启动
  ↓
GET /api/rooms/:roomId/key
  ↓
服务端验证 JWT cookie + 确认 GM 角色
  ↓
返回 { key: "base64(raw-key)" }
  ↓
GM 客户端用 crypto.subtle.importKey() 导入
  ↓
解密 Y.Doc 中所有 entity 的 secretData
```

玩家调用 `/key` 端点返回 403。即使玩家在 F12 中看到了 `secretData` 字段，没有密钥只能看到密文。

### CRDT 与加密的兼容性

加密后 `secretData` 是一个不透明的 string，失去了 Y.Map 的字段级 CRDT 合并能力。并发编辑 secretData 时是 last-write-wins。

**为什么可以接受？**

- secretData 仅 GM 可编辑
- 多 GM 同时编辑同一个 entity 的秘密属性极为罕见
- 即使发生冲突，也只是 secretData 部分丢失上一次编辑，不影响实体的公开属性

### 加密/解密时机

- **写入**：GM 编辑 entity 的秘密属性 → 客户端加密 → 写入 Y.Doc `secretData` 字段
- **读取**：Y.Doc 同步到 GM 客户端 → 检测 `secretData` 存在 → 用本地缓存的密钥解密 → 渲染
- **玩家视角**：Y.Doc 同步到玩家客户端 → `secretData` 存在但无密钥 → 忽略，只渲染明文字段

## 通讯架构总览

系统只有两种客户端-服务端通讯方式：

| 通道          | 协议                     | 用途                                               |
| ------------- | ------------------------ | -------------------------------------------------- |
| **WebSocket** | Yjs 同步协议 + awareness | 实时协作数据（Y.Doc 内容同步、在线状态）           |
| **HTTP REST** | Express API              | 身份认证、文件上传、骰子投掷、备团区管理、密钥分发 |

所有新功能都通过这两个通道实现，不引入额外协议。

## 攻击场景分析

| 攻击方式                  | 过程                                     | 结果                                               |
| ------------------------- | ---------------------------------------- | -------------------------------------------------- |
| **F12 查看 Y.Doc**        | 玩家在 devtools 中遍历 Y.Doc 数据        | 明文字段可见，secretData 只有密文                  |
| **访问备团区 API**        | 玩家调用 GET /api/rooms/:roomId/prep/... | JWT 角色检查 → 403 Forbidden                       |
| **请求加密密钥**          | 玩家调用 GET /api/rooms/:roomId/key      | JWT 角色检查 → 403 Forbidden                       |
| **暴力破解 AES**          | 尝试穷举密钥                             | AES-256 密钥空间 2^256，不可行                     |
| **分析密文模式**          | 观察 secretData 变化规律                 | GCM 模式每次 IV 不同，同一明文产生不同密文         |
| **客户端直写 secretData** | 注入密文覆盖 GM 的加密数据               | transaction.origin 审计检测到非 GM 写入 → 篡改警告 |
| **重放旧密文**            | 把之前的 secretData 值写回               | 密文本身合法，但 origin 审计检测到非 GM 写入       |

### 存在感泄露

玩家可以知道某个 entity 有 secretData 字段（字段名可见），但无法知道内容。这意味着玩家可以推断"这个 NPC 有隐藏属性"，但无法知道具体是什么。

**可接受性**：在 TRPG 中，GM 说"这个骑士看起来不简单"本身就是剧情的一部分。存在感泄露的信息量极低，不影响游戏体验。

## 与身份系统的衔接

权限检查复用 doc 53 设计的 JWT cookie + 服务端角色查询：

```
客户端请求（携带 JWT cookie）
  ↓
服务端中间件：
  1. 解析 JWT → 获取 userId
  2. 查询 seats（Y.Doc 或内存缓存）→ 获取该用户在该房间的角色
  3. 角色 === 'GM' → 允许访问备团区 / 密钥端点
  4. 角色 !== 'GM' → 403
```

WebSocket 连接上的 `conn.userId` + `conn.role` 也用于 transaction.origin 审计中判断写入权限。

## 实现优先级

1. **备团区**（核心价值最高）：保护 GM 的剧情资料不被剧透，独立于其他系统，可先行实现
2. **加密字段**（锦上添花）：F12 防窥，提升体验但非刚需，可后续实现
3. **secretData 审计**（依赖身份系统）：检测非 GM 写入 secretData，需要 transaction.origin 信任链就位后实现

## 未来方向

- **备团区 UI**：GM 面板中的备团管理界面（遭遇编辑器、NPC 模板库）
- **部署预览**：部署前预览 entity 在地图上的位置
- **批量部署**：一次部署整个遭遇（多个 entity + 位置预设）
- **密钥轮换**：GM 可以重新生成房间密钥（重新加密所有 secretData）

## 不变的部分

- Y.Doc 仍然是单文档架构，不拆分
- entity 的公开属性（name, imageUrl, color, ruleData 的公开部分）仍然明文同步
- 玩家仍然可以读写 Y.Doc（transaction.origin 审计检测作弊，不阻止）
- awareness 仍用于在线状态
- 聊天消息的 senderId 仍然是 seatId
