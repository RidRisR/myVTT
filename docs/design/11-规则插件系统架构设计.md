# 规则插件系统架构设计

> **✅ 实现状态：Phase 1 已完成**
>
> RulePlugin 7 层接口、registry 注册机制、SDK 导出、generic + daggerheart 两个插件均已实现（编译时打包）。
> **未落地部分**：Phase 2（独立编译+重启加载）、Phase 3（热加载）仍在路线图上。

## 一、背景与动机

数据层重构完成后，系统已具备：

- 基础实体系统（Entity + `ruleData: unknown`）
- 泛型 UI（资源条、属性、状态标签）
- 临时适配器（`entityAdapters.ts`）作为过渡层
- `src/rules/types.ts` 中的 `RuleSystem` 接口雏形（从未被接线）

系统缺失的是：将游戏规则特定逻辑与基座正式解耦的机制。

**目标**：以 DaggerHeart 为第一个目标规则系统，设计一套清晰的插件架构，探索基座与插件的边界，定义稳定契约，并通过实现第一个插件进行验证。

## 二、设计原则

### 原则 1：房间级激活

一个 Room 对应一套规则系统。GM 建房时选择游戏系统；玩家加入后自动获得对应插件的 UI 和逻辑。无需玩家手动安装任何东西。

### 原则 2：分阶段演化

```
Phase 1（当前）    Phase 2（未来）       Phase 3（终态）
───────────────────────────────────────────────────────
编译时静态打包      独立编译 + 重启加载    运行时热插拔
所有插件同一 bundle 放入 plugins/ 目录     社区市场 / 版本管理
```

Phase 1 → Phase 2 的关键：替换 registry 加载方式，接口不变。

### 原则 3：物理目录隔离

插件与基座文件完全分离，不得交叉：

```
src/       ← 基座
plugins/   ← 所有插件（与 src/ 完全平级）
```

插件只能从 `@myvtt/sdk`（映射至 `src/rules/sdk.ts`）导入，不得直接访问基座内部文件。

### 原则 4：插件只描述意图，基座执行

插件不调用 Store，不直接写状态。通过 props 接收数据，通过回调返回变更。

### 原则 5：三种 UI 操作

插件对 UI 的所有改变，归结为三种正交操作：

```
替换  →  向指定容器（slot）提供自己的 React 组件
隐藏  →  声明需要隐藏哪些基座 UI 元素
叠加  →  在现有 UI 上方渲染自己的 React 组件层
```

三者组合可实现任意视觉效果，基座结构代码无需修改。

## 三、目录结构

```
src/                             ← 基座
  rules/
    sdk.ts                       ← 唯一对外桥接点（插件的门）
    registry.ts                  ← id → RulePlugin 映射（唯一知道插件的基座文件）
    types.ts                     ← 接口类型定义
    useRulePlugin.ts             ← 组件便捷 hook

plugins/                         ← 插件（与 src/ 完全平级隔离）
  generic/                       ← 通用 fallback 插件
    index.ts
    GenericEntityCard.tsx        ← 包裹现有 CharacterEditPanel
  daggerheart/
    index.ts                     ← 导出 daggerheartPlugin: RulePlugin
    types.ts                     ← DH 专属类型（DHRuleData, DomainCard 等）
    adapters.ts                  ← 实现 adapters 层
    diceSystem.ts                ← Hope/Fear 骰子评估逻辑
    templates.ts                 ← createDefaultEntityData()
    ui/
      DaggerHeartCard.tsx        ← 主 EntityCard 组件（入口小卡）
      FullCharacterSheet.tsx     ← 完整角色卡（fullscreen-overlay）
      DomainCardBrowser.tsx      ← 领域卡牌浏览器（floating panel）
      TraitSection.tsx           ← 六维核心属性
      ResourceSection.tsx        ← HP / 压力 / 希望点
      RollActions.tsx            ← 快捷投骰按钮
      DHTeamPanel.tsx            ← 替换 Team Matrix（资源追踪 + 团队状态）
      DHLibraryTab.tsx           ← Dock 标签页：DH 预制内容浏览 + 导入
```

**关键约束：**

- `src/` 中只有 `registry.ts` 知道 `plugins/` 的存在
- 插件只从 `@myvtt/sdk` 导入，不得导入 `src/` 内部其他文件
- 插件之间不互相导入

## 四、RulePlugin 接口

```typescript
// src/rules/types.ts

interface RulePlugin {
  id: string
  name: string
  sdkVersion: '1'

  // ── 必须实现 ────────────────────────────────────────

  // 层 1：适配器 — 基座通用 UI 读取数据用
  adapters: {
    getMainResource(entity: Entity): ResourceView | null
    getPortraitResources(entity: Entity): ResourceView[]
    getStatuses(entity: Entity): StatusView[]
    getFormulaTokens(entity: Entity): Record<string, number>
  }

  // 层 2：角色卡 UI — 替换基座默认角色卡
  characterUI: {
    EntityCard: React.ComponentType<EntityCardProps>
  }

  // ── 可选实现（undefined = 使用通用行为）─────────────

  // 层 3：骰子系统
  diceSystem?: {
    getRollActions(entity: Entity): RollAction[]
    evaluateRoll(terms: DiceTermResult[], total: number, ctx: RollContext): JudgmentResult | null
    getDieStyles(terms: DiceTermResult[]): DieStyle[]
    getJudgmentDisplay(result: JudgmentResult): JudgmentDisplay
    getModifierOptions(): ModifierOption[]
  }

  // 层 4：数据模板 — 新实体初始化与预制内容
  dataTemplates?: {
    // 新实体的空白起点（所有属性归零）
    createDefaultEntityData(): unknown
    // 预制内容目录（随插件分发，不在数据库里）
    // GM 从此处"导入"后才进入数据库，等同于 Blueprint
    getPresetTemplates?(): PresetTemplate[]
  }

  // 层 5：UI 表面注册 — 插件自定义面板与标签页
  surfaces?: {
    // 插件注册的自定义面板（由插件内部触发打开）
    panels?: PluginPanelDef[]
    // 底部 Dock 新增标签页
    dockTabs?: DockTabDef[]
    // GM 工具面板新增标签页
    gmTabs?: GMTabDef[]
    // Team Matrix 面板替换（空壳，内容由插件决定）
    teamPanel?: React.ComponentType<TeamPanelProps>
  }

  // 层 6：UI 隐藏声明 — 插件激活时隐藏指定基座元素
  hideElements?: HideableElement[]

  // 层 7：规则结算（预留，暂不实现，见「延后议题」）
  // ruleResolution?: RuleResolutionModule
}

// ── 相关类型 ────────────────────────────────────────────

interface EntityCardProps {
  entity: Entity
  onUpdate: (patch: Partial<Entity>) => void
  readonly?: boolean
}

interface PluginPanelDef {
  id: string
  component: React.ComponentType<PluginPanelProps>
  defaultSize?: { width: number; height: number }
  placement: 'floating' | 'fullscreen-overlay'
}

interface PluginPanelProps {
  entity?: Entity
  onClose: () => void
  onUpdateEntity: (id: string, patch: Partial<Entity>) => void
  // 用于从插件面板内导入预制模板（创建新 Entity 进入数据库）
  onCreateEntity: (data: Partial<Entity>) => void
}

interface TeamPanelProps {
  trackers: TeamTracker[]
  onUpdate: (id: string, patch: Partial<TeamTracker>) => void
  onCreate: (data: Partial<TeamTracker>) => void
  onDelete: (id: string) => void
}

// 预制内容条目（随插件代码分发，不在数据库里）
// 命名空间 ID 格式：<pluginId>:<contentName>，如 'dh:corrupt-elf-archer'
interface PresetTemplate {
  id: string // 命名空间 ID，如 'dh:corrupt-elf-archer'
  name: string // '腐化精灵弓手'
  category: string // 'adversary' | 'pc-archetype' | ...
  data: Partial<Entity> // 预填的实体数据，结构与 Blueprint 相同
}

interface DockTabDef {
  id: string
  label: string
  component: React.ComponentType
}

interface GMTabDef {
  id: string
  label: string
  component: React.ComponentType
}

type HideableElement = 'dock' | 'portrait-bar' | 'chat-panel' | 'gm-panel' | 'scene-controls'
```

## 六、插件 UI 三种操作的工作方式

### 操作 1：替换（向 slot 提供组件）

基座在渲染角色卡时：

```typescript
const plugin = useRulePlugin()
const Card = plugin.characterUI.EntityCard
return <Card entity={entity} onUpdate={handleUpdate} />
```

### 操作 2：隐藏（声明式）

基座读取当前激活插件的 `hideElements`，有条件地渲染：

```typescript
const plugin = useRulePlugin()
const hidden = new Set(plugin.hideElements ?? [])

{!hidden.has('dock') && <BottomDock />}
{!hidden.has('portrait-bar') && <PortraitBar />}
```

### 操作 3：叠加（Surfaces）

基座维护一个 Panel Registry，插件通过 `usePluginPanels().openPanel()` 触发：

```typescript
// 插件内部（DaggerHeartCard.tsx）
const { openPanel } = usePluginPanels()
<button onClick={() => openPanel('full-sheet', { entity })}>
  完整角色卡
</button>

// 基座渲染 Panel 容器（高 z-index，React Portal）
{activePanels.map(p => (
  <PluginPanelContainer key={p.id} def={p.def} props={p.props} />
))}
```

## 七、房间激活机制

### 数据库层

```sql
ALTER TABLE room_state
  ADD COLUMN rule_system_id TEXT NOT NULL DEFAULT 'generic';

ALTER TABLE room_state
  ADD COLUMN plugin_config JSON DEFAULT '{}';
  -- GM 在「房间设置」中填写的插件级配置（如难度偏好、可选规则开关等）
  -- 非动态游戏状态（动态状态归 entities / teamTrackers）
```

### Store 层

```typescript
// worldStore 新增
ruleSystemId: string // 随 room:state:updated Socket.io 事件同步

// src/rules/useRulePlugin.ts
export function useRulePlugin(): RulePlugin {
  const id = useWorldStore((s) => s.ruleSystemId)
  return getRulePlugin(id) // 永远不为 null，fallback 到 generic
}
```

### API 层

- `GET /api/rooms/:roomId` 返回值含 `ruleSystemId`
- `PATCH /api/rooms/:roomId/state` 允许 GM 更新 `ruleSystemId`
- `room:state:updated` Socket.io 事件携带 `ruleSystemId`

## 八、基座接线点

| 位置                  | 现状                                  | 修改为                                                    |
| --------------------- | ------------------------------------- | --------------------------------------------------------- |
| 角色卡渲染            | 直接渲染 `CharacterEditPanel`         | `plugin.characterUI.EntityCard`                           |
| `KonvaToken` 血条数据 | `entityAdapters.getEntityResources()` | `plugin.adapters.getMainResource()`                       |

| 基座元素渲染          | 无条件渲染                            | 检查 `plugin.hideElements`                                |
| Portal 容器           | 不存在                                | 新增 `PluginPanelContainer`（高 z-index）                 |
| Team Matrix 渲染      | 固定通用组件                          | `plugin.surfaces?.teamPanel ?? DefaultTeamPanel`          |

### 骰子系统：Hope/Fear 双骰

同时投希望骰（d12）+ 恐惧骰（d12），加属性调整值，与难度比较：

- 希望骰 > 恐惧骰 → **乘希望而为**（玩家获得 1 Hope）
- 恐惧骰 > 希望骰 → **带着恐惧**（GM 获得 1 Fear）
- 相等 → **命运之刃**（GM 选择）

四种结果组合：成功/失败 × Hope/Fear

视觉：希望骰金色，恐惧骰深红，判定徽章显示结果文本。

## 十三、参照与教训

| 系统              | 模式                                                                                | 对 myVTT 的借鉴                               |
| ----------------- | ----------------------------------------------------------------------------------- | --------------------------------------------- |
| **Foundry VTT**   | System 包完全替换角色卡，Application 系统管理浮动窗口。无 SDK 边界 → 模组兼容噩梦。 | 参照 Application 浮动窗口概念；避免无边界耦合 |
| **Owlbear Rodeo** | `@owlbear-rodeo/sdk` 单一入口，iframe 隔离。                                        | SDK 单一入口理念；避免 iframe（视觉割裂）     |
| **myVTT 目标**    | Foundry 的视觉整合度 + Owlbear 的边界稳定性                                         | —                                             |

## Assumptions

- Phase 1 插件与基座一起编译，无运行时加载开销
- 每个房间在创建时选择一个规则系统，之后不可切换（`room_state.rule_system_id`）
- 插件不直接访问 store 或数据库，只通过 props 和 SDK 工具函数交互
- `sdkVersion: '1'` 为未来兼容矩阵预留，Phase 1 不做版本校验

## Edge Cases

- 未知的 `rule_system_id` 回退到 generic 插件（`getRulePlugin()` 保证不返回 null）
- 插件的 `surfaces.panels` 注册后可能与基座 UI 布局冲突（z-index 需协调）
- `entityAdapters.ts` 过渡层仍有 6 个调用方，迁移期间需保持两套适配器一致
- daggerheart 插件的 `rollCommands` 可能与基座的内置命令名冲突（当前无冲突检测机制）
