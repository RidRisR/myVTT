# 16 - 事件日志与骰子系统架构

> **状态**：🚧 实施中（Dispatcher/渲染器待接入）| 2026-03-26

## 核心模型：统一日志流

系统的核心是一个**统一的、有序的日志流**。所有游戏内行为——聊天、骰子、判定、状态变更——都是日志条目。

```
用户操作 → workflow 本地执行 → 产生日志条目 → 立即 socket.emit('log:entry') → server 持久化+副作用+广播
```

UI 组件按类型订阅日志流，各取所需：

- 聊天面板订阅 `core:text, core:roll, dh:judgment`
- Tracker UI 订阅 `core:tracker-update`
- 战斗日志订阅 `core:roll, dh:judgment, core:tracker-update`
- 统计面板订阅 `core:roll`

**日志是游戏事件的完整记录和 UI 展示的来源，数据表是世界当前状态的权威来源。** 没有"聊天消息"和"游戏事件"的区分——只有不同类型的日志条目。

### game_log 的边界

只有**游戏内行为**走 game_log：

- 聊天消息（`core:text`）、骰子（`core:roll`）、判定（`dh:judgment`）
- 状态变更（`core:tracker-update`, `core:component-update`）
- 其他游戏事件（攻击、伤害、状态效果等）

**管理操作保持现有 REST + Socket.io 广播模式**：

- 场景管理（scene CRUD）
- 实体管理（entity/component CRUD）
- 战术地图操作（tactical state, token CRUD）
- 资产、蓝图、标签、座位、展示材料等

理由：管理操作有自己的验证逻辑和错误处理，不适合用不透明的 payload 日志条目表达。它们也不需要因果链、triggerable 等日志语义。

### 传输层：Socket.io

日志系统**完全运行在 Socket.io 上**，不使用 REST。理由：

- 已有 Socket.io 长连接，无额外连接开销
- Server 已知 socket 绑定的 seat 身份，无需额外认证
- 上行（emit）和下行（广播）走同一通道，架构更简单
- Socket.io ack 回调原生支持请求-响应模式

所有日志条目通过 Socket.io **逐条即时发送**，不做批量缓冲。WebSocket 长连接的开销极低（一个 frame），TTRPG 的操作频率（每回合几条）远不需要批量优化。

```typescript
// 逐条提交（fire-and-forget，workflow 不等 ack）
socket.emit('log:entry', entry, (ack) => {
  // ack = 带 seq 的完整 GameLogEntry（含 server 写回的 snapshot）
})

// RNG 请求（request-response，workflow 需要 await 返回值）
socket.timeout(5000).emit('log:roll-request', {
  // 日志条目元数据（映射到 GameLogEntry 字段）
  origin,           // MessageOrigin → entry.origin
  parentId,         // string? → entry.parentId（因果链）
  chainDepth,       // number → entry.chainDepth
  triggerable,      // boolean → entry.triggerable（通常 true）
  visibility,       // Visibility → entry.visibility
  // RNG + payload 数据
  dice,             // DiceSpec[] — server 用来生成随机数
  formula,          // string — "2d12"
  resolvedFormula,  // string? — 替换变量后的公式
  rollType,         // string? — 触发器匹配（如 'dh:action-check'）
  actionName,       // string? — UI 展示名（如 "敏捷检定"）
}, (err, ack) => {
  // ack = 完整的 core:roll-result GameLogEntry（含 server 分配的 id, seq, executor）
})

// 历史查询
socket.emit('log:history', { beforeSeq, limit }, (ack) => { ... })
```

两种发送模式的区别：

|                   | `log:entry`                       | `log:roll-request`     |
| ----------------- | --------------------------------- | ---------------------- |
| 谁创建条目        | 客户端（UUIDv7）                  | Server（UUIDv7）       |
| workflow 是否等待 | 否（fire-and-forget）             | 是（await ack）        |
| 用途              | judgment、tracker-update、text 等 | 需要 server RNG 的骰子 |

## 骰子三层模型

一次骰子行为产生三层语义，每层都是独立的日志条目：

| 层              | 含义                     | 举例                                 | 写入者                                           |
| --------------- | ------------------------ | ------------------------------------ | ------------------------------------------------ |
| 第1层：原始结果 | 随机数，无任何含义       | `rolls: [8, 5], formula: "2d12"`     | Server（RNG 权威）                               |
| 第2层：解释     | 骰子的含义，但不影响世界 | `"希望胜出"` / `"20 > AC 15，命中"`  | Workflow（确定性纯函数，但包装为 workflow step） |
| 第3层：世界影响 | 对游戏状态的实际变更     | `"Hope tracker +1"` / `"目标 HP -8"` | Workflow（副作用，通过 CRUD 接口）               |

关键设计决策：

- **解释是确定性纯函数**（给同样的输入，永远返回同样的结果），但仍然包装为 workflow step，统一模型无特例
- **解释可能依赖上下文**：Daggerheart 只看骰子值，但 D&D 需要 DC/AC 等上下文。上下文在投骰时快照到日志条目中
- **每层都是独立的日志条目**，而不是一个消息包含所有信息。不同 UI 订阅不同层级
- **日志必须写入**：即使 T2（解释）是可从 T1 推导的，也要写入日志。UI 订阅日志来渲染，不应该需要知道如何计算解释

## Workflow 触发模型

Workflow 有两种触发方式：

### 命令式触发（用户主动操作）

用户点击按钮、输入聊天命令等直接触发 workflow 执行：

```typescript
// 用户输入 .dd @agility → ChatInput 解析 → 调用 workflow runner
runner.runWorkflow(getDHActionCheckWorkflow(), {
  formula: '2d12',
  actorId: entity.id,
  origin: activeOrigin,
})
```

命令式触发中，第1条日志（如 roll-request）由 workflow 的第一个 step 产生。

### 声明式触发（日志订阅）— LogStreamDispatcher

声明式触发由 **`LogStreamDispatcher`** 实现——它是 WorkflowEngine 的上层调度器/消费者，监听日志流并在条件匹配时调用 WorkflowEngine。WorkflowEngine 本身是纯步骤编排器，不具备事件订阅能力。

插件在注册时声明触发器：

```typescript
// 插件注册触发器（在 IPluginSDK 中）
sdk.registerTrigger({
  id: 'dh:interpret-action-check',
  on: 'core:roll-result', // 监听的日志条目类型
  filter: { rollType: 'dh:action-check' }, // payload 字段匹配
  workflow: 'dh:interpret-action-check', // 触发的 workflow 名称
  mapInput: (entry) => ({
    // 从日志条目提取 workflow 输入
    rolls: entry.payload.rolls,
    total: entry.payload.total,
    origin: entry.origin,
  }),
  executeAs: 'triggering-executor', // 协调：谁来执行（不是权限）
})
```

`LogStreamDispatcher` 的职责：

1. 监听从 server 广播来的日志流
2. 匹配已注册的触发条件（type + filter）
3. 检查 `executor` 字段 — 只有匹配的客户端执行
4. 检查 `triggerable` 标志 — 为 false 的条目不触发任何 workflow
5. 调用 WorkflowEngine 执行对应的 workflow

### 级联保护

日志条目可以触发 workflow，workflow 可以产生新的日志条目，新条目又可能触发新的 workflow。防止无限循环的机制：

1. **`triggerable` 标志**（主要机制）：每条日志条目携带 `triggerable` 布尔字段，由条目作者控制。**默认为 false**（安全优先），只有明确需要触发后续 workflow 的条目才显式设为 true。
2. **`chainDepth` 字段**（安全网）：每条日志条目携带 `chainDepth` 整数，表示当前在因果链中的深度。顶层条目 `chainDepth = 0`，由触发器产生的子条目继承 `parent.chainDepth + 1`。LogStreamDispatcher 在触发前检查 `chainDepth >= MAX_CHAIN_DEPTH`（默认 10）时拒绝触发。

为什么不用 WorkflowEngine 的 `MAX_RECURSION_DEPTH`：日志触发链是异步跨网络的（workflow → flush → server → broadcast → dispatcher → 新 workflow），每次经过 server 广播都是全新的 `runWorkflow()` 调用，WorkflowEngine 的同步 depth 计数器会重置为 0。`chainDepth` 持久化在日志条目中，跨网络不丢失。

### 多阶段计算链

复杂的游戏逻辑（如 D&D 攻击→伤害→死亡检定→死亡效果）通过日志触发串联实现：

```
攻击 workflow → 产生 core:roll-result 日志（chainDepth: 0, triggerable: true）
  → 触发 伤害计算 workflow → 产生 dnd:damage-applied 日志（chainDepth: 1, triggerable: true）
    → 触发 死亡检定 workflow → 产生 dnd:death-check 日志（chainDepth: 2, triggerable: true）
      → 触发 死亡效果 workflow → 产生 core:component-update 日志（chainDepth: 3, triggerable: false，链终止）
```

每一环是独立的 workflow，通过日志条目的 `triggerable` 标志控制链的延续或终止。workflow 作者负责正确设置 `triggerable`。

## 执行者路由（Executor Routing）

### 谁来执行 workflow？

每条日志条目有一个 `executor` 字段，标识哪个客户端是这条日志链的权威执行者。

**`executor` 的含义是协调（coordination），不是权限（permission）。** 它决定哪个客户端运行后续的声明式 workflow，避免所有客户端重复执行。

**关键安全决策**：`executor` 由 server 根据 socket 声明的 seat 身份（`seat:claim`）自动标记，客户端不能伪造。在社会信任模型下，seat 身份是自声明的（self-claimed），不是认证的（authenticated）。

```
玩家 A 投骰 → socket.emit('log:roll-request') → server 标记 executor: 'seat-A'
→ 广播给所有客户端
→ 只有 seat-A 的 LogStreamDispatcher 执行后续链（解释 + 世界影响）
→ 其他客户端的 LogStreamDispatcher 忽略（executor 不匹配）
```

### 因果链（Causal Chain）

`parentId` 字段将日志条目链接为因果关系：

```
core:roll-result (id: "abc", parentId: null)
  └─ dh:judgment (id: "def", parentId: "abc")
      └─ core:tracker-update (id: "ghi", parentId: "def")
```

由执行客户端设置，server 存储。

## 日志提交 — 逐条即时发送

**不做批量缓冲。** 每条日志条目产生后立即通过 Socket.io 发送到 server。

### 为什么默认不做批量

- WebSocket 长连接的 per-message 开销极低（一个 frame），不需要攒批减少请求
- TTRPG 操作频率低（一次攻击 workflow 产生约 3 条日志），每条一个 SQLite 事务完全承受得起
- 批量引入大量复杂度（buffer 管理、flush 定时器、上限控制、SAVEPOINT、batch 原子性语义），收益极小
- 含 `serverRoll()` 的 workflow 天然不是原子的——server RNG 创建不可撤回的条目，批量的"原子性"是虚假的

### Opt-in 原子 Workflow

不含 `serverRoll()` 的纯计算 workflow（如：解释判定 + 更新 tracker）可以声明 `{ atomic: true }`：

```typescript
sdk.defineWorkflow('dh:apply-damage', [
  { id: 'calc', run: (ctx) => { /* 纯计算 */ } },
  { id: 'apply', run: (ctx) => { ctx.updateTeamTracker(...); ctx.updateComponent(...) } },
], { atomic: true })
```

行为差异：

|                | 默认（非原子）              | `{ atomic: true }`                         |
| -------------- | --------------------------- | ------------------------------------------ |
| 发送时机       | 每条 `emitEntry()` 立即发送 | workflow 完成后一次性发送所有条目          |
| abort 行为     | 已发送的条目不可撤回        | 全部丢弃，server 无感知                    |
| `serverRoll()` | 允许                        | **禁止**（运行时抛错）                     |
| 实现           | 直接 `socket.emit`          | per-workflow 本地 buffer → 完成后逐条 emit |

**不是通用 batch 机制**——没有全局 buffer、没有定时 flush。只是 per-workflow 的 opt-in 缓冲，workflow 作者显式承担原子性约束。

### Workflow 执行流程

```
workflow 执行中
  → step 1: ctx.serverRoll()
      → emit('log:roll-request') → await ack → 拿到 core:roll-result（server 已持久化+广播）
  → step 2: 产生 dh:judgment
      → emit('log:entry', judgment) → fire-and-forget（不等 ack）
  → step 3: 产生 core:tracker-update
      → emit('log:entry', trackerUpdate) → fire-and-forget
workflow 结束
```

Workflow 自行决定是否等待 server 确认：

- **需要返回值的操作**（如 `serverRoll()`）：`await` ack，workflow 挂起直到 server 响应
- **纯日志/副作用操作**（如 judgment、tracker-update）：fire-and-forget，workflow 立即继续

### RNG 请求路径

RNG 请求是唯一需要 server 参与构造日志条目的场景：

```
1. client emit('log:roll-request', { origin, dice, formula, rollType, parentId, chainDepth, triggerable, visibility, ... })
2. server 生成随机数（纯 RNG）
3. server 构造 core:roll-result 日志条目：
   - id: server 生成（UUIDv7）
   - type: 'core:roll-result'
   - origin, parentId, chainDepth, triggerable, visibility: 来自请求
   - executor: server 根据 socket 身份覆盖
   - payload: { dice, formula, resolvedFormula, rollType, actionName, rolls, total }
4. server 写入 game_log（获得 seq）
5. server 广播 log:new 给所有客户端（visibility 过滤）
6. server ack 返回完整条目给请求者
   → 请求者也会收到 broadcast，用 entriesById 去重
7. 客户端 workflow 消费 ack 中的 roll-result → 继续执行后续步骤（产生的条目逐条即时发送）
```

`log:roll-request` 是 Socket.io 事件名（传输层），**不写入 game_log**。只有 server 创建的 `core:roll-result` 是日志条目。

### 失败处理

Server 端 effect handler 几乎不会失败（`INSERT OR IGNORE`、`UPDATE ... WHERE id = ?` 不会抛异常）。如果失败，说明有 bug——ack 返回错误，客户端 toast 提示，开发者看日志修复。不做自动重试、不做降级。

## Side-Effect Registry（服务端副作用注册表）

Server 不懂游戏规则，但需要在日志写入的同时更新对应的数据表。**Side-Effect Registry** 解决了日志状态与表状态一致性问题。

### 机制

```typescript
// server/effectRegistry.ts
type EffectHandler = (db: Database, entry: GameLogEntry) => void
const registry = new Map<string, EffectHandler>()

// CRUD 类型条目注册副作用处理器
// effect handler 执行表更新后，将最新快照写回 entry.payload（广播时携带）
registry.set('core:tracker-update', (db, entry) => {
  const { trackerId, delta } = entry.payload as { trackerId: string; delta: number }
  db.prepare('UPDATE team_trackers SET current = current + ? WHERE id = ?').run(delta, trackerId)
  // 写回快照：客户端直接用此快照替换 store，无需自己计算 delta
  const snapshot = db.prepare('SELECT * FROM team_trackers WHERE id = ?').get(trackerId)
  ;(entry.payload as Record<string, unknown>).snapshot = snapshot
})

registry.set('core:component-update', (db, entry) => {
  const { entityId, key, data } = entry.payload as { entityId: string; key: string; data: unknown }
  db.prepare(
    'INSERT OR REPLACE INTO entity_components (entity_id, component_key, data) VALUES (?, ?, ?)',
  ).run(entityId, key, JSON.stringify(data))
  // component-update 的 data 本身就是全量替换，无需额外快照
})
```

### 执行流程

每条日志条目一个事务，逻辑极简：

```
socket 'log:entry' entry
  → BEGIN TRANSACTION
    → const info = INSERT OR IGNORE INTO game_log ... (entry)
    → if (info.changes > 0)                    // 实际插入了（非重复）
        effectRegistry.get(entry.type)?.(db, entry)
        → effect handler 将执行后的快照写回 entry.payload
  → COMMIT
  → 广播给客户端（visibility 过滤，payload 包含快照）
  → ack 返回带 seq 的完整 GameLogEntry
```

**关键保证**：日志写入和表更新在同一事务中。`info.changes > 0` 检查确保重复提交（相同 `id`）的条目不会重复执行副作用。Node.js 单线程 + better-sqlite3 同步执行，天然保证事务串行，无并发问题。

### 客户端状态同步

CRUD 类型的日志条目在 payload 中同时携带**操作（delta）和执行后的快照（snapshot）**：

```typescript
// core:tracker-update payload（server effect handler 执行后）
{ trackerId: "t1", delta: -5, snapshot: { id: "t1", label: "HP", current: 45, max: 100, ... } }

// core:component-update payload
{ entityId: "e1", key: "stats", data: { /* 完整 component 数据 */ } }
```

Server 的 effect handler 执行完表更新后，将最新状态写回 payload 的 `snapshot` 字段。广播出去的日志条目自带权威快照。

客户端收到 `log:new` 后的处理：

1. **日志条目** → 存入 `logEntries`，渲染到聊天面板（展示"HP -5"）
2. **snapshot** → 直接替换 zustand store 中的对象（和现有 `tracker:updated` 广播模式一致）

好处：

- **自愈**：客户端错过一条广播，下一条广播的 snapshot 直接修正，不会越漂越远
- **不重复逻辑**：客户端不需要实现 effect 计算（`current += delta`），直接用 server 给的权威快照
- **历史可读**：回看日志时既能看到"发生了什么"（delta），也能看到"当时状态是什么"（snapshot）

### 类型命名约定

所有日志类型必须带**命名空间前缀**，格式为 `namespace:type`：

- **`core:`** — 基座提供的核心类型（`core:text`, `core:roll`, `core:roll-result`, `core:tracker-update`, `core:component-update`, `core:retract` 等）
- **`dh:`** — Daggerheart 插件类型（`dh:judgment`, `dh:action-check` 等）
- **`dnd:`** — D&D 插件类型（`dnd:damage-applied`, `dnd:death-check` 等）

不允许无前缀的裸类型名。好处：

- 拼错命名空间时更容易发现（`core:` vs `cor:`）
- 可以用 `type.startsWith('core:')` 做快速分类
- 避免插件类型和基座类型冲突

### 类型分类

- **有副作用的类型**（CRUD）：`core:tracker-update`, `core:component-update`, `core:token-create` 等 — 在 registry 中注册处理器
- **纯日志类型**：`core:text`, `core:roll`, `dh:judgment` 等 — 不注册处理器，仅写入日志

## 信任模型 — 社会信任

**不做写入权限校验。** 像实体桌游一样，信任参与者。

理由：

- TTRPG 是合作游戏，玩家之间有社会契约
- 写入权限引入巨大的架构复杂度（GM 委托、权限拆分、跨客户端协调）
- 在 POC 阶段投入这些复杂度是不值得的
- 日志是完整的审计线索，任何误操作都可追溯

权限模型简化为：

- **Visibility 处理 READ 权限**：控制谁能看到哪些日志条目
- **UI 引导操作**：UI 层面限制玩家只看到自己可以操作的按钮
- **日志审计**：所有操作都有完整记录，可事后追查

`executor` 字段保留，但含义仅为**协调**（谁来执行 workflow），不是权限校验。

### 投骰的所有权

投骰属于投骰者。玩家发起投骰请求，server 提供 RNG 服务。

### 特权客户端（GM）

GM 客户端是特权执行者。桌游的本质决定了 GM 必须在线游戏才能进行，因此 GM 在线是前提条件，不是需要解决的可靠性问题。

## EventBus 与跨客户端效果

### 现有问题

当前 EventBus（`src/events/eventBus.ts`）有一个 executor-only bug：`ctx.events.emit()` 只在执行 workflow 的客户端触发。其他客户端看不到 toast、动画、音效等效果。

### 解决方案：渲染器驱动

**跨客户端效果改为由日志条目渲染器驱动，不再由 workflow step 触发。**

```typescript
// 旧方式（有 bug）— workflow step 中 emit
ctx.events.emit(toastEvent, { text: '命中！' })  // ← 只有执行者看到

// 新方式 — 日志渲染器中触发效果
// DaggerheartJudgmentRenderer.tsx
function DaggerheartJudgmentRenderer({ entry }: { entry: GameLogEntry }) {
  useEffect(() => {
    // 所有收到这条日志的客户端都会执行
    playAnimation(entry.payload.outcome)
    playSound(entry.payload.outcome === 'critical' ? 'fanfare' : 'impact')
  }, [entry.id])

  return <JudgmentCard {...entry.payload} />
}
```

### EventBus 定位调整

- **废弃**：跨客户端效果（动画、音效、公告消息）
- **保留**：纯本地 UI 事件（toast 通知等不需要跨客户端的场景）
- `announceEvent` 不再通过 EventBus → sendMessage，而是直接产生 `core:text` 类型日志条目

## WorkflowContext — 即时发送模式

WorkflowContext 的副作用接口不再直接发 REST 请求，改为通过 Socket.io 即时发送日志条目：

```typescript
// 新模型：直接 emit，fire-and-forget
updateComponent<T>(entityId: string, key: string, updater: (current: T | undefined) => T): void {
  const current = this.read.component<T>(entityId, key)
  const next = updater(current)
  this.emitEntry({
    type: 'core:component-update',
    payload: { entityId, key, data: next },
    triggerable: false,
  })
}

updateTeamTracker(label: string, patch: { current?: number }): void {
  this.emitEntry({
    type: 'core:tracker-update',
    payload: { label, ...patch },
    triggerable: false,  // CRUD 操作默认不触发后续 workflow
  })
}

// emitEntry: socket.emit('log:entry', entry) 的封装，自动填充 origin/executor/timestamp 等元数据
```

每次调用立即发送到 server，server 在同一事务中完成日志写入和表更新。Workflow 不等 ack，立即继续执行下一步。如果 workflow 需要等待 server 确认（如依赖 server 返回值的场景），可以显式 `await`。

### 并发模型

两种更新接口对应两种并发语义：

| 接口                | Server 端行为                   | 并发安全性                                     |
| ------------------- | ------------------------------- | ---------------------------------------------- |
| `updateTeamTracker` | `current + delta`（原子增量）   | 天然安全——多个 delta 按 seq 顺序累加，结果正确 |
| `updateComponent`   | `INSERT OR REPLACE`（全量替换） | Last-write-wins——后到的写入覆盖先到的          |

`updateComponent` 的 `updater` 回调只是本地便利函数（读当前 store → 计算新值 → 发送全量），**不提供跨客户端原子性**。对于天然单人编辑的数据（角色名、描述、配置）这不是问题。

如果未来某种 component 需要并发安全的增量更新，正确的做法是为它设计专用的 delta 类型（像 `tracker-update` 一样），而不是给 `updateComponent` 加冲突检测。冲突检测能发现问题，但无法自动解决——拒绝写入会丢数据，合并任意 JSON 不可行。

每条日志条目携带 `baseSeq`（客户端声明的最后已知 seq），为未来按需启用乐观并发检测提供数据基础，当前不做任何检查。

## 重连与状态恢复

**各数据表就是当前世界的权威快照。** 重连时不需要回放日志历史。

### 两层重连机制

**层1：Socket.io Connection State Recovery（自动，覆盖 90% 短暂断线）**

Socket.io v4.6+ 内置的 connection state recovery：

```typescript
// server 配置
const io = new Server(server, {
  connectionStateRecovery: {
    maxDisconnectionDuration: 2 * 60 * 1000, // 2 分钟内重连
  },
})
```

断线 2 分钟内重连时，Socket.io 自动恢复 socket.data（含 seatId）并补发断线期间所有 `log:new` 事件。客户端用 `entriesById` 去重，无需额外处理。

**层2：完整状态重建（Socket.io recovery 失败时的 fallback）**

recovery 失败的情况：超过 maxDisconnectionDuration、server 重启、内存 buffer 溢出。

```
阶段1（静默恢复）：
  socket.emit('room:init', roomId, (ack) => {
    // ack = { tables: {...}, logWatermark: 102 }
    // → 加载当前世界状态（权威快照）
  })

  socket.emit('log:history', { beforeSeq: 102, limit: 50 }, (ack) => {
    // → 加载最近日志（仅用于 UI 展示，不触发任何 workflow）
  })

阶段2（上线）：
  LogStreamDispatcher 开始监听 log:new 广播
  → 收到的条目：entry.seq <= logWatermark → 跳过（已反映在表状态中）
  → 收到的条目：entry.seq > logWatermark → 新条目，正常触发 workflow
```

关键保证：`logWatermark` 是 server 分配的 `seq`（单调递增），不依赖任何客户端时钟，不存在时钟偏移问题。

## Server 的职责

Server 不懂任何游戏规则。它的职责：

1. **RNG** — 生成随机数（防作弊），通过 `log:roll-request` 事件即时响应
2. **持久化** — 在同一事务中存储日志条目 + 执行 Side-Effect Registry 更新（含 `changes()` 去重检查）
3. **广播** — 将日志条目（含 server 分配的 `seq`）推送给所有客户端（逐 socket visibility 过滤）
4. **强制覆盖 `executor`** — 根据 socket 声明的 seat 身份无条件覆盖，客户端传入值被忽略
5. **分配 `seq`** — 单调递增序号，保证全局排序和重连水位线

## 可见性模型（Visibility）

每条日志条目有一个 `visibility` 字段控制谁能看到：

- `{}` — 所有人可见（默认）
- `{ include: ['gm', 'seat-3'] }` — 白名单，仅指定 seat 可见
- `{ exclude: ['seat-5'] }` — 黑名单，排除指定 seat

Server 在广播时根据 visibility 过滤，只推送给有权查看的客户端。历史查询同理。

## 插件架构影响

插件不再需要单独暴露 `diceSystem.evaluateRoll()` 接口。解释逻辑注册为 workflow step，只有 workflow 引擎消费它。

插件提供：

- **Workflow 注册**：注册命令式 workflow 和声明式触发器（通过 LogStreamDispatcher）
- **渲染器**：注册对特定日志类型的 UI 渲染组件（`logEntryRenderers[type]`）
- 渲染器是纯展示，可以调用插件内部的纯函数来丰富展示效果
- **渲染器驱动效果**：跨客户端的动画、音效等由渲染器触发，不由 workflow step 触发

## 存储设计

### 单表统一日志

```sql
CREATE TABLE IF NOT EXISTS game_log (
  seq INTEGER PRIMARY KEY AUTOINCREMENT,   -- server 分配，全局排序键 + 重连水位线
  id TEXT NOT NULL UNIQUE,                 -- 客户端生成的 UUIDv7，去重键 + 因果链标识
  type TEXT NOT NULL,          -- 'core:text' | 'core:roll' | 'dh:judgment' | 'core:tracker-update' | ...
  origin TEXT NOT NULL,        -- JSON: MessageOrigin { seat, entity? }
  executor TEXT NOT NULL,      -- seat ID，由 server 根据 socket 身份强制覆盖（协调用，非权限）
  parent_id TEXT,              -- 因果链：指向触发本条目的父条目（引用 id，非 seq）
  chain_depth INTEGER NOT NULL DEFAULT 0,  -- 因果链深度（顶层=0，子条目=parent+1），用于级联保护
  triggerable INTEGER NOT NULL DEFAULT 0,  -- 是否允许触发声明式 workflow（0/1，默认不触发）
  visibility TEXT NOT NULL DEFAULT '{}',   -- JSON: {} | { include: [] } | { exclude: [] }
  base_seq INTEGER NOT NULL DEFAULT 0,     -- 客户端声明的"我看到的最后一条日志 seq"（乐观并发控制预留）
  payload TEXT NOT NULL DEFAULT '{}',      -- 不透明 JSON，只有对应的渲染器/workflow 理解
  timestamp INTEGER NOT NULL               -- 客户端时间戳，仅用于展示（排序用 seq）
);
CREATE INDEX IF NOT EXISTS idx_game_log_type ON game_log(type);
CREATE INDEX IF NOT EXISTS idx_game_log_executor ON game_log(executor);
CREATE INDEX IF NOT EXISTS idx_game_log_parent ON game_log(parent_id);
```

### 双 ID 设计

|        | `id` (UUIDv7, 客户端生成)                          | `seq` (INTEGER, server 分配)                |
| ------ | -------------------------------------------------- | ------------------------------------------- |
| 生成者 | 客户端                                             | Server（AUTOINCREMENT）                     |
| 用途   | 去重键（`INSERT OR IGNORE`）、因果链（`parentId`） | 全局排序、重连水位线、UI 展示顺序、游标分页 |
| 可信度 | 不可信（客户端时钟）                               | 权威（server 单调递增）                     |

`id` 是客户端的"身份证"——保证不重复提交。`seq` 是 server 的"排队号"——保证全局有序。

### 日志条目格式

```typescript
interface GameLogEntry {
  seq: number // server 分配的全局排序序号（排序、水位线）
  id: string // 客户端生成的 UUIDv7（去重、因果链）
  type: string // 语义类型，用于 workflow 触发和 UI 渲染匹配
  origin: MessageOrigin // 谁产生的（seat + entity）
  executor: string // 哪个 seat 是权威执行者（server 强制覆盖，协调用）
  parentId?: string // 因果链父条目（引用 id）
  chainDepth: number // 因果链深度（顶层=0，子条目=parent+1），LogStreamDispatcher 检查上限
  triggerable: boolean // 是否允许触发声明式 workflow（默认 false）
  visibility: {} | { include: string[] } | { exclude: string[] }
  baseSeq: number // 客户端声明的"我看到的最后一条日志 seq"（乐观并发控制预留）
  payload: Record<string, unknown> // 不透明数据，只有类型对应的消费者理解
  timestamp: number // 客户端时间戳，仅用于展示
}
```

## 日志不可变性（Append-Only）

日志条目一旦写入**不可撤回、不可删除**。如需修正，通过新条目表达：

- 撤回消息：产生 `type: 'core:retract'` 条目，`payload.targetId` 指向被撤回的条目 `id`
- UI 渲染时检查是否有对应的 retract 条目，有则标记为已撤回（灰显/折叠）
- 因果链上的后续条目不受影响（已经产生的副作用不回滚）

理由：append-only 模型避免了"撤回后因果链断裂"和"副作用回滚"的复杂性。

## UI 渲染模型

### 因果链分组

三层骰子模型产生多条独立日志条目（`core:roll-result` → `dh:judgment` → `core:tracker-update`），UI 通过 `parentId` 将它们组装为一个视觉单元。渲染策略：

- 聊天面板按 `parentId` 分组，将同一因果链的条目渲染为关联卡片
- 条目逐条到达（各自独立发送），UI 逐条追加渲染，后续条目到达时自动归入已有分组

## Assumptions

- 项目未上线，无需向后兼容。迁移策略为一次性切换（新表替换旧表），不做渐进式迁移
- GM 在线是游戏进行的前提条件（桌游本质决定）
- 参与者之间存在社会信任（合作游戏，非对抗）；seat 身份是自声明的，不是认证的
- SQLite 单写者模型足够处理并发（server 是唯一写入者）
- Server 分配的 `seq` 保证全局排序；UUIDv7 仅用于去重和因果链
- Socket.io 是日志系统的唯一传输层；管理操作保持现有 REST 模式

## Edge Cases

- **并发写入**：多客户端同时修改同一资源 → `tracker-update` 用原子 delta 天然安全；`component-update` 用 last-write-wins（server 按到达顺序处理）
- **Side-Effect 处理器失败**：每条日志条目独立事务，失败只影响该条目。ack 返回错误，客户端 toast 提示。不做自动重试
- **触发器循环**：`triggerable` 标志配错导致无限触发 → `chainDepth >= MAX_CHAIN_DEPTH`（默认 10）时 LogStreamDispatcher 拒绝触发。`chainDepth` 持久化在日志条目中，跨网络不丢失
- **Executor 离线**：任何 executor 掉线（不仅是 GM）都会导致该 seat 的触发链中断——其 LogStreamDispatcher 不在线，无人执行后续 workflow。这是桌游本质决定的：玩家离桌 = 该玩家的行为暂停。重连后 LogStreamDispatcher 重新开始监听新日志，不补执行断线期间错过的触发
- **Workflow 执行中断线**：Workflow 是客户端的临时计算过程，没有 server 端状态，不可恢复。网络闪断（页面还在）时 Socket.io 缓冲未发送的 emit，重连后自动发出；但如果 `serverRoll()` 在断线期间超时，workflow 失败——非原子 workflow 留下部分因果链（已发送的条目已持久化），原子 workflow 干净丢弃（buffer 未发送）。页面刷新则 workflow 完全消失。两种情况都不尝试恢复，用户重连后重新操作
- **触发执行顺序**：一条 `log:new` 广播到达后，如果匹配多个 trigger，LogStreamDispatcher 按 trigger 注册顺序串行执行。不并行执行，避免竞态条件
- **重复条目的 ack**：`INSERT OR IGNORE` 后 `changes() === 0` → ack 返回 `{ id, seq, status: 'duplicate' }`，其中 seq 通过额外 SELECT 查询已有条目获得
- **历史加载 vs 实时触发**：客户端通过 `logWatermark`（server 分配的 `seq`）区分历史和实时条目。`seq <= watermark` 跳过，`seq > watermark` 交给 LogStreamDispatcher。不依赖时序判断
- **客户端重试（网络超时）**：同一条目重复提交 → `INSERT OR IGNORE` 按 UUIDv7 主键去重，`info.changes > 0` 检查确保已存在条目的副作用不重复执行
- **短暂断线重连**：Socket.io connection state recovery（2 分钟内）自动补发错过的广播，客户端 `entriesById` 去重。CSR 补发的是原始广播时已按 visibility 过滤的事件，不会泄露信息
- **长时间断线重连**：Socket.io recovery 失败 → fallback 到完整状态重建（`room:init` + `log:history` + watermark）
