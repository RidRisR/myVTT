# 16 - 战术地图工具系统与插件地图集成设计

> **实现状态：未开始**
>
> 本文档设计工具注册架构、选择系统重构、交互模式改造、以及 RulePlugin 地图扩展点。
> 基于现有战术系统（文档 10）和规则插件系统（文档 11）进行扩展。

## 一、背景与动机

### 1.1 当前状态

战术系统（文档 10）已实现核心功能：KonvaMap 层级、Token 拖拽、网格吸附、测量/范围工具。
规则插件系统（文档 11）已实现 7 层接口，覆盖数据适配、角色卡、骰子、模板、UI 表面、元素隐藏。

**但两个系统之间存在断层**：插件完全无法感知或参与地图交互。

### 1.2 具体问题

| 问题            | 现状                                                                           | 影响                                       |
| --------------- | ------------------------------------------------------------------------------ | ------------------------------------------ |
| 工具硬编码      | `ActiveTool` 是联合类型 `'select' \| 'measure' \| ...`，新增工具需改 4+ 个文件 | 不可扩展，插件无法注册工具                 |
| 选择系统原始    | `selectedTokenId: string \| null` 单选                                         | 无多选、无框选、无批量操作                 |
| 插件零地图集成  | RulePlugin 7 层均不涉及地图                                                    | 无法实现"选中敌人→攻击掷骰"等核心 VTT 交互 |
| 事件路由 ad-hoc | 工具各自绑定 stage 事件，无统一接口                                            | 工具间事件冲突，新增工具需改 KonvaMap      |
| 右键菜单封闭    | 仅 GM 可见，硬编码菜单项                                                       | 玩家无右键操作，插件无法注入菜单项         |

### 1.3 设计目标

1. **建立工具注册架构**：内置工具与插件工具走同一注册路径
2. **重构选择系统**：支持多选、框选、意图标记、动作目标选择
3. **开放地图扩展点**：扩展 RulePlugin Layer 5 (surfaces)，使插件能注册工具、提供选择动作、扩展右键菜单
4. **分离地图层与游戏层**：明确交互层的职责边界

## 二、设计原则

### 原则 1：工具即交互模式（方案 C）

工具决定画布上什么可以交互。激活测量工具时 token 不可点击/拖拽；激活 select 工具时 token 可交互。通过 Konva Layer 的 `listening` 属性实现零冲突事件路由。

**参考**：Foundry VTT 层级路由、Photoshop 工具模式。

### 原则 2：One-shot 默认

大部分工具用完自动回到 Select 工具（one-shot 模式），消除工具切换成本。持续性工具（如绘图）显式标记为 persistent。

**参考**：Figma 单次/持续模式、Excalidraw 锁定图标。

### 原则 3：统一注册路径

内置工具和插件工具通过同一个 `ToolRegistry` 注册，TacticalToolbar 和键盘系统从注册表读取，不硬编码任何工具。

**参考**：VS Code Extension API 的 `registerCommand` 模式。

### 原则 4：选择 ≠ 目标

VTT 的选择涉及两个角色：**发起者**（Selection：我控制谁）和**目标**（Targeting：我的动作作用于谁）。两者分别建模，目标选择由动作（插件）驱动。

**参考**：Foundry VTT 的 Selection + Targeting 分离。

## 三、工具注册架构

### 3.1 三层设计

```
层 1：ToolDefinition 接口     → 约束所有工具的结构（类型安全）
层 2：BuiltinToolId 常量对象   → 内置工具 ID 有自动补全和拼写检查
层 3：ToolRegistry 类          → 统一注册/查询/遍历
```

### 3.2 ToolDefinition 接口

```typescript
// src/combat/tools/types.ts

type ToolCategory = 'interaction' | 'measurement' | 'drawing' | 'gm' | 'camera' | 'plugin'

interface ToolLayerProps {
  stageRef: React.RefObject<Konva.Stage>
  tacticalInfo: TacticalInfo
  stageScale: number
  stagePos: { x: number; y: number }

  id: string // 'select', 'measure', 'plugin:dh:aura'
```

### 3.4 ToolRegistry

```typescript
// src/combat/tools/toolRegistry.ts

class ToolRegistry {
  private tools = new Map<string, ToolDefinition>()

  register(def: ToolDefinition): void {
    if (this.tools.has(def.id)) {
      console.warn(`Tool "${def.id}" already registered, skipping`)
      return
    }
    this.tools.set(def.id, def)
  }

  get(id: string): ToolDefinition | undefined {
    return this.tools.get(id)
  }

  getAll(): ToolDefinition[] {
    return [...this.tools.values()]
  }

  getByCategory(cat: ToolCategory): ToolDefinition[] {
    return this.getAll().filter((t) => t.category === cat)
  }

  has(id: string): boolean {
    return this.tools.has(id)
  }
}

export const toolRegistry = new ToolRegistry()
```

模块级单例。应用启动时注册内置工具，插件加载时注册插件工具。不使用 zustand store——工具定义是静态的，运行期不变。

## 四、交互模式

### 4.1 Konva 层级顺序（底→顶）

```
Stage (draggable={false}, onWheel={zoom})
├── Layer [listening={false}]
│   ├── BackgroundLayer          // 地图图片/视频
│   └── KonvaGrid               // 网格线
│
├── KonvaTokenLayer [listening = activeCategory === 'interaction']
│   ├── KonvaToken × N          // 可拖拽/可选择（仅 select 工具时）
│   └── GhostToken              // 远程拖拽预览
│
├── ActiveToolCanvas [listening = true, 仅激活工具渲染]
│   └── 当前工具的 CanvasLayer   // MeasureTool / RangeTemplate / SelectMarquee / ...
│
└── Layer [listening={false}]
    └── PersistedOverlays        // 已锁定的测量线/范围模板（Shift+click 持久化的）
```

### 4.2 事件路由规则

| 交互                 | 路由目标                 | 不受工具影响          |
| -------------------- | ------------------------ | --------------------- |
| 中键拖 / 空格+左键拖 | 全局平移处理器           | ✅ 始终可用           |
| 滚轮                 | 全局缩放处理器           | ✅ 始终可用           |
| 右键                 | 上下文菜单（工具可拦截） | ⚠️ 工具可选拦截       |
| 左键点击/拖 token    | KonvaTokenLayer          | ❌ 仅 select 工具时   |
| 左键点击/拖空白      | ActiveToolCanvas         | ❌ 仅非 select 工具时 |
| 左键框选空白         | SelectTool.CanvasLayer   | ❌ 仅 select 工具时   |

### 4.3 平移实现

Stage `draggable` 设为 `false`。平移通过自定义事件处理实现：

```typescript
// 中键拖拽
onMouseDown(e) { if (e.evt.button === 1) startPan(e) }
onMouseMove(e) { if (isPanning) updatePan(e) }
onMouseUp(e)   { if (isPanning) endPan(e) }

// 空格+左键拖拽
onKeyDown(e)   { if (e.key === ' ') enableSpacePan() }
onKeyUp(e)     { if (e.key === ' ') disableSpacePan() }
```

### 4.4 One-shot 模式

工具的 `CanvasLayer` 在操作完成时调用 `props.onComplete()`：

```typescript
// KonvaMap 中
function handleToolComplete() {
  const tool = toolRegistry.get(activeTool)
  if (tool?.defaultMode === 'one-shot') {
    setActiveTool(BuiltinToolId.Select)
  }
}

<ActiveToolCanvas onComplete={handleToolComplete} />
```

用户可通过双击工具栏按钮临时切换到 persistent 模式（UI 显示锁头图标）。

## 五、选择系统

### 5.1 三层模型

```
Selection（控制选择）     — 本地 uiStore 状态
Marking（意图标记）       — awareness 广播，纯视觉沟通
Action Targeting（动作目标） — 插件驱动的临时工具
```

### 5.2 Selection

```typescript
// uiStore 变更

// 变更前
selectedTokenId: string | null
setSelectedTokenId: (id: string | null) => void

// 变更后
selectedTokenIds: string[]       // 有序数组，最后一个 = 主选择（焦点）

// 派生（不存储）
get primarySelectedTokenId(): string | null  // selectedTokenIds 的最后一个元素

// 操作方法
setSelectedTokenIds: (ids: string[]) => void
addToSelection: (id: string) => void
removeFromSelection: (id: string) => void
toggleSelection: (id: string) => void
clearSelection: () => void
```

**交互映射**：

| 用户操作             | 效果                                   |
| -------------------- | -------------------------------------- |
| 左键点击 token       | 替换选择（清除旧的，选新的）           |
| Shift+左键点击 token | 添加/移除到选择集                      |
| 左键拖空白区域       | 框选（marquee）— 框内所有 token 被选中 |
| Delete 键            | 删除所有选中 token（带 undo toast）    |
| 方向键               | 移动所有选中 token（按网格步进）       |
| Escape               | 清除选择                               |

**视觉区分**：

- 主选择（最后一个）：白色高亮环 + 阴影（当前样式）
- 次要选择：白色高亮环，无阴影，稍低透明度

### 5.3 Marking（意图标记）

通过 Socket.io awareness 事件广播，不持久化到数据库：

```typescript
// 广播
socket.emit('awareness:mark', { tokenIds: string[], color: string })
// 接收
socket.on('awareness:mark', (data) => { /* 在 token 上显示标记图标 */ })
socket.on('awareness:markClear', (data) => { /* 清除标记 */ })
```

可作为 Select 工具的子操作（如对 token 按 T 键标记），也可实现为独立轻量工具。所有玩家看到标记（颜色标识谁标记的），token 上显示十字准星或类似标记图标。

纯沟通用途——"我打算攻击这个"——不影响游戏机制。

### 5.4 Action Targeting（动作目标选择）

当插件动作需要目标时，base 临时激活一个"目标选择工具"：

```typescript
// 插件声明动作的目标需求
interface TargetingRequest {
  mode: 'single' | 'multiple' | 'sequential'
  count?: number // sequential 模式：需要选几个（如闪电链 = 3）
  filter?: 'enemy' | 'ally' | 'any'
  labels?: string[] // sequential 模式：每步提示文案
}

// 目标选择结果
interface TargetInfo {
  tokenId: string
  entity: Entity
  index: number // sequential 模式中的选择顺序
  label?: string // 对应的标签（如"主目标"、"连锁目标1"）
}
```

**流程**：

```
1. 玩家选中自己的 token（Selection）
2. Action Bar 显示插件提供的动作按钮
3. 玩家点击"闪电链"按钮
4. Base 检查 action.targeting → 有 TargetingRequest
5. 激活 ActionTargetingTool（临时 one-shot 工具）
   - 画布顶部显示提示："选择主目标（1/3）— 点击目标 token，右键取消"
   - 鼠标变为十字准星样式
   - 可选 token 高亮，不可选 token 变暗
6. 玩家逐一点击目标 token
7. 全部选完 → ActionTargetingTool 自动结束（one-shot）
8. 调用 action.onExecute(actorEntity, targetInfos)
9. 回到 Select 工具
```

**取消**：右键或 Escape 取消目标选择，回到 Select 工具。

### 6.2 渲染逻辑

TacticalToolbar 从 `toolRegistry.getByCategory()` 读取工具列表渲染，不硬编码任何工具按钮：

```typescript
function TacticalToolbar() {
  const categories: ToolCategory[] = ['interaction', 'measurement', 'gm', 'camera', 'plugin']

  return (
    <div className="toolbar-container">
      {categories.map((cat) => {
        const tools = toolRegistry.getByCategory(cat)
        if (tools.length === 0) return null
        const visibleTools = tools.filter((t) => !t.gmOnly || role === 'GM')
        if (visibleTools.length === 0) return null
        return (
          <Fragment key={cat}>
            <ToolGroup tools={visibleTools} activeTool={activeTool} />
            <Divider />
          </Fragment>
        )
      })}
    </div>
  )
}
```

### 6.3 键盘快捷键

`useTacticalKeyboard` 从注册表数据驱动：

```typescript
function useTacticalKeyboard() {
  useEffect(() => {
    function handleKeyDown(e: KeyboardEvent) {
      if (isTyping(e)) return

      // 遍历注册表匹配快捷键
      for (const tool of toolRegistry.getAll()) {
        if (tool.shortcut && matchShortcut(e, tool.shortcut)) {
          setActiveTool(tool.id)
          return
        }
      }

      // 全局快捷键（非工具）
      // ... Escape, Delete, 方向键等
    }
    window.addEventListener('keydown', handleKeyDown)
    return () => window.removeEventListener('keydown', handleKeyDown)
  }, [])
}
```

### 6.4 Measurement Split Button

保留现有 split button 交互模式。measurement 分组的工具显示为 split button：主按钮显示上次使用的测量工具，展开面板显示全部测量工具。

## 八、地图层 vs 游戏层边界

### 8.1 职责划分

| 地图层（Base 系统）                  | 游戏层（插件系统）                 |
| ------------------------------------ | ---------------------------------- |
| Token 空间属性 (x, y, width, height) | HP/资源/状态数据 (Entity.ruleData) |
| 网格渲染与吸附                       | 先攻/回合顺序                      |
| 相机控制（缩放/平移）                | 属性/技能检定                      |
| 拖拽物理与 awareness                 | 掷骰动作与结算                     |
| Token 可见性渲染                     | 角色卡 UI                          |
| 测量/范围工具                        | 战斗规则与自动化                   |
| Selection / Marquee                  | Action Targeting 流程              |
| 右键菜单框架                         | 右键菜单游戏项                     |

### 8.2 桥接模式

Base 通过 `plugin.adapters.*`（Layer 1）获取通用视图渲染 token 覆盖层：

```
KonvaToken 渲染流程:
  entity.ruleData → plugin.adapters.getMainResource() → ResourceView → HP 条
  entity.ruleData → plugin.adapters.getStatuses()      → StatusView[] → 状态标签
```

插件不直接控制 token 渲染方式（颜色、形状、大小等由 Base 统一处理），只控制数据内容。

### 8.3 先攻/回合系统

先攻是游戏概念，不同规则系统差异大（DH 无传统先攻、D&D 用 d20+DEX）。

当前 schema 已预留字段（`tactical_state.current_turn_token_id`），但暂不在 Base 中使用。先攻系统留待插件通过 Layer 7 (Rule Resolution) 或自定义面板实现。

## Assumptions

- 工具数量在可预见范围内不超过 15 个（含插件工具），无需虚拟化/分页
- 所有工具定义是静态的（应用启动时注册，运行期不变），无需响应式 store
- 插件在 Phase 1 仍然是编译时打包（文档 11 Phase 1），工具注册在模块加载时完成
- 中键和空格键在目标浏览器环境中可用（桌面端优先，移动端延后）
- 每个 ToolDefinition.CanvasLayer 是独立的 React 组件，不需要跨工具共享内部状态

## Edge Cases

- 插件工具 ID 与内置工具冲突 → skip + `console.warn`，内置优先
- 插件快捷键与内置快捷键冲突 → 内置优先，插件快捷键被忽略 + `console.warn`
- 选择的 token 在其他客户端被删除 → `selectedTokenIds` 中的无效 ID 在渲染时过滤（lazy cleanup）
- Action Targeting 过程中目标 token 被删除 → 取消当前目标选择流程，回到 Select 工具
- One-shot 工具的 `onComplete` 未被调用（用户右键/Escape 取消）→ 工具自行处理取消逻辑，不自动切回
- 多个插件注册同 category 的工具 → 按插件加载顺序排列，不做去重
- 框选区域内无 token → 清除选择（等同于点击空白区域）
- 中键不可用（某些触控板）→ 空格+左键拖作为备用平移方式
